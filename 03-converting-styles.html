Part 1 of 7 · Concept

Three ways to say the same thing

Haskell gives you three different dialects for processing lists: explicit recursion, list comprehensions, and higher-order functions (map/filter). They're not competing, they're synonyms. Any problem in one can be rewritten in the other two. Converting fluently between them is the whole skill being tested in P.3.

The big idea

The three styles are equivalent for the same problem. Here's the same function written three ways:

Style 1Recursion with patterns
doubleEvens :: [Int] -> [Int]
doubleEvens [] = []
doubleEvens (x:xs)
  | even x    = (x * 2) : doubleEvens xs
  | otherwise = doubleEvens xs
Style 2List comprehension
doubleEvens :: [Int] -> [Int]
doubleEvens xs = [x * 2 | x <- xs, even x]
Style 3Higher-order functions
doubleEvens :: [Int] -> [Int]
doubleEvens xs = map (*2) (filter even xs)

All three produce [4, 8] when given [1, 2, 3, 4, 5]. Each style has its own vibe, but every P.3 problem is a translation exercise.

What the exam asks

P.3 gives you a function written in one style and asks you to rewrite it in another. Sometimes it's fill-in-the-blanks, sometimes it's write-from-scratch.

His exam from the study guide:

  1. Given creek as recursion, fill in blanks for creek1 using map + filter.
  2. Given creek, fill in blanks for creek2 as a list comprehension.
  3. Given pond as a list comprehension, write pond1 using recursion with patterns.

Three translations. Same pair of ideas (transform, filter). Different syntax.

The two questions to ask about any list-processing function

Every one of these problems boils down to two questions:

The two questions
  1. What's the filter? Which elements do we keep? (Sometimes: all of them.)
  2. What's the transformation? What does each kept element become? (Sometimes: unchanged.)

Once your student identifies these two pieces, the translation between styles is mechanical. We'll drill this throughout the lesson.

The plan for this lesson

  1. Part 2, recursion with patterns. You know this from Lesson 1; we formalize the structure.
  2. Part 3, list comprehensions. The concise cousin of Python's [x for x in ...].
  3. Part 4, higher-order functions. map, filter, and (optionally) combining them.
  4. Part 5, the translation template. How to go between all three mechanically.
  5. Part 6, his exam problems, step-by-step.
  6. Part 7, practice problems with hidden answers.
The teaching payoff

Once your student sees all three styles as "the same function wearing different clothes," his Haskell fluency doubles. He'll recognize patterns faster, write cleaner code, and ace P.3 without thinking.

Part 2 of 7 · Style 1

Recursion with patterns

You've already written list recursion in Lesson 1. Here we formalize the shape so you can recognize it instantly. Every list-processing function written recursively follows the same skeleton.

The universal list-recursion skeleton

Every recursive list function, no exceptions, has these two equations:

The list recursion skeleton
f :: [A] -> [B]        -- (or some return type)
f []     = ???         -- the empty list case (base case)
f (x:xs) = ???         -- the non-empty list case (recursive case)

Two equations. First matches empty. Second pattern matches head and tail. Fill in the blanks for the specific problem.

Example 1 · Double every element

No filtering. Just transform each element.

doubleAll :: [Int] -> [Int]
doubleAll []     = []
doubleAll (x:xs) = (x * 2) : doubleAll xs

Reading the recursive case:

  • x * 2 is the transformation: what each element becomes.
  • : prepends the transformed head to the result of processing the tail.
  • doubleAll xs recursively processes everything after the head.

Visualizing on [1, 2, 3]:

doubleAll [1, 2, 3]
= 2 : doubleAll [2, 3]
= 2 : 4 : doubleAll [3]
= 2 : 4 : 6 : doubleAll []
= 2 : 4 : 6 : []
= [2, 4, 6]

Example 2 · Keep only evens

No transformation. Just filtering.

keepEvens :: [Int] -> [Int]
keepEvens [] = []
keepEvens (x:xs)
  | even x    = x : keepEvens xs
  | otherwise = keepEvens xs

Reading the recursive case, now with a guard:

  • If the predicate is True, keep x by consing it onto the result.
  • If False, skip x and recurse on the tail only.

Example 3 · Double the evens (filter + transform)

Both at once. This is the shape of every meaningful list processing problem.

doubleEvens :: [Int] -> [Int]
doubleEvens [] = []
doubleEvens (x:xs)
  | even x    = (x * 2) : doubleEvens xs
  | otherwise = doubleEvens xs

Same skeleton. Guard decides "keep or skip." When keeping, apply the transformation x * 2. Otherwise, just recurse.

The filter+transform skeleton
f [] = []
f (x:xs)
  | predicate x = transform x : f xs
  | otherwise   = f xs

This is the shape to memorize. Every recursive version of P.3 problems fits here.

Patterns within patterns

When the list elements are tuples or ADTs, destructure them directly in the pattern.

sumPairs :: [(Int, Int)] -> [Int]
sumPairs [] = []
sumPairs ((a, b) : rest) = (a + b) : sumPairs rest

The pattern ((a, b) : rest) does two jobs at once:

  • Outer: (... : rest), a non-empty list.
  • Inner: (a, b), destructure the first tuple.

His exam's pond problem uses exactly this nested-pattern structure. We'll see it in Part 6.

Try all three in practice.hs
doubleAll :: [Int] -> [Int]
doubleAll []     = []
doubleAll (x:xs) = (x * 2) : doubleAll xs

keepEvens :: [Int] -> [Int]
keepEvens [] = []
keepEvens (x:xs)
  | even x    = x : keepEvens xs
  | otherwise = keepEvens xs

doubleEvens :: [Int] -> [Int]
doubleEvens [] = []
doubleEvens (x:xs)
  | even x    = (x * 2) : doubleEvens xs
  | otherwise = doubleEvens xs

Save, :r, then:

-- [2,4,6,8]
-- [2,4,6,8,10]
-- [4,8,12,16,20]
Part 3 of 7 · Style 2

List comprehensions

Directly inspired by mathematical set-builder notation. If you know Python's [x for x in ...], this is the same idea with different punctuation. Cleanest syntax of the three styles for simple filter+transform problems.

Anatomy

[ x * 2 | x <- xs, even x ]
-- ^^^^^   ^^^^^^^^  ^^^^^^
--   |         |        |
--   |         |        filter: keep x only if this is True
--   |         generator: x comes from xs, one at a time
--   output expression: what each element becomes

Read as: "for each x in xs, where x is even, give me x * 2."

Comparison to Python

PythonHaskell
for| (the pipe starts the conditions)
in<- (read "drawn from")
ifcomma (just a comma)

Side by side

# Python
[x * 2 for x in xs if x % 2 == 0]
-- Haskell
[x * 2 | x <- xs, even x]

Same structure. for → |, in → <-, if → ,.

The four shapes

Shape 1 · Just transform (no filter)

[x * 2 | x <- xs]
-- equivalent to: map (*2) xs

Shape 2 · Just filter (no transform)

[x | x <- xs, x > 10]
-- equivalent to: filter (>10) xs
-- the output expression is just x, unchanged

Shape 3 · Transform + filter

[x * 2 | x <- xs, even x]
-- equivalent to: map (*2) (filter even xs)

Shape 4 · Multiple generators (cartesian product)

[(x, y) | x <- [1, 2, 3], y <- ['a', 'b']]
-- [(1,'a'), (1,'b'), (2,'a'), (2,'b'), (3,'a'), (3,'b')]

Two generators produce every combination. Rightmost generator changes fastest (like nested loops). Shape 4 won't appear on his exam, but good to know it exists.

Multiple filters

You can chain filter conditions with commas.

[x | x <- [1..100], even x, x > 50]
-- [52, 54, 56, 58, 60, ..., 100]

Read: "each x from 1 to 100, where x is even AND x is greater than 50."

Tuple destructuring in generators

This is a huge feature and it's on his exam. You can pattern-match the elements coming out of the generator.

[x + y | (x, y) <- [(1, 2), (3, 4), (5, 6)]]
-- [3, 7, 11]

Instead of binding the whole tuple to a name and using fst/snd, destructure directly in the generator.

His exam's pond:

pond ms = [ even (floor e) | (x, d, e) <- ms, x + d < e ]

(x, d, e) <- ms destructures each 3-tuple in ms as you pull it out. Then the filter x + d < e uses the destructured names directly.

Teaching tip

This tuple destructuring is the #1 thing students miss when first learning list comprehensions. They try to use fst p and snd p everywhere. Show them (a, b) <- and watch the lightbulb.

Play with comprehensions
[x * x | x <- [1..5]]
[x | x <- [1..20], x `mod` 3 == 0]
[x * y | x <- [1..5], y <- [1..5], x /= y]
[a + b | (a, b) <- [(1,10), (2,20), (3,30)]]

Expected outputs:

[1, 4, 9, 16, 25]
[3, 6, 9, 12, 15, 18]
[2, 3, 4, 5, 2, 6, 8, 10, 3, 6, 12, 15, 4, 8, 12, 20, 5, 10, 15, 20]
[11, 22, 33]
Part 4 of 7 · Style 3

Higher-order functions

The most Haskell-idiomatic style: pass functions as arguments to map and filter. We saw these in Lessons 0 and 2; now we formalize how they work together and introduce lambdas, which his exam uses.

map · transform every element

map :: (a -> b) -> [a] -> [b]

Type reads: "takes a function from a to b, and a list of a's, returns a list of b's." It applies the function to every element.

map (*2) [1, 2, 3]         -- [2, 4, 6]
map show [1, 2, 3]         -- ["1", "2", "3"]
map reverse ["abc", "de"]  -- ["cba", "ed"]

filter · keep what passes a test

filter :: (a -> Bool) -> [a] -> [a]

Takes a predicate (function returning Bool) and a list. Returns only the elements where the predicate is True.

filter even [1..10]       -- [2, 4, 6, 8, 10]
filter (> 0) [-3, 5, -1]  -- [5]
filter isUpper "HeLLo"    -- "HLL"

Combining map and filter

Here's the crucial pattern. "Filter then transform" is written right to left because of how function composition reads.

map (*2) (filter even [1..10])
-- step 1: filter even [1..10]  →  [2, 4, 6, 8, 10]
-- step 2: map (*2) [2, 4, 6, 8, 10]  →  [4, 8, 12, 16, 20]

Haskell reads from the inside out. Inner function runs first. So map f (filter p xs) means: first filter, then map.

Common mistake

Writing filter even (map (*2) [1..10]) gives the wrong answer. That doubles everything first (making everything even), then "filters" for even, which keeps everything. Order matters. Always filter first, then transform.

Operator sections (a quick recap)

You've seen these in Lesson 0. They turn an operator into a function.

SectionIs the functionApplied to x
(+1)add 1x + 1
(*2)doublex * 2
(>5)is it > 5?x > 5
(10-)10 minus10 - x
(`div` 2)divide by 2x `div` 2

Operator sections are the cleanest way to pass simple functions to map and filter.

Lambdas · anonymous functions

When the function you want to pass isn't a simple operator, define it inline with a lambda.

\x -> x * x + 1

The backslash \ is trying to look like the Greek letter λ (lambda). Read as: "a function taking x, returning x*x + 1."

Used with map / filter

map (\x -> x * x + 1) [1, 2, 3]
-- [2, 5, 10]

filter (\x -> x > 3 && x < 8) [1..10]
-- [4, 5, 6, 7]

Multi-argument lambdas

\x y -> x + y
-- function of 2 args returning their sum

\(a, b) -> a * b
-- destructures a tuple, multiplies

The last one is key: lambdas can pattern match on their argument. Very useful when working with tuples.

His exam's creek1 fill-in uses a lambda:

creek1 ch as = map (\c -> [c, c, ch]) (filter isAlpha as)

The lambda \c -> [c, c, ch] takes a character c, returns the 3-char list [c, c, ch].

When to use a lambda vs a named helper

For one-off small functions passed to map/filter, use a lambda. For anything used more than once, define it as a named function. His exam expects lambdas in one-off spots.

Try lambdas and HOF combinations
map (\x -> x * 3 + 1) [1, 2, 3, 4]
filter (\x -> x > 5 && even x) [1..20]
map (\(a, b) -> a + b) [(1, 2), (3, 4), (5, 6)]
map (*3) (filter (> 5) [1..10])
Part 5 of 7 · The translation template

The translation template

This is the heart of the lesson. A single mechanical procedure for translating any list-processing function between the three styles. Memorize this and every P.3 question becomes a lookup.

The universal template

For any filter+transform problem, the same three pieces appear in all three styles. Find them once, they slot into each template.

The 1-to-1 template
-- Style 1: RECURSION
f []     = []
f (x:xs)
  | PRED x   = TRANS x : f xs
  | otherwise = f xs
↕ ↕ ↕
-- Style 2: LIST COMPREHENSION
f xs = [ TRANS x | x <- xs, PRED x ]
↕ ↕ ↕
-- Style 3: HIGHER-ORDER FUNCTIONS
f xs = map TRANS (filter PRED xs)

The same two pieces, PRED (filter condition) and TRANS (what each element becomes), appear in all three styles. Find them once and the translation is mechanical.

Translation by example: doubleEvens

Let's translate the same function all three ways, showing the template in action.

Starting from recursion

doubleEvens :: [Int] -> [Int]
doubleEvens [] = []
doubleEvens (x:xs)
  | even x    = (x * 2) : doubleEvens xs
  | otherwise = doubleEvens xs

Identify the pieces:

  • PRED = even x → as a function: even
  • TRANS = x * 2 → as a function: (*2)

Slot into the list comprehension template

doubleEvens xs = [ TRANS x | x <- xs, PRED x ]
-- substitute:
doubleEvens xs = [ x * 2 | x <- xs, even x ]

Slot into the HOF template

doubleEvens xs = map TRANS (filter PRED xs)
-- substitute:
doubleEvens xs = map (*2) (filter even xs)

Done. Same function, three styles, same behavior.

Special cases

No filter (keep everything)

Drop the predicate.

StyleCode
Recursionf [] = [] ; f (x:xs) = TRANS x : f xs
Comprehensionf xs = [ TRANS x | x <- xs ]
HOFf xs = map TRANS xs

No transform (keep elements as-is)

TRANS is just the identity: output x unchanged.

StyleCode
Recursionf [] = [] ; f (x:xs) | PRED x = x : f xs | otherwise = f xs
Comprehensionf xs = [ x | x <- xs, PRED x ]
HOFf xs = filter PRED xs

Recognizing the pieces: a step-by-step method

When your student faces a P.3 conversion, have him follow this procedure:

The conversion procedure
  1. Read the given code, whatever style it's in.
  2. Find PRED. What's the condition for keeping an element?
    • In recursion: the guard that leads to cons-ing.
    • In comprehension: whatever comes after the comma.
    • In HOF: the function passed to filter.
  3. Find TRANS. What does each kept element become?
    • In recursion: whatever gets consed (before the :).
    • In comprehension: the output expression (before the |).
    • In HOF: the function passed to map.
  4. Plug into the target template.

Worked example: convert comprehension to recursion

Given:

f xs = [ x + 1 | x <- xs, x > 10 ]
Conversion trace
1
PRED = x > 10 (the condition after the comma)
2
TRANS = x + 1 (the output expression before |)
3
Plug into recursion template:
f [] = []
f (x:xs)
  | x > 10    = (x + 1) : f xs
  | otherwise = f xs

Worked example: convert HOF to comprehension

Given:

g xs = map reverse (filter (\s -> length s > 3) xs)
Conversion trace
1
PRED = \s -> length s > 3. When applied to x, that's length x > 3.
2
TRANS = reverse. Applied to x: reverse x.
3
Plug into comprehension template:
g xs = [ reverse x | x <- xs, length x > 3 ]
The shortcut to teach

Every problem = (filter) + (transform) + (plug into target template). Once your student does a few, the pattern clicks and translation takes seconds.

Problem 3 · Write pond1 using recursion with patterns

You're given the comprehension version:

pond :: [(Float, Float, Float)] -> [Bool]
pond ms = [ even (floor e) | (x, d, e) <- ms, x + d < e ]

Write the recursive equivalent.

Identify the pieces from the comprehension

Finding PRED and TRANS
1
The comprehension:
[ even (floor e) | (x, d, e) <- ms, x + d < e ]
2
Generator: (x, d, e) <- ms. Each element of ms is a 3-tuple, destructured into x, d, e.
3
PRED = the filter after the comma: x + d < e.
4
TRANS = the output expression: even (floor e).

Slot into the recursion template · with tuple destructuring

The key: since elements are 3-tuples, we destructure them right in the pattern: ((x, d, e) : rest).

pond1 :: [(Float, Float, Float)] -> [Bool]
pond1 [] = []
pond1 ((x, d, e) : rest)
  | x + d < e = even (floor e) : pond1 rest
  | otherwise = pond1 rest

Reading each line:

  • pond1 [] = [], empty list base case.
  • ((x, d, e) : rest), match a non-empty list; destructure the first tuple at the same time.
  • Guard: if the predicate holds, cons even (floor e) onto the recursive result.
  • Otherwise: skip this element, recurse on the rest.
Verify
pond :: [(Float, Float, Float)] -> [Bool]
pond ms = [ even (floor e) | (x, d, e) <- ms, x + d < e ]

pond1 :: [(Float, Float, Float)] -> [Bool]
pond1 [] = []
pond1 ((x, d, e) : rest)
  | x + d < e = even (floor e) : pond1 rest
  | otherwise = pond1 rest
let testData = [(1.0, 2.0, 5.5), (3.0, 1.0, 3.0), (0.0, 0.0, 10.7)]
pond  testData
pond1 testData
-- both return the same result

Practice & cheat sheet

Fresh conversion problems in the same style as his exam. Each asks to translate between two of the three styles. Work them with the template, then reveal the answer.

Practice problems

Problem 1 · Recursion → Comprehension

Rewrite as a list comprehension:

trim :: [Int] -> [Int]
trim [] = []
trim (x:xs)
  | x > 0     = (x - 1) : trim xs
  | otherwise = trim xs
Show answer & explanation

PRED = x > 0. TRANS = x - 1.

trim xs = [ x - 1 | x <- xs, x > 0 ]

Test: trim [-2, 3, 0, 5] → [2, 4].

Problem 2 · Recursion → HOF

Rewrite using map and filter:

shouts :: [String] -> [String]
shouts [] = []
shouts (s:ss)
  | length s > 3 = (s ++ "!") : shouts ss
  | otherwise    = shouts ss
Show answer & explanation

PRED = length s > 3, as a function \s -> length s > 3.
TRANS = s ++ "!", as a function \s -> s ++ "!".

shouts xs = map (\s -> s ++ "!") (filter (\s -> length s > 3) xs)

Test: shouts ["hi", "hello", "yo", "world"] → ["hello!", "world!"].

Problem 3 · Comprehension → Recursion

Rewrite recursively:

upper :: String -> String
upper s = [ toUpper c | c <- s, isAlpha c ]
Show answer & explanation

PRED = isAlpha c. TRANS = toUpper c.

upper :: String -> String
upper [] = []
upper (c:cs)
  | isAlpha c = toUpper c : upper cs
  | otherwise = upper cs

Test: upper "Hello, World!" → "HELLOWORLD".

Problem 4 · Comprehension → HOF

Rewrite using map and filter:

squared :: [Int] -> [Int]
squared xs = [ x * x | x <- xs, x >= 5 ]
Show answer & explanation

PRED = x >= 5, as a section (>=5).
TRANS = x * x. No clean operator section for this one, use a lambda: \x -> x * x.

squared xs = map (\x -> x * x) (filter (>=5) xs)
Problem 5 · HOF → Comprehension

Rewrite as a list comprehension:

result :: [Int] -> [Int]
result xs = map (+10) (filter odd xs)
Show answer & explanation

PRED = odd, applied: odd x.
TRANS = (+10), applied: x + 10.

result xs = [ x + 10 | x <- xs, odd x ]
Problem 6 · Tuple destructuring (comprehension → recursion)

Rewrite recursively:

sums :: [(Int, Int)] -> [Int]
sums ps = [ a + b | (a, b) <- ps, a /= b ]
Show answer & explanation

Key: destructure the tuple in the pattern, like the pond1 exam problem.

sums :: [(Int, Int)] -> [Int]
sums [] = []
sums ((a, b) : rest)
  | a /= b    = (a + b) : sums rest
  | otherwise = sums rest

Test: sums [(1, 2), (3, 3), (4, 5)] → [3, 9].

Problem 7 · Tuple destructuring (recursion → HOF)

Rewrite using map and filter:

trios :: [(Int, Int, Int)] -> [Int]
trios [] = []
trios ((a, b, c) : rest)
  | a + b == c = (a * b * c) : trios rest
  | otherwise  = trios rest
Show answer & explanation

PRED = \(a,b,c) -> a + b == c.
TRANS = \(a,b,c) -> a * b * c.

Both functions take a tuple and destructure it in the lambda.

trios xs = map (\(a,b,c) -> a * b * c)
                (filter (\(a,b,c) -> a + b == c) xs)

Test: trios [(1,2,3), (2,2,5), (3,4,7)] → [6, 84].

Problem 8 · Only filter, no transform

Rewrite as a comprehension and also using HOF:

positives :: [Int] -> [Int]
positives [] = []
positives (x:xs)
  | x > 0     = x : positives xs
  | otherwise = positives xs
Show answer & explanation

TRANS is identity (just x), so it simplifies.

Comprehension:

positives xs = [ x | x <- xs, x > 0 ]

HOF:

positives xs = filter (>0) xs

When there's no transform, the HOF version drops map entirely.

Problem 9 · Only transform, no filter

Rewrite as a comprehension and also using HOF:

lengths :: [String] -> [Int]
lengths [] = []
lengths (s:ss) = length s : lengths ss
Show answer & explanation

No filter (no guard), so PRED is absent.

Comprehension:

lengths xs = [ length s | s <- xs ]

HOF:

lengths xs = map length xs

When there's no filter, drop filter and the condition. Just map.
