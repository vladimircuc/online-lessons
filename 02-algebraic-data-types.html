Part 1 of 7 · Concept

Algebraic data types

Pattern matching in Lesson 1 was reactive, matching against data someone else defined. Now you design your own data types. Custom types let you model a problem precisely, eliminate impossible states at the type level, and write functions that handle every case the compiler can verify.

What's an algebraic data type?

An algebraic data type (ADT) is a user-defined type whose values fall into a fixed set of "shapes," called constructors. You list the shapes, Haskell makes the type.

The simplest kind is an enum, like Bool, which is actually built from exactly this machinery:

data Bool = True | False

That's the real definition. Bool is a type. True and False are its two constructors. The | means "or."

Why it's called "algebraic"

Because types combine like algebra:

  • Sum (OR): a value is this or that. The | in a data definition.
  • Product (AND): a constructor wraps multiple fields. Fields next to each other.

Every ADT is a sum of products. "A Shape is EITHER a Circle with 1 radius, OR a Rectangle with 2 sides, OR a Triangle with 3 sides." You'll see this combo constantly.

Why ADTs matter

In Python you'd model something like a traffic light with strings, constants, or an Enum. Errors are possible: someone writes "Geen" instead of "Green" and nothing complains until runtime.

In Haskell, the compiler tracks the type:

data TrafficLight = Red | Yellow | Green

action :: TrafficLight -> String
action Red    = "stop"
action Yellow = "slow"
action Green  = "go"

Misspell a case and the compiler complains immediately. Forget a case and it warns. Pass the wrong type and it errors. The bugs you'd hunt for hours in Python are impossible here.

Teaching framing

The big shift for a Python/Java student: types encode your problem's structure. You don't use strings and dictionaries to pretend you have a Flower, you define a Flower type and the compiler enforces it. This is the heart of Haskell's "if it compiles, it works" reputation.

Three things you'll learn this lesson

  1. Defining types, both simple enums and constructors that carry data.
  2. Writing functions over them, using pattern matching on each constructor.
  3. Solving his exam's special and yellows problems, step-by-step.

The 4-step template in Part 4 works for every ADT problem you'll see. Your student will internalize it and every P.2 question becomes mechanical.

Part 2 of 7 · Basics

Simple types

Start with the easiest shape: an enum-style type where each constructor is just a tag with no data attached. This is a stepping stone, constructors that hold data come in Part 3.

The syntax

data TypeName = Constructor1 | Constructor2 | Constructor3

Rules:

  • data is a keyword.
  • The type name must start with an uppercase letter.
  • Each constructor name must also start with an uppercase letter.
  • Constructors are separated by |.
  • You can put them on one line or spread across many.

Examples

data TrafficLight = Red | Yellow | Green

data Direction = North | South | East | West

data DayOfWeek = Mon | Tue | Wed | Thu | Fri | Sat | Sun

Now Red, Green, North, Mon, etc. are values. Their type is the type you declared.

The deriving (Show) magic

If you try to print a value of your type in ghci:

ghci> Red
<interactive>:1:1: error: No instance for (Show TrafficLight)
   arising from a use of `print'

Haskell doesn't know how to display your type yet. Fix: add deriving (Show). This tells Haskell to auto-generate a way to turn the value into text.

data TrafficLight = Red | Yellow | Green deriving (Show)

Now ghci> Red prints Red. Every data definition on his exam uses deriving (Show).

Functions over simple types

Write one equation per constructor. Pattern matching picks the right one.

action :: TrafficLight -> String
action Red    = "stop"
action Yellow = "slow down"
action Green  = "go"

opposite :: Direction -> Direction
opposite North = South
opposite South = North
opposite East  = West
opposite West  = East

Notice: no if, no switch. Just one equation per case. This is the ADT pattern-match idiom.

Exhaustiveness

If you forget a case, Haskell warns: "Non-exhaustive patterns in function action". And if someone passes the missing case, your program crashes. Always handle every constructor, or use a wildcard _ as a catch-all.

Try it in practice.hs
data TrafficLight = Red | Yellow | Green deriving (Show)

action :: TrafficLight -> String
action Red    = "stop"
action Yellow = "slow down"
action Green  = "go"

Save, :r, then:

action Red
action Yellow
action Green

Equality with deriving (Eq)

By default you can't compare values of your custom type with == or /=. To enable it, add Eq to the deriving list:

data TrafficLight = Red | Yellow | Green deriving (Show, Eq)

Now Red == Red is True, Red == Green is False.

His exam warns against this

Look at the problem statement carefully: "These types do not belong to the Eq class, so you will not be able to use == or /=." His types don't derive Eq on purpose, to force him to use pattern matching instead of comparison. Teach him to never reach for == on a custom type. Pattern match.

Part 3 of 7 · Real ADTs

Constructors with data

Here's where ADTs become powerful. Each constructor can carry its own data, and different constructors can carry different data. This is how you model real-world problems in Haskell.

The syntax

data TypeName = Constructor1 Field1Type Field2Type
              | Constructor2 OtherFieldType
              | Constructor3

Constructor names followed by the types of the fields they wrap. Each constructor can have zero, one, two, or more fields, independently.

Example: Shapes

data Shape = Circle Double
           | Rectangle Double Double
           | Triangle Double Double Double
           deriving (Show)

Read: "A Shape is either:

  • a Circle wrapping one Double (radius)
  • a Rectangle wrapping two Doubles (width, height)
  • a Triangle wrapping three Doubles (three sides)

Now these are all valid Shape values:

Circle 5.0                  -- a Shape
Rectangle 3.0 4.0           -- a Shape
Triangle 3.0 4.0 5.0        -- a Shape

To create a value, write the constructor followed by its data, space-separated. Same syntax as calling a function (because constructors are functions under the hood).

Functions that destructure constructors

Pattern match on the constructor AND bind names to the data inside.

area :: Shape -> Double
area (Circle r)          = 3.14159 * r * r
area (Rectangle w h)     = w * h
area (Triangle a b c)    =
  let s = (a + b + c) / 2
  in sqrt (s * (s-a) * (s-b) * (s-c))

Each equation matches a different constructor. The names inside the parens bind to the fields.

Add & test
data Shape = Circle Double
           | Rectangle Double Double
           | Triangle Double Double Double
           deriving (Show)

area :: Shape -> Double
area (Circle r)          = 3.14159 * r * r
area (Rectangle w h)     = w * h
area (Triangle a b c)    =
  let s = (a + b + c) / 2
  in sqrt (s * (s-a) * (s-b) * (s-c))

Save, :r, test:

area (Circle 5)
area (Rectangle 3 4)
area (Triangle 3 4 5)
-- 78.53975
-- 12.0
-- 6.0

Constructors can hold any type

A constructor's fields can be any type at all: primitives, lists, tuples, even other ADTs you defined.

data Event = Login String               -- holds a username
           | Purchase String Double     -- item and price
           | Logout
           deriving (Show)

Examples:

Login "vlad"
Purchase "laptop" 899.99
Logout

Nesting types: his exam setup

           deriving (Show)

             | Trim String            -- customer's instructions

Part 4 of 7 · The template

The 4-step template

Every P.2 problem asks you to write a function over an ADT. There's one template that works for all of them. If your student memorizes this, every P.2 question becomes fill-in-the-blanks.

The template

Universal ADT function template
  1. Read the type. List every constructor and what data each wraps.
  2. Write one equation per constructor. Haskell's exhaustiveness guarantee.
  3. Pattern match to destructure. Grab names for the data inside each constructor.
  4. Use guards if needed. Branch further on the data you just destructured.

Walking through an example

Let's write a function describe :: Shape -> String using the template.

Step 1 · Read the type

data Shape = Circle Double
           | Rectangle Double Double
           | Triangle Double Double Double

Three constructors: Circle (one Double), Rectangle (two Doubles), Triangle (three Doubles).

Step 2 · Write one equation per constructor

describe :: Shape -> String
describe ??? = ???    -- Circle
describe ??? = ???    -- Rectangle
describe ??? = ???    -- Triangle

Step 3 · Pattern match to destructure

Bind names to the data inside each constructor.

describe :: Shape -> String
describe (Circle r)       = ???
describe (Rectangle w h)  = ???
describe (Triangle a b c) = ???

Step 4 · Use guards if needed

No branching needed here, just return the description:

describe :: Shape -> String
describe (Circle r)       = "circle of radius " ++ show r
describe (Rectangle w h)  = "rectangle " ++ show w ++ " by " ++ show h
describe (Triangle a b c) = "triangle with sides " ++ show a ++ ", " ++ show b ++ ", " ++ show c

Done. The template works, no guesswork.

When to use guards (step 4)

Add guards when you need to branch on the destructured data.

Example: discount based on size.

discount :: Shape -> Double
discount (Circle r)
  | r > 10    = 0.15
  | otherwise = 0.0
discount (Rectangle w h)
  | w * h > 50 = 0.20
  | otherwise  = 0.05
discount (Triangle _ _ _) = 0.10

One equation per constructor (step 2). Destructure the fields (step 3). Guard on them (step 4).

Matching on specific data inside

Sometimes you want to match not just the constructor but a specific value inside it. You already saw this in Lesson 1, it works in function definitions too.

isYellow :: Flower -> Bool
isYellow Daffodil       = True
isYellow (Rose Yellow)  = True
isYellow (Lily Yellow)  = True
isYellow _              = False

Breakdown:

  • Daffodil, match this exact constructor (no data needed). Returns True.
  • (Rose Yellow), match a Rose holding specifically Yellow. Returns True.
  • (Lily Yellow), match a Lily holding specifically Yellow.
  • _, catch-all. Any other flower (like Rose Blue or Lily Pink) returns False.
Order matters (again)

Just like Lesson 1, patterns are checked top-to-bottom. Put specific cases first, wildcards last. If isYellow _ came first it would catch everything.

Nested destructuring in function definitions

Since an ADT's fields can be any type, including lists and other ADTs, you can destructure multiple layers in one pattern.

firstFlower :: Service -> Maybe Flower
firstFlower (Plant (f:_)) = Just f
firstFlower _             = Nothing

Reading (Plant (f:_)):

  • Outer: Plant (...), a Plant constructor.
  • Inner: (f:_), a non-empty list; first element binds to f.

(Don't worry about Maybe for now, it's another ADT built into Haskell: data Maybe a = Nothing | Just a. You'd rarely use it on this exam.)

The mental shortcut

When your student sees a P.2 problem, ask him to count the constructors first. If the type has 3 constructors, his function needs (at minimum) 3 equations. This prevents the #1 mistake: forgetting a case.

Part 5 of 7 · Exam problem 1

Exam: special

First of two exam functions. special applies the gardening shop's sales promotion to an order. We'll build it step-by-step using the 4-step template, verify against every example, then move to yellows.

The problem statement

From his study guide:

Sales promotion:

  • Anyone buying at least 3 cubic yards of mulch will receive an additional cubic yard for free.
  • Anyone having at least 3 flowers planted will have a pink rose bush planted for free.

The function signature:

special :: Service -> Service

Takes a Service, returns the same Service with the promo applied.

Expected outputs

InputExpected output
special (Mulch 4)Mulch 5
special (Mulch 2)Mulch 2
special (Trim "oak tree in back")Trim "oak tree in back"
special order1
(Plant [Rose Blue, Lily Pink])		Plant [Rose Blue, Lily Pink]
(only 2 flowers, no promo)
special order2
(Plant with 5 flowers)		Plant [Rose Pink, Daffodil, Rose Red, Lily Blue, Rose Yellow, Daffodil]

Applying the template

Step 1 · Read the type

Service has 3 constructors: Mulch Integer, Trim String, Plant [Flower].

Step 2 · Write one equation per constructor

special :: Service -> Service
special (Mulch n)       = ???
special (Trim s)        = ???
special (Plant flowers) = ???

Step 3 · Pattern match to destructure

Already done in step 2. Each constructor has its fields bound to names.

Step 4 · Use guards for the promos

Mulch case: if n >= 3, return Mulch (n + 1), else Mulch n.

special (Mulch n)
  | n >= 3    = Mulch (n + 1)
  | otherwise = Mulch n

Trim case: no promo applies. Return as-is.

special (Trim s) = Trim s

Plant case: if the list has 3+ flowers, prepend a pink rose bush.

special (Plant flowers)
  | length flowers >= 3 = Plant (Rose Pink : flowers)
  | otherwise           = Plant flowers

Recall from Lesson 0: : prepends an element to a list. Rose Pink : flowers is a new list with Rose Pink at the front, followed by all original flowers.

The complete function

special :: Service -> Service
special (Mulch n)
  | n >= 3    = Mulch (n + 1)
  | otherwise = Mulch n
special (Trim s) = Trim s
special (Plant flowers)
  | length flowers >= 3 = Plant (Rose Pink : flowers)
  | otherwise           = Plant flowers
Verify each example

Add special to practice.hs, save, :r, then:

special (Mulch 4)
-- Mulch 5

special (Mulch 2)
-- Mulch 2

special (Trim "oak tree in back")
-- Trim "oak tree in back"

special order1
-- Plant [Rose Blue,Lily Pink]

special order2
-- Plant [Rose Pink,Daffodil,Rose Red,Lily Blue,Rose Yellow,Daffodil]

Walking through special order2 in detail

Let's trace this one because it uses every feature.

Trace · special order2
1
order2 = Plant [Daffodil, Rose Red, Lily Blue, Rose Yellow, Daffodil]. Pattern (Plant flowers) matches, binding flowers = [Daffodil, Rose Red, Lily Blue, Rose Yellow, Daffodil].
2
Evaluate first guard: length flowers >= 3. length = 5, 5 >= 3 is True. Take this branch.
3
Build the result: Plant (Rose Pink : flowers). The cons operator prepends Rose Pink to the existing list.
4
Rose Pink : [Daffodil, Rose Red, Lily Blue, Rose Yellow, Daffodil]
= [Rose Pink, Daffodil, Rose Red, Lily Blue, Rose Yellow, Daffodil].
ResultPlant [Rose Pink, Daffodil, Rose Red, Lily Blue, Rose Yellow, Daffodil]

Common mistakes

Forgetting to re-wrap with the constructor

A common error is writing Rose Pink : flowers by itself. But that's a [Flower], not a Service. The function signature says we must return a Service, so we need to wrap the list back in Plant: Plant (Rose Pink : flowers).

Using > instead of >=

The problem says "at least 3," which is >= 3, not > 3. If your student writes > 3, the example special (Mulch 3) would fail. Always re-read the problem wording, "at least" includes the boundary.

Trying to use ==

Reminder from Part 2: the types don't derive Eq. If your student writes something like flowers == [] he'll get a compile error. Use pattern matching ([] as a pattern) or length checks instead.

Part 6 of 7 · Exam problem 2

Exam: yellows

Second exam function. yellows extracts all yellow flowers from an order. It introduces a helper function and shows when and why to pattern match on constructors' specific contents, not just the constructor type.

The problem statement

From the study guide:

yellows order produces a list that contains all of the yellow flowers included in order.

Signature:

yellows :: Service -> [Flower]

Expected outputs

InputExpected output
yellows (Mulch 4)[] (not a plant order)
yellows (Trim "hello")[] (not a plant order)
yellows order1[] (no yellows in Rose Blue, Lily Pink)
yellows order2[Daffodil, Rose Yellow, Daffodil]

The two sub-problems

  1. Which flowers count as yellow? Daffodils (all yellow), Rose Yellow, Lily Yellow. Write a helper for this.
  2. Extract the list of flowers from the service, then filter the yellows.

Step 1 · The helper: isYellow

We already saw this in Part 4.

isYellow :: Flower -> Bool
isYellow Daffodil       = True
isYellow (Rose Yellow)  = True
isYellow (Lily Yellow)  = True
isYellow _              = False

Notice the use of specific-data matching: (Rose Yellow) only matches a Rose whose color is exactly Yellow. Any other color falls through to the wildcard.

Teaching detail

Ask your student to list every Flower value that returns True: Daffodil, Rose Yellow, Lily Yellow. That's it. Every other flower (Rose Blue, Lily Pink, Rose Red, etc.) returns False via the wildcard.

Step 2 · The main function using the template

Template step 1 · Read the type

3 Service constructors: Mulch, Trim, Plant.

Template step 2 · One equation per constructor

But wait: Mulch and Trim both have zero yellow flowers, both return []. We can combine them with a wildcard.

yellows :: Service -> [Flower]
yellows (Plant flowers) = ???
yellows _               = []

The Plant case must come first so it's tried before the wildcard.

Template step 3 · Pattern match & process

We have flowers, a [Flower]. We want only the yellow ones. That's exactly what filter does.

yellows (Plant flowers) = filter isYellow flowers
yellows _               = []

Done. filter isYellow flowers keeps each flower where isYellow returns True.

The complete solution

yellows :: Service -> [Flower]
yellows (Plant flowers) = filter isYellow flowers
yellows _               = []

isYellow :: Flower -> Bool
isYellow Daffodil       = True
isYellow (Rose Yellow)  = True
isYellow (Lily Yellow)  = True
isYellow _              = False
Verify every example

Add both functions, save, :r, then:

yellows (Mulch 4)
-- []

yellows (Trim "hello")
-- []

yellows order1
-- []

yellows order2
-- [Daffodil,Rose Yellow,Daffodil]

Walking through yellows order2

Trace · yellows order2
1
order2 = Plant [Daffodil, Rose Red, Lily Blue, Rose Yellow, Daffodil]. Matches (Plant flowers), binds flowers = [Daffodil, Rose Red, Lily Blue, Rose Yellow, Daffodil].
2
Evaluate filter isYellow flowers. Walk through each element:
· Daffodil → isYellow Daffodil = True, keep.
· Rose Red → falls through to wildcard, False, drop.
· Lily Blue → wildcard, drop.
· Rose Yellow → matches (Rose Yellow), True, keep.
· Daffodil → keep.
3

Result[Daffodil, Rose Yellow, Daffodil]

Alternative without filter: explicit recursion

To deepen understanding, here's the same function written recursively, no filter. This connects directly to Lesson 3 (converting between styles).

yellows :: Service -> [Flower]
yellows (Plant flowers) = collectYellows flowers
yellows _               = []

collectYellows :: [Flower] -> [Flower]
collectYellows [] = []
collectYellows (f:fs)
  | isYellow f = f : collectYellows fs
  | otherwise  = collectYellows fs

Same behavior. filter is just a shorthand for this recursive pattern. If your student is still fuzzy on higher-order functions, start with this version, then show that filter isYellow does the same thing.

The big unlock

Every P.2 problem follows: pattern match the outer ADT → do list processing on the interior. Once your student sees this two-layer structure, the exam becomes mechanical. Outer: ADT pattern. Inner: whatever you learned about lists in Lessons 0 & 1 (and 3).

Practice problems

Problem 1 · Simple ADT function

Write sizeLabel :: Size -> String that returns "S", "M", or "L" for each size.

Show answer & explanation 
sizeLabel :: Size -> String
sizeLabel Small  = "S"
sizeLabel Medium = "M"
sizeLabel Large  = "L"

Template in action: one equation per constructor. Size has 3, so 3 equations. No data inside, so no destructuring needed.

Problem 2 · Constructor with data

Write price :: Drink -> Double. Coffee: Small $3, Medium $4, Large $5. Tea: always $3. Water: $1. Smoothie: $6 + $0.50 per ingredient.

Show answer & explanation 
price :: Drink -> Double
price (Coffee Small)  = 3
price (Coffee Medium) = 4
price (Coffee Large)  = 5
price (Tea _)         = 3
price Water           = 1
price (Smoothie ingredients) = 6 + 0.5 * fromIntegral (length ingredients)

Notes:

  • Coffee has 3 sub-cases, one per Size. Using specific-data matching.
  • Tea ignores the size (always $3). Use _ to match any Size.
  • Water has no data, no parens needed.
  • Smoothie destructures the list of ingredients. fromIntegral converts length's Int result to a Double.
Problem 3 · Nested pattern

Write isCold :: Drink -> Bool that returns True for Water and any Smoothie, False for any Coffee or Tea.

Show answer & explanation 
isCold :: Drink -> Bool
isCold Water        = True
isCold (Smoothie _) = True
isCold _            = False

Specific cases first (True), wildcard catches everything else. The wildcard in (Smoothie _) ignores the ingredients list since we only care about the constructor.

Problem 4 · Specific sub-value match

Write hasLargeCoffee :: Drink -> Bool that returns True only for a Large Coffee, False for anything else.

Show answer & explanation 
hasLargeCoffee :: Drink -> Bool
hasLargeCoffee (Coffee Large) = True
hasLargeCoffee _              = False

Match the specific combination. The pattern (Coffee Large) requires both the constructor AND the specific Size value. Everything else falls through to the wildcard.

Problem 5 · Using a helper + list processing

Write coldDrinks :: [Drink] -> [Drink] that returns only the cold drinks from the list. Use filter and the isCold helper from Problem 3.

Show answer & explanation 
coldDrinks :: [Drink] -> [Drink]
coldDrinks ds = filter isCold ds

filter isCold ds keeps elements where isCold returns True. Identical pattern to yellows from the exam walkthrough, just with a different helper.

Test: coldDrinks [Water, Coffee Small, Smoothie ["banana"]] → [Water, Smoothie ["banana"]].

Problem 6 · Order type (nested ADT)

Write orderPrice :: Order -> Double. A Dine adds $1 service charge. A Takeaway has no extra charge. A Combo is the sum of both drinks minus $2 (combo discount). Use the price function from Problem 2.

Show answer & explanation 
orderPrice :: Order -> Double
orderPrice (Dine d)       = price d + 1
orderPrice (Takeaway d)   = price d
orderPrice (Combo d1 d2)  = price d1 + price d2 - 2

One equation per Order constructor. Each destructures the Drink(s) inside. Then we call the existing price helper, exactly the kind of composition Haskell encourages.

Test: orderPrice (Combo (Coffee Large) Water) → 5 + 1 - 2 = 4.0.

Problem 7 · Guards inside an ADT equation

Write discount :: Drink -> Double that returns 0.10 (10% off) for a Smoothie with 5 or more ingredients, 0.05 for a Large Coffee, and 0 for everything else.

Show answer & explanation 
discount :: Drink -> Double
discount (Smoothie ingredients)
  | length ingredients >= 5 = 0.10
  | otherwise               = 0
discount (Coffee Large)     = 0.05
discount _                  = 0

Uses guards for the Smoothie case (step 4 of the template). Specific sub-value match for (Coffee Large). Wildcard for everything else.

Problem 8 · Transform and rewrap

Write supersize :: Drink -> Drink. If it's a Coffee or Tea, upgrade to Large. Water and Smoothie unchanged.

Show answer & explanation 
supersize :: Drink -> Drink
supersize (Coffee _)     = Coffee Large
supersize (Tea _)        = Tea Large
supersize d              = d

For Coffee and Tea, ignore the current size (_) and rebuild with Large. For others, bind the whole drink to d and return it unchanged.

Important: supersize d = d as a catch-all. This is the ADT version of "no change."
