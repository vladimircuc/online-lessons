    QUESTION 01
    Dynamic Dispatch

    The SceneItAll IoT hierarchy from lecture:

    IoTDevice <<interface>>
└── BaseIoTDevice <<abstract>>
    ├── Fan
    └── Light <<abstract>>
        ├── SwitchedLight
        └── DimmableLight
            └── TunableWhiteLight
    Light light = new TunableWhiteLight("living-room", 2700, 100);
((DimmableLight) light).turnOn();

    TunableWhiteLight overrides turnOn(). Which version is called?

    Answer: D · TunableWhiteLight's turnOn()
    Concept: Dynamic dispatch, runtime type wins

    What's happening, step by step:

    Line 1 creates a TunableWhiteLight object on the heap, but stores the reference in a variable declared as Light. The compiler only lets you call methods that exist on Light through that variable, that's the declared-type check.

    Line 2 does two things. (DimmableLight) light is a cast, a compile-time assertion. It does NOT modify the object in memory, it just tells the compiler "trust me, this is a DimmableLight, let me call DimmableLight methods on it." The object on the heap is still a TunableWhiteLight, nothing has changed about it.

    Then .turnOn() is called. Java uses dynamic dispatch for instance methods, it looks at the actual object in memory (runtime type), walks up the vtable, and picks the most-derived override. The runtime type is TunableWhiteLight, which overrides turnOn(), so that version runs.

    The cast was only needed to satisfy the compiler if turnOn() is declared on DimmableLight but not on Light. At runtime the cast is invisible.

    Why the others are wrong

    A. A cast does not change the runtime type. The object stored on the heap was created with new TunableWhiteLight(...), and no cast rewrites bytes in memory. Casts are purely a compile-time permission.

    B. Java uses dynamic, not static, dispatch for instance methods by default. Static dispatch only applies to static, private, or final methods.

    C. The cast is legal. TunableWhiteLight IS-A DimmableLight (subclass). If the object weren't a DimmableLight we'd get a ClassCastException at runtime, but it is, so this is a valid upcast-then-downcast chain.

    Most common wrong pick, A. Students read "cast to DimmableLight" and assume the cast magically "converts" the object. They're thinking of it like primitive casts (int to double), which actually do change the value. Object casts are different, they never change the object, they just pass a type check.
    Exam tip: Dispatch always follows the actual object. Declared types and casts control what you're allowed to call; runtime types decide which version runs.
    QUESTION 02
    Static Context
    public class Example {
    int x = 10;
    public static void printX() {
        System.out.println(x);
    }
}

    Why doesn't this compile?

    Answer: B · Static methods cannot reference instance variables directly
    Concept: Static context has no `this`

    The key distinction:

    A static method belongs to the class itself, not to any particular instance. You can call it without creating an object, Example.printX(). Because it's not tied to any instance, it has no implicit this reference.

    An instance field like int x = 10 only exists on an instance. Every new Example() creates a fresh x in that object's memory. Reading x always means "read this.x", the x of some specific instance.

    So when printX() tries to reference x, the compiler asks: "which instance's x?" There isn't one, printX() wasn't called on an instance, and it has no this. Compile error: "non-static variable x cannot be referenced from a static context."

    Two legal fixes:

    1. Make x static (shared across all instances, belongs to the class): static int x = 10;

    2. Pass in or hold an instance: public static void printX(Example e) { System.out.println(e.x); }

    Why the others are wrong

    A. Access modifiers (public/private) are a separate axis from static vs instance. Making x public wouldn't help, a static method still has no instance to read from. Access control is about WHO can see; static-vs-instance is about WHERE the field lives.

    C. System.out.println is heavily overloaded and accepts int directly. Printing primitives is never a compile error.

    D. final means "can't be reassigned after initialization", it has nothing to do with static-vs-instance access rules.

    Most common wrong pick, A. Beginners learn "public means accessible" and over-generalize. But accessibility (who can see it) is independent of where it lives (static or instance). A static method has no path to an instance field regardless of its access modifier.
    Exam tip: If a method is static and it touches non-static state, it won't compile. Either make the state static too, or pass an instance.
    QUESTION 03
    Collections Big-O

    Version A uses ArrayList<Device> and tests membership with a for-each loop. Version B uses HashSet<Device> and contains(). What's the Big-O?

    Answer: B · A is O(n), B is O(1)
    Concept: Structure-specific complexity

    ArrayList behavior: An ArrayList is a backing array. To check membership with a for-each loop, you walk from index 0 and compare each element. Worst case, you scan the whole array, linear time, O(n). Even the average case is O(n/2), still O(n).

    HashSet behavior: A HashSet is backed by a hash table. contains(x) computes x.hashCode(), goes to a bucket index, and compares against the (usually very few) entries in that bucket. Average-case: constant time, O(1). Worst case (many collisions) degrades to O(n), but with a reasonable hashCode the expected time is constant.

    Takeaway: Choice of data structure is the #1 determinant of performance. Switching from ArrayList to HashSet for membership testing takes your lookup from "walk the whole list" to "compute one hash and check one bucket."

    Why the others are wrong

    A. Only the ArrayList is O(n); HashSet lookup is constant-time. Treating them as equivalent ignores the entire point of choosing a hash-based structure.

    C. O(log n) is TreeSet / TreeMap (binary search on a balanced tree). HashSet is O(1), not O(log n).

    D. Completely fabricated. JIT does many great things (inlining, escape analysis, loop unrolling) but never rewrites an algorithm from linear to constant time. JIT can't change the fundamental complexity of your code.

    Most common wrong pick, D. "JIT will optimize it" appeals to students who don't fully understand JIT. JIT is about micro-optimization (faster machine code), not algorithmic transformation. If your algorithm is O(n), JIT makes each step faster but the total work still scales linearly.
    Exam tip: Memorize the core lookup table: ArrayList contains = O(n), HashSet contains = O(1), TreeSet contains = O(log n).
    QUESTION 04
    Nullability

    In a @NullMarked package:

    public String formatIngredient(@Nullable String prefix, String name) {
    if (prefix == null) {
        return name;
    }
    return prefix + " " + name;
}

    Which is correct?

    Answer: A · name is assumed non-null
    Concept: @NullMarked inverts the default

    What @NullMarked does:

    By default in raw Java, every reference type is implicitly nullable, any parameter might be null. @NullMarked (at the package or class level) flips this default, inside that scope, every unannotated reference is treated as non-null. You must opt IN to nullability with an explicit @Nullable.

    Applied to this method:

    The package is @NullMarked. The parameter prefix is explicitly @Nullable, callers may pass null, and the body correctly handles that with a null check. The parameter name has no annotation, so the tool (Checker Framework, NullAway, IntelliJ, whatever's enforcing it) treats it as non-null. If a caller passes null for name, the static analyzer flags it as a likely bug.

    Important nuance: These annotations feed a static checker, they don't add runtime checks. No NPE is injected by the annotation itself. The whole point is catching mistakes at compile time without runtime cost.

    Why the others are wrong

    B. Without @NullMarked, Java's default is "everything could be null." WITH @NullMarked, the default flips to "nothing is null unless marked." So @Nullable is necessary, not redundant.

    C. Annotations are compile-time metadata, they don't inject any runtime checks or throw NPEs by themselves. The NPE happens if you dereference a null, but it's the dereference that throws, not the annotation.

    D. Annotations are per-parameter. Marking one parameter @Nullable has no effect on other parameters in the same signature.

    Most common wrong pick, C. Students confuse compile-time tooling with runtime behavior. Annotations are inert by themselves, they're just data that static analysis tools read. Knowing this distinction is what the question tests.
    Exam tip: In a @NullMarked package, unannotated = non-null. @Nullable is opt-in. Analysis happens at compile time, no runtime enforcement.
    QUESTION 05
    Specifications

    Why can you use HashMap without reading its 2000-line implementation?

    Answer: B · Specifications enable chunking
    Concept: Abstraction and specification

    The big idea, specifications are contracts.

    A specification tells you two things: what you need to bring in (preconditions, parameter types, state requirements) and what you'll get out (postconditions, return value, exceptions, side effects). If a class has a clear spec, you never need to read its implementation, you just trust the contract.

    For HashMap, the spec says: put(k, v) stores the pair, returns the previous value for that key or null; get(k) returns the value if present or null; containsKey(k) is amortized O(1). That's enough. You can use HashMap while knowing nothing about its internal resizing, treeification-at-collision-threshold, bucket hashing, etc.

    This is called "chunking" in human cognition. Just like you don't think of a word as a sequence of letters once you learn to read, you don't think of map.put(k, v) as a series of internal operations. The specification lets the class become a single unit in your mental model.

    Modularity, information hiding, and readability all depend on this principle. When specifications are clear, developers can work on big systems without knowing how every piece is implemented.

    Why the others are wrong

    A. Java does not inline standard library classes at compile time. JIT may inline at runtime, but that's an optimization, not a reason you don't need to read the code.

    C. Private fields are about encapsulation (who can access), not about whether you need to read the code. You can read HashMap's source on GitHub; the private modifiers don't hide it from a human reader.

    D. Flat wrong. TreeMap has ordering guarantees HashMap doesn't. LinkedHashMap preserves insertion order. Different Map implementations behave very differently.

    Most common wrong pick, D. Students learning Collections sometimes think an interface guarantees uniform behavior across implementations. It doesn't. Map specifies the operations, but each implementation's spec adds its own guarantees.
    Exam tip: Specifications are why software is buildable at scale. "Read the spec, trust the spec, ignore the implementation."
    QUESTION 06
    Comparator Chaining
    List lights = new ArrayList<>(List.of(
    new DimmableLight("bedroom", 80),
    new DimmableLight("kitchen", 30),
    new DimmableLight("bedroom", 50)
));
lights.sort(Comparator.comparing(DimmableLight::getId)
    .thenComparingInt(DimmableLight::getBrightness));

    Assuming toString() is "id/brightness", what's printed?

    Answer: A · [bedroom/50, bedroom/80, kitchen/30]
    Concept: Comparator.comparing().thenComparing()

    Step-by-step sort:

    1. Primary sort, by id (alphabetical, String.compareTo). "bedroom" < "kitchen", so both bedroom entries come before the kitchen entry.

    2. Tiebreaker, by brightness (ascending int). Both bedroom entries tie on id. Among ties, thenComparingInt(::getBrightness) breaks by brightness ascending. 50 < 80, so bedroom/50 comes before bedroom/80.

    3. Final order: [bedroom/50, bedroom/80, kitchen/30].

    How chaining works: Comparator.comparing(X) builds a comparator on X. .thenComparing(Y) or .thenComparingInt(Y) returns a new comparator that first checks X; if X is equal (ties), then checks Y. So "thenComparing" only kicks in when the preceding comparator returns 0.

    Why the others are wrong

    B. This order [bedroom/80, bedroom/50, kitchen/30] preserves the original insertion order within the ties. The chained comparator DOES re-sort within ties, it doesn't fall back to insertion order.

    C. Sorted by brightness primary (30, 50, 80). That would only happen if the brightness comparator were first. Our primary is id, so bedrooms come before kitchen.

    D. This doesn't follow either the id or brightness ordering consistently, it's just plausible-looking noise.

    Most common wrong pick, B. Students forget that thenComparingInt is not "preserve insertion order among ties", it actively sorts by the second key. Java sorts ARE stable (truly equal elements keep relative order), but only for elements that are truly equal under the full comparator. Here the two bedrooms aren't equal, they tie on id but differ on brightness.
    Exam tip: .thenComparing = "after ties on the first key, break by this." It fires only when the prior comparator returns 0.
    QUESTION 07
    Refactoring for Testability
    devices.stream()
    .filter(d -> d.isConnected() && d.getBattery() < 20
        && !d.getType().equals("wired")
        && d.getLastSeen().isBefore(cutoff))
    .collect(Collectors.toList());

    Best refactor for testability?

    Answer: A · Extract to a named method
    Concept: Testability, controllability

    The problem with the current code:

    The filter logic is a multi-condition lambda buried inside a stream pipeline. To test whether a specific Device satisfies this predicate, you'd have to: (1) construct a list with that Device in it, (2) run the whole stream, (3) check if the Device made it through. That's an integration test for what should be a tiny unit check.

    The refactor:

    public static boolean isLowBatteryWirelessStale(Device d, Instant cutoff) {
    return d.isConnected()
        && d.getBattery() < 20
        && !d.getType().equals("wired")
        && d.getLastSeen().isBefore(cutoff);
}

    Then: .filter(d -> isLowBatteryWirelessStale(d, cutoff)).

    Benefits:

    • Controllability: your test can call isLowBatteryWirelessStale(mockDevice, cutoff) directly with crafted inputs.
    • Observability: the boolean return is directly assertable.
    • Readability: the name documents intent, the filter reads like English.
    • Reusability: you can apply the same predicate elsewhere.
    Why the others are wrong

    B. toUnmodifiableList() affects the output collection's mutability. It has nothing to do with testability of the filtering logic.

    C. Splitting into multiple .filter() calls can help readability, but each filter is still an inline lambda, each clause is still not independently callable from a test.

    D. Anonymous classes are more verbose than lambdas but functionally equivalent. Logic is still buried inside the pipeline, no testability improvement.

    Most common wrong pick, C. Splitting filters reads cleaner, so it feels like progress. But readability ≠ testability. Each mini-filter still lives inside the stream and can't be invoked as a standalone unit.
    Exam tip: Named methods are testable; lambdas hidden in pipelines are not. If you want to unit-test a predicate, give it a name.
    QUESTION 08
    Fragile Base Class

    LoggingList extends ArrayList overrides both add() and addAll() to increment an addCount. Both JUnit tests pass. What's the problem?

    Answer: B · Fragile base class problem
    Concept: Inheritance coupling to parent internals

    What "fragile base class" means:

    A subclass that overrides methods on a parent class can become dependent on the parent's internal call patterns. If the parent is later changed, the subclass can silently break, even though neither the parent's spec nor the subclass's spec changed.

    Applied here:

    LoggingList's spec says, "addCount = total number of items added." Current implementation:

    @Override
public boolean add(E e) {
    addCount++;
    return super.add(e);
}
@Override
public boolean addAll(Collection<? extends E> c) {
    addCount += c.size();
    return super.addAll(c);
}

    Suppose in a future Java release, ArrayList.addAll is rewritten to internally call this.add() for each element. Then a single addAll(list) call would increment addCount twice per element (once in the override of addAll, once per add). The spec silently breaks, and existing tests may still pass if they were written against the old implementation.

    The deeper issue: the subclass's correctness depends on implementation details of the parent that aren't guaranteed by the parent's spec. Inheritance coupling to implementation, not to interface.

    Why the others are wrong

    A. Making addCount static would share count across ALL LoggingList instances, a semantic change, not a correctness fix. And it's not the issue.

    C. Interface Segregation Principle is about not forcing clients to depend on interfaces they don't use. Overriding two parent methods has nothing to do with ISP.

    D. LSP says subclasses should be substitutable for their base. LoggingList IS substitutable, it still behaves like an ArrayList externally. The fragile base issue is separate from LSP.

    Most common wrong pick, D. Both "fragile base class" and "LSP violation" involve inheritance, so students conflate them. Distinction: LSP is about EXTERNAL behavior (can I substitute?); fragile base is about INTERNAL coupling (does my correctness depend on the parent's implementation?).
    Exam tip: Prefer composition over inheritance when you need to intercept behavior. Wrap the list, don't extend it.
    QUESTION 09
    Mockito verify
    when(mockSensor.readTemperature("livingRoom")).thenReturn(65.0);
ThermostatController controller = new ThermostatController(mockSensor, mockHVAC);
controller.adjustToTargetTemperature(72.0, "livingRoom");
verify(mockHVAC).activate("livingRoom");

    What does the verify line do?

    Answer: A · Asserts the call happened exactly once
    Concept: Mockito verify vs when/thenReturn

    Two core Mockito operations:

    1. Stubbing (setup): when(mock.method(args)).thenReturn(value). Says "the NEXT time method is called with these args, return this value." Configures the mock's behavior BEFORE the code under test runs.

    2. Verification (assertion): verify(mock).method(args). Says "I expect method was called with these args during the test." Past-tense assertion after the code under test has run.

    Applied here:

    The test first stubs the sensor to return 65.0 (mock behavior setup). Then it runs adjustToTargetTemperature(72.0, "livingRoom") on the real controller. The controller, seeing that current is 65 and target is 72, should call hvac.activate("livingRoom"). The verify line asserts this call happened.

    Default multiplicity: verify(mock).method(args) with no times(n) asserts exactly once. If the method was called zero times or two+ times with those args, the verify fails.

    Other multiplicities: verify(mock, times(3)).method(), verify(mock, never()).method(), verify(mock, atLeast(1)).method().

    Why the others are wrong

    B. That's what when/thenReturn does, stubbing future behavior. verify has the opposite direction: it checks past events.

    C. Mockito doesn't observe constructor calls that way. verify only applies to method calls on the mock after it's been created.

    D. Mocks are fake objects; they don't interact with real systems. Nothing about verify hooks into external HVAC hardware.

    Most common wrong pick, B. Students mix up when and verify because they both look like "method-on-a-mock" syntax. Memory aid: when = setup future, verify = assert past.
    Exam tip: when → thenReturn (setup). verify (assert). Default verify = exactly once. Know these patterns cold.
    QUESTION 10
    Testability · Static Calls
    public class ScheduledTask {
    public void runIfDue() {
        if (TimeUtils.isBusinessHours()) {
            doWork();
        }
    }
    private void doWork() { /* ... */ }
}

    Dev wants to test that doWork() is NOT called outside business hours. What's the fundamental testing problem?

    Answer: B · Static calls can't be swapped in tests
    Concept: Designing for testability · DI over static

    The core issue:

    To test both branches (business hours vs not), you need to control what isBusinessHours() returns during the test. With a static call like TimeUtils.isBusinessHours(), there's no seam to inject a fake, the class is reaching into a global, hard-wired dependency. Your test is stuck with whatever the real system clock returns at the moment the test runs.

    Why static calls are a testability anti-pattern:

    Static calls create hidden dependencies. Looking at the class's constructor or fields, you can't tell that ScheduledTask depends on the clock. The dependency is baked into the method body, and swapping it out requires either bytecode manipulation (PowerMock) or language-level tricks. Regular Mockito cannot mock static methods out of the box.

    The refactor:

    public class ScheduledTask {
    private final Clock clock;
    public ScheduledTask(Clock clock) { this.clock = clock; }
    public void runIfDue() {
        if (clock.isBusinessHours()) { doWork(); }
    }
}

    Now the test can pass a fake Clock and control both branches deterministically.

    Why the others are wrong

    A. Private methods aren't a blocker, you test them indirectly through public callers. Calling runIfDue() and checking its effects is the correct way to test doWork(). Privacy ≠ untestable.

    C. Mockito doesn't need an explicit constructor. It uses reflection and bytecode generation; default constructors are fine.

    D. Void methods are tested by observing their side effects (state changes, calls on collaborators via verify). Return type isn't the blocker.

    Most common wrong pick, A. "Private is untestable" is a beginner misconception. You don't test private methods directly, you test them through their public callers. The real testability killer here is the STATIC call.
    Exam tip: If you see a static call to something time/env/IO-dependent inside a method, that's a testability red flag. Inject the dependency instead.
    QUESTION 11
    Modular Monolith

    A team adopts a modular monolith where Submissions and Grading communicate only through public APIs. Primary benefit vs an unstructured monolith?

    Answer: C · Preserves boundaries for future extraction
    Concept: Architectural option value

    What a modular monolith is:

    Still one deployable process, one JAR, one WAR, whatever, but the codebase inside is partitioned into modules (Submissions, Grading, Users, etc.), and those modules communicate only through defined public APIs. Internal classes of one module aren't reachable from another. All the code runs in the same process, but the architecture looks like microservices from the inside.

    What you gain:

    The biggest benefit is architectural option value. If Grading ever needs to scale independently (say, it's compute-heavy and needs 10x the cores), you can extract it into its own service. The boundaries are already there, you've been programming as if the modules were separate. Extracting becomes a mechanical exercise (wrap the API in HTTP, move the code, switch the callers), not a massive rewrite.

    Compare to an unstructured monolith, where Submissions code reaches into Grading's internal data classes, shares a DB table, and calls 20 Grading methods from all over. Extracting is nearly impossible, you'd first have to do the boundary work (modularization) before you could even start the extraction.

    Why the others are wrong

    A. Network latency isn't a factor either way, both are in-process calls. And schemas can't "evolve independently" yet, they share one DB and one deployment.

    B. That's a microservices benefit. A monolith (modular or not) is one deploy unit. Modify Grading, redeploy everything.

    D. Also a microservices benefit (blast radius isolation). In a monolith, an uncaught exception or OOM can crash the whole process regardless of module structure.

    Most common wrong pick, B. Students conflate modular monolith with microservices. They both involve module boundaries, the key difference is DEPLOYMENT granularity. Monolith = one deploy unit. Microservices = many deploy units.
    Exam tip: Modular monolith = clean boundaries with option to extract later. No deployment isolation, no network overhead, no fault isolation.
    QUESTION 12
    Distributed Fallacies

    Pawtograder batches 100 test results into one submitFeedback() call instead of 100 individual calls. Which distributed fallacy most directly motivates this?

    Answer: A · Latency is zero
    Concept: 8 Fallacies of Distributed Computing

    The 8 classic fallacies (memorize these):

    1. The network is reliable
    2. Latency is zero
    3. Bandwidth is infinite
    4. The network is secure
    5. Topology doesn't change
    6. There is one administrator
    7. Transport cost is zero
    8. The network is homogeneous

    Why "Latency is zero" fits this question:

    Every network round-trip has an unavoidable latency, at minimum the speed-of-light delay plus protocol overhead. Even on a fast LAN, that's typically 1 to 10 ms per round trip. Cross-region, it's 50 to 200 ms. If you do 100 sequential calls, you pay that cost 100 times. 100 × 50 ms = 5 seconds wasted on round trips alone.

    Batching collapses those 100 round trips into one. You still send all the same data, but you pay the latency cost ONCE. This is the textbook justification for batch APIs.

    Why the others are wrong

    B. Admin/policy variation can cause failures, but it's not the direct motivation for batching. Batching helps regardless of who owns what on the path.

    C. Security is important and more round trips = more attack surface, but that's minor. Batching is primarily a latency optimization.

    D. Path heterogeneity is real but doesn't get solved by batching. Batching reduces round-trip count, not path variability.

    Most common wrong pick, C. Security is a valid concern, students see "100 calls" and think "more attack surface." But the PRIMARY reason to batch is performance, not security. If security were the driver, you'd encrypt, not batch.
    Exam tip: Batching = latency fix. When you see "N calls collapsed into 1", the fallacy is almost always "latency is zero."
    QUESTION 13
    FaaS / Serverless

    Using FaaS instead of managing your own server, what do you still control?

    Answer: B · The algorithm inside the function
    Concept: FaaS, what you bring vs what the platform handles

    FaaS in one sentence: You bring the function (code that takes input, produces output). The platform handles everything else.

    What the platform owns in FaaS:

    • Physical hardware (data centers, servers, cooling)
    • OS (you don't install packages, patch kernels, configure firewalls)
    • Runtime (Node.js / Python / Java version, they pick)
    • Scaling (they spin up containers/instances as traffic arrives)
    • Load balancing, request routing, networking
    • Billing per-invocation (not per-hour-of-server)

    What you own:

    • Your function's code, the algorithm, dependencies, business logic
    • Configuration of memory/timeout limits (within the platform's bounds)
    • Triggers (HTTP endpoint? S3 event? Cron?)

    Real examples: AWS Lambda, Google Cloud Functions, Azure Functions, Cloudflare Workers.

    Why the others are wrong

    A. OS config is hidden from you in FaaS. You can pick a runtime version, but you don't SSH into a machine.

    C. You never see the hardware. It's shared infrastructure you don't manage or know the specs of.

    D. Auto-scaling is a FaaS feature, but it's the platform doing it, not you. You don't write "scale from 1 to 1000 instances", that's already happening automatically.

    Most common wrong pick, D. Scaling is a selling point of FaaS, so students assume they control it. But that's exactly what the platform abstracts away. You don't say "scale me now", the platform reacts to incoming requests.
    Exam tip: FaaS = your code + their everything. You control the function body, nothing else.
    QUESTION 14
    Licensing (GPL)

    Startup distributes a desktop app. Chooses Library B (GPL v3) over Library A (MIT). What's the risk?

    Answer: D · Copyleft requires GPL release of derivatives
    Concept: Copyleft vs permissive licenses

    Two license families:

    Permissive (MIT, BSD, Apache 2.0): You can do almost anything with the code, including ship it as part of a closed-source product. You must preserve the copyright notice and, for some, disclose patent terms, but you don't have to release your own source.

    Copyleft (GPL v2/v3, LGPL, AGPL): You can use the code, but if you distribute a product that includes it (or a derivative work), you must release your own source under the same license. This is "viral", your changes "catch" GPL on distribution.

    Applied to the startup:

    They're distributing a desktop app that links Library B (GPL v3). Because the app is a derivative work (GPL's definition is expansive, linking usually counts), distributing it means the ENTIRE app must be released under GPL. That means the startup has to publish their source code. For a closed-source commercial product, that's typically a deal-breaker.

    LGPL is a softer copyleft: dynamic linking to an LGPL library doesn't force your app to be GPL. This is why most widely-used C/C++ libraries (glibc, Qt's LGPL option) use LGPL.

    Why the others are wrong

    A. Attribution-only is a permissive license (MIT, BSD). GPL is far stricter.

    B. GPL has no fee. It's free to use commercially, the obligation is sharing source, not paying money.

    C. GPL absolutely allows commercial use. Red Hat, Canonical, and many others sell GPL software. The obligation is source disclosure, not a commercial ban.

    Most common wrong pick, C. "GPL = no commercial" is a persistent myth. GPL explicitly permits commercial distribution, it just requires your derivative to also be GPL (and thus source-available). Many sell GPL software commercially.
    Exam tip: GPL = copyleft = source must be released on distribution. MIT = permissive = do almost anything. Apache = permissive + explicit patent grant.
    QUESTION 15
    Nielsen Heuristic

    To scale a recipe, user types target serving count but can't see original serving size, they must remember it from a previous screen. Which heuristic is violated?

    Answer: A · H6 Recognition rather than Recall
    Concept: Nielsen's 10 usability heuristics

    The failure:

    User must remember "this recipe serves 4 people" from a previous screen in order to type in a scaled target. That's a recall task, actively retrieving information from short-term memory. Short-term memory is limited (Miller's 7±2, often less under cognitive load). Forcing users to remember things between screens is a known cause of errors and friction.

    Recognition vs recall:

    • Recognition: "Is this it?" You're given the info and identify it. Easy.
    • Recall: "What was it?" You must pull from memory unprompted. Hard.

    H6 says: make objects, actions, and options visible. Don't make users remember information from one part of the interface to another. Good UIs show context.

    The fix: On the scaling screen, display "Original recipe: serves 4". Now the user just looks, no memory required.

    Why the others are wrong

    B. H3 is about undo/cancel/back out of mistakes. No mistake has been made yet, this is about memory load, not reversibility.

    C. H5 is about designing to prevent invalid input (validation, disabling impossible options). Input validation is separate from displaying context.

    D. H2 Match with the Real World is about using the user's vocabulary and conventions. Not relevant, the issue isn't jargon or metaphor.

    Most common wrong pick, C. Students see a form input and default to "validate the input = H5 Error Prevention." But validation doesn't solve the memory problem. Even with perfect validation, the user still has to remember the original.
    Exam tip: If the user has to "remember from another screen", it's H6 Recognition Rather Than Recall. Show the info where they need it.
    QUESTION 16
    POUR Accessibility

    Difficulty shown as green/yellow/red dots with no text or shape. Which POUR principle is violated?

    Answer: C · Perceivable
    Concept: POUR, color must not be the sole cue

    POUR quick refresher:

    • Perceivable: Users can sense the info. Works with screen readers, colorblindness, low vision, hearing loss.
    • Operable: Users can interact. Keyboard-only, no time limits, no seizure triggers.
    • Understandable: Content and operation are clear. Readable language, predictable behavior.
    • Robust: Works with current and future assistive tech. Semantic HTML, valid markup.

    Applied here:

    Red/yellow/green with no text or shape means someone with red-green colorblindness (about 1 in 12 men) literally cannot perceive the difference. A screen reader sees "dot, dot, dot" because color isn't announced, the info is invisible to blind users too.

    This violates "Perceivable", specifically WCAG Success Criterion 1.4.1 "Use of Color": color must not be the only visual means of conveying information.

    Fix: Pair each dot with a label ("Easy", "Medium", "Hard") or a shape (circle, triangle, square) or an icon. Add aria-label for screen readers.

    Why the others are wrong

    A. Operable is about physical interaction (keyboard, timing, motion). Viewing a difficulty dot doesn't require motor control, it requires SEEING.

    B. Robust is about rendering across browsers and assistive tools. Cross-browser consistency isn't the issue here.

    D. Cultural color meaning (red = danger in some cultures, good luck in others) is real but secondary. Colorblindness is the universal, primary accessibility failure.

    Most common wrong pick, D. Cultural-meaning concerns feel deep and global, so students pick it. But POUR's Perceivable is more specific, color alone fails perception for colorblind and blind users, no matter the culture.
    Exam tip: Color alone as a signal = Perceivable violation. Always pair color with text, shape, or pattern.
    QUESTION 17
    Event Registration
    submitButton.setOnAction(event -> {
    System.out.println("Submitted");
});
System.out.println("Button registered");

    In what order do the prints appear?

    Answer: A · Registered first, fired later via event loop
    Concept: Callback registration vs invocation

    The fundamental distinction:

    A callback is a function-valued argument you hand to some system, saying "please call this later when something happens." Registration and invocation are two different events, separated in time.

    Line-by-line:

    1. submitButton.setOnAction(event -> { ... }) takes the lambda and stores it in the button as its action handler. The lambda body does NOT run here. setOnAction is just a setter. Execution is fast.

    2. System.out.println("Button registered") runs immediately after.

    3. Sometime later, user clicks the button. The FX event loop receives a click event, looks up the button's onAction handler, and invokes the lambda. "Submitted" prints.

    Mental model: setOnAction is like telling someone, "when the phone rings, pick it up." You're giving instructions for later, not picking up the phone now. The ring (click) happens when it happens.

    Why the others are wrong

    B. Callbacks don't auto-execute on registration. The whole point: they run in response to events, not at definition time.

    C. Both happen on the FX thread (the only thread JavaFX UI uses). There's no parallelism, and the callback certainly doesn't run "simultaneously" with its own registration.

    D. If the lambda ran immediately on the same call stack, you'd see "Submitted" before "Button registered", which contradicts the order stated. Also, no click has happened yet, so nothing can trigger the callback.

    Most common wrong pick, D. Students read "top to bottom" code and assume "Button registered" prints, then the lambda fires. They forget that the lambda needs a USER ACTION (click) to trigger. Without a click, the lambda just sits there.
    Exam tip: setOnAction stores the handler; the event loop invokes it later when the click arrives. Registration ≠ invocation.
    QUESTION 18
    MVC Responsibility

    Version A's handler iterates ingredients and calls ing.scale(factor). Version B's handler calls model.scale(servings). Which follows MVC better?

    Answer: B · Model owns the operation
    Concept: MVC, separation of concerns

    MVC responsibilities in one line each:

    • Model: business data and rules. Doesn't know about UI at all.
    • View: visual presentation. Shows what the Model says.
    • Controller: glue. Takes user input, tells Model to do stuff, tells View to refresh.

    Version A's mistake:

    The Controller is iterating the recipe's ingredients and calling .scale() on each one. That's business logic in the Controller. Now the scaling rule is tied to the UI, if you want to scale a recipe without a GUI (batch import, CLI tool, automated test), you have to reimplement the loop. The Model is a passive data bag, which means every consumer has to understand its internals.

    Version B's approach:

    The Controller forwards the intent, "scale to N servings." The Model's scale(servings) method encapsulates the logic, compute the factor, iterate ingredients, update each one. Controller is dumb glue. Benefits:

    • Model is reusable (CLI, tests, other UIs) without touching Controller.
    • Model is unit-testable without booting the UI.
    • If scaling logic changes (round to nice fractions, respect units), one place to update.
    Why the others are wrong

    A. Controller should NOT drive business iteration. Its job is to route input and refresh output, not do the work.

    C. "Keep Model simple by putting logic in Controller" is the OPPOSITE of MVC. The Model is where logic lives; the Controller should be simple.

    D. Bidirectional binding is an MVVM feature, not MVC, and not the reason B is correct. B is correct because of responsibility placement.

    Most common wrong pick, C. Students see a fat Controller doing "real work" and think that's the Controller's job. It's not. Business logic belongs in the Model. Fat Controllers are a classic MVC anti-pattern.
    Exam tip: If the Controller is doing anything a CLI couldn't also do, that logic should be in the Model.
    QUESTION 19
    MVVM Testing

    A test sets a ViewModel property and verifies the result without simulating a button click or starting the UI. Why does this work?

    Answer: D · ViewModel is View-agnostic
    Concept: MVVM, testability through decoupling

    MVVM vs MVC quick distinction:

    In MVC, the Controller mediates and has direct references to Model and View. In MVVM, the ViewModel exposes observable properties. The View binds to those properties. The ViewModel has NO reference back to the View, it doesn't know the View exists.

    Why that matters for testing:

    Because the ViewModel only reads and writes observable data (properties, bindings), a test can:

    1. Instantiate the ViewModel (possibly with mocked services).
    2. Set input properties: vm.servingsProperty().set(8);
    3. Read output properties: assertEquals(expected, vm.scaledIngredientsProperty().get());

    No window. No button clicks. No Stage.show(). No TestFX robot. Just JUnit and method calls. That's the MVVM testability payoff, the ViewModel is a pure component that can be exercised like any other POJO.

    Architectural contrast:

    In MVC, to test controller logic triggered by a button, you typically need TestFX to simulate the click. MVVM skips that layer, you test the state machine directly.

    Why the others are wrong

    A. TestFX is a UI-level test robot, it simulates clicks, types text, queries widgets. It doesn't "automatically intercept" ViewModel changes. The whole point of MVVM tests is that you DON'T need TestFX.

    B. The test explicitly uses the ViewModel (sets a property on it). Not bypassing. The reason it works is "VM is testable by itself", not "going around the VM."

    C. Laziness isn't relevant, and JavaFX property bindings actually update eagerly when their inputs change (not lazy in the FP sense).

    Most common wrong pick, A. TestFX is a real tool students hear about, so they associate it with UI testing. But MVVM testability is about NOT needing TestFX. If your test mentions TestFX, you're probably testing MVC's View-Controller wiring, not MVVM's state logic.
    Exam tip: MVVM ViewModel = testable without starting the UI. Key property: "ViewModel doesn't reference the View."
    QUESTION 20
    Race Condition

    Ten threads each call registerDevice() which does deviceCount++. Why might getDeviceCount() return less than 10?

    Answer: D · Non-atomic increment
    Concept: Atomicity, race condition on i++

    Why count++ is not atomic:

    Even though you write it as one character (well, two), count++ compiles to THREE JVM operations:

    1. Read current value of count into a register.
    2. Add 1 to the register.
    3. Write result back to count.

    Between steps, another thread can preempt. Picture two threads running simultaneously with count initially at 5:

    Thread A reads count = 5
Thread B reads count = 5   ← both saw the same "before" value
Thread A computes 5 + 1 = 6
Thread B computes 5 + 1 = 6
Thread A writes 6
Thread B writes 6          ← final value 6, should be 7

    One increment is lost. With 10 threads, this can happen multiple times, ending up with any value from (theoretically) 1 to 10.

    The fix, three options:

    • AtomicInteger, use count.incrementAndGet(), CAS-backed, lock-free, atomic.
    • synchronized, wrap the increment in a synchronized block to serialize access.
    • Lock, use ReentrantLock for more control.
    Why the others are wrong

    A. int is fine as a shared field type. The issue is the operation not being atomic, not the type.

    B. Reads of int are atomic (in Java's memory model for 32-bit types). Visibility issues exist too, but the primary failure here is atomicity. Even with perfect cache coherence, the read-modify-write pattern still interleaves.

    C. Fabricated. JVM doesn't "round down" anything. Increments that race are lost due to overwrites.

    Most common wrong pick, B. Visibility (one thread sees stale cache value from another core) IS a real concurrency issue, but it's separate from atomicity. Memory fence/volatile solves visibility, they don't solve the read-modify-write race. Students confuse the two.
    Exam tip: i++ = three ops = not atomic. Every time you see unsynchronized increment across threads, it's a race.
    QUESTION 21
    Deadlock

    Thread A: synchronized(room) { synchronized(device) {...} }. Thread B: synchronized(device) { synchronized(room) {...} }. What can happen?

    Answer: C · Deadlock from inconsistent lock ordering
    Concept: Classic deadlock scenario

    The deadlock recipe (four conditions, all met):

    1. Mutual exclusion: Each lock held by one thread at a time. ✓
    2. Hold and wait: Threads hold one lock while trying to acquire another. ✓
    3. No preemption: Locks can't be forcibly taken away. ✓
    4. Circular wait: A waits for a lock B holds; B waits for a lock A holds. ✓

    The interleaving that causes it:

    Thread A: acquire lock on room     ✓
Thread B: acquire lock on device   ✓
Thread A: try to acquire device    [blocked, B has it]
Thread B: try to acquire room      [blocked, A has it]
                                   [forever]

    Neither thread can make progress. They're both waiting on each other. Program doesn't crash, it HANGS.

    The standard fix, lock ordering:

    Impose a global rule: always acquire locks in the same order. "Always acquire room before device, everywhere." If both threads follow this, the circular wait can't form.

    // Both threads use same order:
synchronized(room) {
    synchronized(device) { ... }
}

    Other fixes: try-lock with timeout, single coarse-grained lock, or use higher-level concurrent structures.

    Why the others are wrong

    A. Contention level doesn't determine completion order, and deadlock means neither completes. Misreads the situation.

    B. Locks on different objects would be independent IF each thread only held one at a time. Issue is both threads try to hold BOTH. Independence breaks down.

    D. Race condition = both proceed and produce wrong data. Deadlock = neither proceeds at all. This is deadlock, not a race.

    Most common wrong pick, B. Students think "synchronized on different objects = no interaction." Normally true! The twist: each thread tries to hold BOTH at the same time. It's the ordering mismatch, not the objects, that creates the deadlock.
    Exam tip: Nested synchronized on two objects acquired in different orders = deadlock. Fix with consistent lock ordering.
    QUESTION 22
    FX Thread Blocking

    A button handler does a 2-second database search directly. What happens?

    Answer: B · UI freezes
    Concept: JavaFX single-threaded UI

    How the FX Application Thread works:

    JavaFX uses one thread, the FX Application Thread, for every UI-related thing: rendering frames, dispatching events, processing animations, running handlers. Event handlers run ON this thread. If a handler takes 2 seconds, the thread is busy for 2 seconds. During those 2 seconds:

    • No new frames are rendered, the window appears frozen.
    • No other events are processed, clicks, key presses, hover, all queued up.
    • No animations advance.
    • On some OSes, you get "Application not responding" after a few seconds.

    Why this is a common bug:

    Students write handlers like they're plain methods, "when the button is clicked, do the thing." They forget the handler runs on the FX thread. If the "thing" involves I/O, network, DB, file access, heavy computation, the UI freezes.

    The fix, 3 rules for JavaFX:

    1. UI touches happen ONLY on the FX thread.
    2. Long operations happen OFF the FX thread (use Task, a background Executor, or BackgroundTaskRunner).
    3. After background work completes, post the UI update back to the FX thread via Platform.runLater() (or use BackgroundTaskRunner's onSuccess which does this for you).
    Why the others are wrong

    A. JavaFX doesn't magically create background threads for your handler. The handler runs on whatever thread calls it, which is the FX thread.

    C. Wishful thinking. JavaFX never moves your code off the FX thread unless you explicitly ask.

    D. There's no "JavaFX cache" for arbitrary DB queries. Even if there were, the first call would still freeze.

    Most common wrong pick, C. The "framework handles it for me" assumption. JavaFX is notably hands-off about threading, it gives you the primitives but doesn't auto-background anything.
    Exam tip: Any slow operation in a handler freezes the UI. Never do I/O, network, or DB directly on the FX thread.
    QUESTION 23
    onSuccess Callback

    In BackgroundTaskRunner, onSuccess always runs on the FX thread, not the background thread. Why?

    Answer: C · Safe UI updates
    Concept: FX thread rules · UI updates must be on FX

    The rule:

    In JavaFX, touching a Node (changing label text, updating a progress bar, swapping images) from a thread other than the FX Application Thread throws IllegalStateException. The restriction isn't arbitrary, the scene graph isn't thread-safe, and allowing background threads to mutate it would corrupt state or trigger visual glitches.

    What BackgroundTaskRunner does:

    BackgroundTaskRunner.run(
    () -> doSlowWork(),          // runs on a background thread
    result -> updateUI(result),  // onSuccess, runs on FX thread
    err -> showError(err)        // onError, also on FX thread
);

    Internally, the callable runs off-thread (so UI stays responsive). When it completes, the runner posts the onSuccess callback back to the FX thread (via Platform.runLater()) before invoking it with the result.

    Best of both worlds: slow work doesn't freeze the UI, but the callback can safely do things like label.setText(result).

    Why the others are wrong

    A. FX thread isn't "faster", it's required for UI ops. The FX thread is actually a bottleneck; you want to do as little as possible on it.

    B. Background threads aren't immediately killed. Thread pools keep threads alive for reuse. Lifecycle isn't the reason.

    D. Cancellation has nothing to do with which thread onSuccess runs on. The callable's cancellation state is checked before the success path fires.

    Most common wrong pick, A. "FX thread is faster" is a persistent misconception. The FX thread is actually heavily loaded already (rendering, events), so you use it sparingly. The reason onSuccess runs there is access rules, not speed.
    Exam tip: Any code that updates UI widgets must run on the FX thread. BackgroundTaskRunner.onSuccess is on FX so your UI updates are safe.
    QUESTION 24
    Execution Order
    BackgroundTaskRunner.run(
    () -> loadRecipes(),            // 2 seconds
    recipes -> showRecipes(recipes),
    error   -> showError(error)
);
log("Ready");

    Assuming success, in what order do things happen?

    Answer: B · Ready first (FX), then bg work, then FX callback
    Concept: Non-blocking dispatch · async scheduling

    Trace it event by event:

    1. FX thread enters the handler. Hits BackgroundTaskRunner.run(...).
    2. run() is non-blocking. Queues the callable on a background thread pool and immediately returns. Nothing of loadRecipes has executed yet, the callable is just sitting in an executor's queue.
    3. Next line runs on FX thread: log("Ready"). Executes immediately because run() returned without blocking. "Ready" logs first.
    4. Meanwhile, on the background thread: loadRecipes() starts. Takes 2 seconds. The FX thread is free (handles more clicks, renders frames).
    5. loadRecipes returns its result after 2 seconds. The BackgroundTaskRunner sees success, posts showRecipes(recipes) back to the FX thread via Platform.runLater.
    6. FX thread picks up the queued task and executes showRecipes(recipes).

    Wall-clock order: "Ready" first, then 2 seconds of quiet on the UI thread while bg thread runs, then the UI updates.

    Why this design matters:

    If run() blocked (option A), the UI would freeze for 2 seconds. The entire point of BackgroundTaskRunner is to AVOID blocking.

    Why the others are wrong

    A. The blocking/synchronous interpretation. If it worked that way, UI would freeze for 2 seconds. The whole library exists to prevent that.

    C. "loadRecipes and Ready simultaneously" isn't quite right, they run on different threads, but "Ready" completes almost immediately while loadRecipes takes 2 seconds. "Simultaneously" misses the ordering.

    D. log("Ready") runs on whatever thread is executing the calling code (the FX thread, since this is typically in a handler). And showRecipes is posted back to FX.

    Most common wrong pick, A. "Code runs top to bottom on one thread" assumption. Students forget that run() is async, the call returns immediately while work is still queued.
    Exam tip: BackgroundTaskRunner.run is non-blocking. Code after the run() call executes before the callable completes.
    QUESTION 25
    FX vs Background Threads

    Which operation works equally well on either the FX thread or a background thread?

    Answer: C · LocalDateTime.now()
    Concept: FX thread rules · when it doesn't matter

    Two rules for FX thread:

    1. UI updates MUST be on FX thread. Anything touching a Node or property binding visible to the UI.
    2. Slow operations MUST NOT be on FX thread. Anything that could take more than ~16ms (one frame) or blocks on I/O.

    An operation is "thread-neutral" only if it violates NEITHER rule: not a UI update, and not slow. Reading the clock, LocalDateTime.now(), is neither. Fast system call, no scene graph touching. Safe on either thread.

    Evaluating each option:

    • A. Dark/light mode toggle, UI update. FX thread only.
    • B. Cutting power, slow I/O (network to IoT controller). Background thread only.
    • C. LocalDateTime.now(), fast, no UI. ✓ Thread-neutral.
    • D. Backup to server, slow network I/O. Background thread only.
    Why the others are wrong

    A. UI toggle (dark/light) = mutating the scene graph. FX thread only.

    B. Powering down devices likely involves network calls to each device's controller. Slow I/O, must be background.

    D. Backups = lots of network I/O. Slow. Must be background.

    Most common wrong pick, D. Backup "feels" neutral because it doesn't visibly touch UI. But anything involving a network call is slow by FX-thread standards and freezes the UI if run there.
    Exam tip: Thread-neutral = not a UI touch AND not slow. Clock reads, simple computation, in-memory data access all qualify.

    Case Studies

    CASE STUDY 1 · 10 POINTS
    SceneItAll · JavaFX Handler

    Given the following classes:

    HouseDashboardController
  -checkStatusButton: Button
  -areaComboBox: ComboBox<String>
  -infoLabel: Label
  -model: Model
  -zigbeeController: ZigbeeController

Model: getArea(String) → Area
ZigbeeController: getStatus(Area) → String
Label: setText(String), setTextFill(Color)
ComboBox<T>: getValue() → T
Button: addEventHandler(...)

    Task: add a handler for checkStatusButton that asks ZigbeeController for the status of the area from areaComboBox. Display the returned string in black in infoLabel. If either getArea() or getStatus() throws, show a user-friendly error in red. Later tasks will also write messages (black) and errors (red) to infoLabel.

    Part (a), 1 pt · Which class do you modify?
    HouseDashboardController
    Concept: MVC responsibilities

    Why the Controller: The Controller's job in MVC is to connect UI widgets to business services. It's the only class here that already knows about the UI (holds refs to checkStatusButton, areaComboBox, infoLabel) AND knows about services (model, zigbeeController).

    Why not the Model or ZigbeeController: They have no reason to know about infoLabel. Pushing UI concerns into them would violate separation of concerns, create a dependency from Model → JavaFX, and make the Model untestable without booting the UI.

    Part (b), 1 pt · Private helper methods
    Two helpers for reuse across later tasks
    Concept: DRY, plan for reuse
    private void showInfo(String text) {
    infoLabel.setText(text);
    infoLabel.setTextFill(Color.BLACK);
}

private void showError(String text) {
    infoLabel.setText(text);
    infoLabel.setTextFill(Color.RED);
}

    Why two helpers: The prompt tells you future tasks will also do info/error display. That's a hint, don't inline the color + text logic. Centralize it now so later tasks are one-liners.

    DRY benefit: If you later need to add a prefix ("ERROR: ") or log the error, you change it in one place.

    Private because: These are internal helpers. External callers (tests, other classes) should invoke the public handler, not these directly.

    Part (c), 1 pt · What is T in BackgroundTaskRunner<T>.run()?
    T = String
    Concept: Generic type inference

    Trace it: The callable is () -> zigbeeController.getStatus(...). getStatus(Area) returns a String, per the given API. So the Callable<T> has T = String.

    The other two arguments must match that T:

    • Consumer<String> onSuccess, consumes the returned String to display.
    • Consumer<Throwable> onError, takes any exception thrown by the callable.
    Part (d), 4 pts · Implementation
    Attach handler in constructor/init; call BackgroundTaskRunner inside
    Concept: Event handler + async work + error handling
    checkStatusButton.addEventHandler(ActionEvent.ACTION,
    event -> BackgroundTaskRunner.run(
        () -> zigbeeController.getStatus(
                  model.getArea(areaComboBox.getValue())),
        this::showInfo,
        t -> showError("Unable to retrieve status")
    )
);

    Breakdown:

    Event registration: addEventHandler(ActionEvent.ACTION, handler) wires the button press to the lambda. This whole call goes in the constructor or initialize() method.

    Callable (runs off FX thread): () -> zigbeeController.getStatus(model.getArea(areaComboBox.getValue())). Two chained calls, getArea to convert the combo's String to an Area, then getStatus to get the display string. Both can throw. Running on a background thread prevents UI freeze.

    onSuccess: this::showInfo, a method reference. The returned String is passed in, which showInfo displays in black. Runs on FX thread, safe to touch the label.

    onError: t -> showError("Unable to retrieve status"). Any throwable from either method lands here. User gets a friendly red message, the exception object itself isn't shown (could confuse users). Also FX thread.

    Key pattern: The overall structure is handler → async work → onSuccess/onError with UI updates via FX. That's the pattern the rubric grades.

    Part (e), 2 pts · Controllability & Observability
    Weak controllability and weak observability; both are fixable via DI + extraction
    Concept: Testability, controllability and observability

    Controllability problems (what the test can't control):

    • Input comes from a GUI ComboBox widget, not a method parameter. Test has to launch a UI and manipulate the widget (TestFX) or bypass it somehow.
    • Trigger is a button press, not a direct method call. Again requires TestFX or extraction.
    • Depends on a real ZigbeeController, which talks to actual hardware. Test can't control what it returns without a test double.

    Observability problem (what the test can't easily check):

    • Result is a widget mutation, infoLabel.setText(...) and setTextFill(...). Test must read back infoLabel.getText() and getTextFill() after the async work completes, requiring timing awareness.

    Fixes:

    1. Dependency Injection: Constructor takes ZigbeeController (and Model). Test passes a Mockito mock: when(mockZig.getStatus(any())).thenReturn("OK").
    2. Extract to named method: public void onCheckStatus(String area). Test calls this directly, no UI, no button simulation.
    3. Observability: Use TestFX to query the widget, or expose "current status" as a ViewModel property the test can read without UI.
    Part (f), 1 pt · Usability issue
    Color alone signals info vs error · POUR Perceivable violation
    Concept: Accessibility, color must not be sole cue

    The issue: Black = info, red = error, with no text difference or shape. Colorblind users (red-green affects ~1 in 12 men) can't distinguish. Screen-reader users can't hear color.

    Fixes:

    • Prefix errors with "ERROR: " (or use an icon like ⚠️).
    • Make errors bold or italic in addition to red.
    • Set infoLabel.setAccessibleText("Error: " + text) so assistive tech announces the state.
    • Optionally, change the label's background or add a border in error state.

    WCAG reference: Success Criterion 1.4.1 "Use of Color", color must not be the only visual means of conveying information.

    CASE STUDY 2 · 12 POINTS
    The CYB Plugin · Strategy & Builder

    CookYourBooks needs multiple recipe import formats (JSON, XML, binary, OCR). The current RecipeImporter uses a switch over format:

    public Recipe importRecipe(String source, String format) {
    switch (format) {
        case "json":   return parseJson(source);
        case "xml":    return parseXml(source);
        case "binary": return parseBinary(source);
        case "ocr":    return parseOcr(source);
        default: throw new IllegalArgumentException("Unknown format");
    }
}

    Proposed redesign uses Strategy pattern:

    interface RecipeParser {
    Recipe parse(String source);
    boolean supports(String format);
}
    Part (a), 2 pts · SOLID violations in the switch design
    OCP and SRP, both violated
    Concept: SOLID · Open-Closed & Single Responsibility

    OCP (Open/Closed Principle) violation:

    "Open for extension, closed for modification." Adding a new format (say, YAML) forces you to MODIFY RecipeImporter, edit the switch, add a case, add a parseYaml method. Every change risks breaking existing cases. The class is NOT closed to modification, it must change for every new format.

    The Strategy redesign fixes this: add a new format = add a new class implementing RecipeParser. RecipeImporter itself never changes.

    SRP (Single Responsibility Principle) violation:

    "A class should have one reason to change." The current RecipeImporter has four reasons to change:

    • JSON spec changes → edit parseJson.
    • XML schema changes → edit parseXml.
    • Binary format revision → edit parseBinary.
    • OCR library upgrade → edit parseOcr.

    Four orthogonal concerns in one class. Editing for XML risks touching JSON code. Editing for OCR risks touching binary. Each change has a larger blast radius than it should.

    Strategy fixes this: each parser class has ONE responsibility. JsonRecipeParser only handles JSON.

    Part (b), 1 pt · Arrange the redesigned body
    Order: B, D, C, A
    Concept: Strategy dispatch via supports() check
    return parsers.stream()                      // B
    .filter(p -> p.supports(format))        // D
    .findFirst().orElseThrow()              // C
    .parse(source);                          // A

    Flow:

    1. stream the parsers, turn the list of injected parsers into a stream.
    2. filter, keep only parsers whose supports(format) returns true.
    3. findFirst + orElseThrow, get the first match; throw NoSuchElementException if none.
    4. parse, delegate to the matched strategy.

    Why this is nice: adding a new format means adding a new parser to the injected list. The dispatch code never changes.

    Part (c), 2 pts · Two major deficiencies in the Javadoc
    Vague pre/post-conditions · No exception/null info
    Concept: Specifications, restrictive/general/clear

    The Javadoc says "Parses a recipe. @param source the data. @return the recipe". It's technically present but adds no information beyond what the method signature provides.

    1. Preconditions unspecified:
      • Can source be null? If so, what happens? If not, will we throw or return null?
      • Can it be empty? Whitespace? Malformed?
      • What format does source represent? That's the whole question.
    2. Postconditions and exceptions unspecified:
      • Can the returned Recipe be null?
      • Will the Recipe's fields be non-null? Title non-empty? Ingredients non-empty?
      • Which exception types are thrown, under what conditions? ParseException? IOException? Custom?

    Why this matters: A caller of parse() cannot write defensive code, cannot test edge cases, and cannot reason about nullability without reading the implementation. That defeats the purpose of a specification.

    Part (d), 1 pt · Feedback to the teammate
    Be respectful AND specific
    Concept: Code review, HRT pillars (Humility, Respect, Trust)

    Sample feedback:

    "Hey, thanks for adding documentation, great we're raising the bar on specs. A few suggestions to make this doc more useful for implementers:

    • Could we clarify what source can be? Specifically, what happens on null, empty, or malformed input?
    • What does the method promise on success? Does it guarantee non-null Recipe? Non-empty title? Non-empty ingredients list?
    • Which exceptions does it throw, under what conditions? ParseException? RuntimeException? That would help callers write defensive code.

    Right now the doc repeats the method signature. If we tighten up those three areas, it'll be easier for me and others to use this API correctly. Happy to help draft it if useful!"

    Rubric: Respectful + concrete + actionable = full credit. Insulting ("this doc is useless"), vague ("needs more detail"), or empty ("looks fine") lose credit.

    Part (e), 1 pt · LSP and OcrRecipeParser
    Subclass weakens the postcondition, LSP violation
    Concept: Liskov Substitution Principle

    LSP in one sentence: A subclass (or interface implementation) must be substitutable for its parent, meaning anywhere the parent type is expected, the subclass can take its place without surprising the caller.

    Formal version: Subclasses may strengthen postconditions and weaken preconditions, but never the reverse.

    Applied to OcrRecipeParser:

    The interface RecipeParser promises "returned Recipe has non-empty ingredients." That's a postcondition. Callers rely on it, e.g., they might do recipe.getIngredients().get(0) without a null/empty check.

    OcrRecipeParser sometimes returns a Recipe with an empty ingredients list (maybe the OCR couldn't read the ingredients section). That weakens the postcondition, the subclass promises LESS than the interface did. A client using the interface might crash (IndexOutOfBoundsException) when swapped onto an OcrRecipeParser, even though it was fine with JsonRecipeParser.

    Substitutability is broken → LSP violation.

    Fixes:

    • Make OcrRecipeParser conform: throw a ParseException if ingredients can't be extracted, rather than returning empty.
    • OR weaken the interface contract ("may return empty ingredients") and force all callers to handle that case.
    Part (f), 2 pts · Three test cases
    One test per contract clause, postcondition, exception, precondition
    Concept: Spec-driven testing

    Test 1, happy path (verifies postcondition):

    Pass well-formed source. Assert: Recipe is non-null, has a non-empty title, and a non-empty ingredients list. Confirms the positive promise of the spec.

    Test 2, malformed input (verifies exception behavior):

    Pass corrupt or truncated source. Assert: ParseException (or documented exception) is thrown. Use assertThrows(ParseException.class, () -> parser.parse(badSource)).

    Test 3, null input (verifies precondition handling):

    Pass null. Assert: IllegalArgumentException or NullPointerException is thrown (both acceptable by Java convention, IAE is more informative).

    Why these three: Together they cover the three contract clauses (normal behavior, error behavior, input validation) with minimal overlap. Each is a reason to change of the spec, so each deserves its own test.

    Part (g), 1 pt · Arrange the Builder calls
    Order: D, E, F, B, C, A
    Concept: Builder pattern, fluent API
    Recipe recipe = new RecipeBuilder()             // D, start
    .title("Chocolate Cake")                       // E, set title
    .addIngredient(new MeasuredIngredient("flour", 2, "cups"))  // F
    .addIngredient(new MeasuredIngredient("sugar", 1, "cup"))   // B
    .instructions(List.of("Mix ingredients", "Bake at 350F"))     // C
    .build();                                        // A, finish

    The Builder pattern: Each setter returns this so calls chain. Order: construct the builder, set fields in logical order, call build() to produce the final object.

    Why it's used: Lets you construct objects with many optional parameters without a giant constructor, while maintaining readability. Especially good for immutable domain objects.

    Part (h), 2 pts · Problem with no-args builder()
    No-args lets you .build() without required fields set
    Concept: Invariant enforcement at construction

    The problem:

    new RecipeBuilder() with no arguments creates a builder in a state where required fields haven't been set. Nothing prevents the developer from calling .build() immediately, producing a Recipe with a null title, empty ingredients, and null instructions. That Recipe violates its own invariants. The bug surfaces far from the construction site ("why is this Recipe's title null?") and is hard to trace back.

    The fix, require via constructor:

    public RecipeBuilder(String title) {
    if (title == null || title.isBlank())
        throw new IllegalArgumentException("title required");
    this.title = title;
}

    Now new RecipeBuilder("Chocolate Cake") is the only way to construct. Title is guaranteed non-null from the start. .build() still validates other invariants (ingredients non-empty) before returning.

    Stronger version: take all required fields in the constructor, new RecipeBuilder(title, ingredients). Optional fields (instructions, metadata) stay as chained setters.

    Why it matters: "Make illegal states unrepresentable." Enforcing invariants at construction time catches mistakes earlier and closer to the source.

    CASE STUDY 3 · 6 POINTS
    MediQueue · Architecture Decision Record

    Hospital patient intake system. Two options for estimated wait time:

    • Option A, Real-time estimation, each check-in triggers a live DB query
    • Option B, Pre-computed cache, background job recomputes every 5 min; kiosks read cache

    Write ADR-001 with Context, Decision, Consequences (at least 2 benefits and 2 drawbacks).

    Full ADR answer (sample full-credit response)
    Sample full-credit ADR, walkthrough below
    Concept: ADR structure, Context → Decision → Consequences

    ADR-001, Wait Time Estimation Strategy

    Context. MediQueue kiosks display estimated wait times at patient check-in. During peak hours (weekday mornings, shift handoffs), hundreds of patients may check in within a few minutes. If every check-in triggers a live DB query, the database becomes a bottleneck exactly when load is highest, potentially causing kiosk slowness or timeouts. Wait-time accuracy is desirable but not safety-critical, a slightly stale estimate (a few minutes old) is acceptable to patients and staff. An unresponsive or unavailable kiosk, however, directly harms the patient experience and can create bottlenecks in the physical intake line.

    Forces at play, accuracy vs. performance, peak-load scalability, patient-facing responsiveness, operational simplicity.

    Decision. Adopt Option B, the pre-computed cache. A background job recomputes the wait-time estimate every 5 minutes; kiosks read from the cache on each check-in. The primary architectural driver is performance under peak load, decoupling check-in from live computation keeps kiosks responsive even when the database is under strain.

    Consequences.

    • ✅ Benefit 1: Kiosks respond instantly to check-in events, no DB round-trip on the critical path. Reduced p99 latency.
    • ✅ Benefit 2: Estimation logic is isolated in the background job, independently testable and replaceable without touching the check-in code.
    • ⚠️ Drawback 1: Estimates may be up to ~5 minutes stale during sudden surges (mass casualty influx). Communicate to staff.
    • ⚠️ Drawback 2: Operational complexity rises, the team must now monitor both the check-in service AND the background job (liveness, errors, stale cache alerts).
    Rubric · How the 6 points are distributed
    6 pt rubric breakdown
    Concept: ADR grading, each section earns credit independently
    • Context, clearly describes the situation and forces at play, 1 pt.
    • Decision, picks one option AND justifies with at least one architectural driver from Context, 1 pt. (Naming the option alone = partial; must tie to a driver.)
    • Benefit 1, a concrete positive consequence, 1 pt.
    • Benefit 2, DISTINCT from Benefit 1, 1 pt. (Two benefits that are basically the same = partial credit.)
    • Drawback 1, a real trade-off accepted, 0.5 pt.
    • Drawback 2, DISTINCT from Drawback 1, 0.5 pt.

    Total: 6 pts.

    Common point-losses:

    • No Context or very shallow Context (what is MediQueue, what are the forces?)
    • Picks an option but doesn't say WHY (missing architectural driver).
    • Two benefits that are paraphrases of each other ("fast" and "responsive").
    • Drawbacks that are benefits in disguise, or unrelated risks.
