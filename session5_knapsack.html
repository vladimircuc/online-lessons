All of these give a fixed ratio: the approximation factor is a constant (or depends on a problem parameter like f), and you're stuck with it. You can't ask for "a little more accuracy" at the cost of "a little more time."

But back in Session 2, we defined something more powerful:

Recall: PTAS and FPTAS (from Session 2)

PTAS (Polynomial-Time Approximation Scheme): For any ε > 0, there's an algorithm giving a (1−ε)-approximation (for maximization) in time polynomial in n, but possibly exponential in 1/ε. Example: O(n1/ε).

FPTAS (Fully Polynomial-Time Approximation Scheme): Same guarantee, but runtime is polynomial in both n and 1/ε. Example: O(n³/ε).

FPTAS is strictly stronger than PTAS. With an FPTAS, doubling your desired accuracy only polynomially increases the runtime.

We defined PTAS and FPTAS in Session 2 but never built one. Today, we build the crown jewel: an FPTAS for the 0/1 Knapsack problem. This is the most important FPTAS in the course.

Quick Self-Check

A 2-approximation always gives you an answer within factor 2 of optimal, no matter how much time you're willing to spend. You can't "turn the dial" to get closer.

An FPTAS lets you choose your accuracy. Want within 1% of optimal? Set ε = 0.01. Want within 0.001%? Set ε = 0.00001. The runtime grows, but only polynomially. This is the ultimate tradeoff between speed and accuracy for NP-hard problems.

Part 2 / 8 — The 0/1 Knapsack Problem

What is 0/1 Knapsack?

You have a knapsack with weight capacity W, and n items. Each item i has a value vi and a weight wi. You must decide, for each item: take it (1) or leave it (0). The goal is to maximize total value without exceeding the weight capacity.

0/1 Knapsack Problem Input: n items with values v₁,...,vₙ and weights w₁,...,wₙ Capacity W Find: Subset S ⊆ {1,...,n} that maximizes Σᵢ∈S vᵢ subject to Σᵢ∈S wᵢ ≤ W

This is a maximization problem, and it is NP-hard (it can be shown via reduction from Subset Sum, which reduces from 3-SAT).

Greedy approaches for knapsack can be made into a 2-approximation with some tricks, but for better guarantees we need the FPTAS. The key ingredient is dynamic programming.

Part 3 / 8 — Exact DP for Knapsack

The Weight-Indexed DP

You've likely seen this DP before. It solves knapsack exactly, but its runtime depends on the numeric value of W.

DP Recurrence (weight-indexed): State: dp[i][w] = max value using items {1,...,i} with total weight ≤ w Base: dp[0][w] = 0 for all w Transition: dp[i][w] = max( dp[i-1][w], // don't take item i dp[i-1][w - wᵢ] + vᵢ // take item i (if w ≥ wᵢ) ) Answer: dp[n][W]

The table has n rows (one per item) and W+1 columns (capacities 0 through W). Each cell takes O(1) time. Total runtime: O(n · W).

Key Fact

This DP solves Knapsack exactly. It gives the true optimal. No approximation needed. But there's a catch: the runtime is O(n · W), and W can be astronomically large. This is the heart of the problem.

Part 4 / 8 — Why O(nW) is NOT Polynomial

The Subtlety: Input Size vs. Numeric Value

The DP runs in O(n · W). This looks polynomial, but it's not. The issue is how W is encoded in the input.

When we say "polynomial time," we mean polynomial in the input size, measured in bits. The number W is written in binary, so it takes log₂(W) bits to encode. If we let B = log₂(W) (the number of bits), then W = 2B.

The Critical Distinction

O(n · W) = O(n · 2B) where B = number of bits to write W.

This is exponential in the input size B! Adding one more bit to W doubles the runtime.

This is called pseudo-polynomial time: polynomial in the numeric value of the input, but exponential in the input length.

The FPTAS Idea (Preview)

If W being large makes the DP slow, what if we make the numbers smaller? Specifically, if we round down the item values and run a different DP on the smaller numbers, the table shrinks. We lose some accuracy, but if we're careful, the loss is bounded by ε. This is the core of the FPTAS.

Part 5 / 8 — The FPTAS Idea: Scaling Values

Two Different DPs for Knapsack

We saw the weight-indexed DP: dp[i][w] = max value with capacity w. Runtime O(nW). There's a second DP, less well-known but crucial for the FPTAS:

Weight-Indexed DP

dp[i][w] = max value using items {1,...,i} with weight limit w

Table size: n × W

Runtime: O(n · W)

Profit-Indexed DP ⭐

dp[i][p] = min weight to achieve profit exactly p using items {1,...,i}

Table size: n × P (where P = sum of all values)

Runtime: O(n · P)

The profit-indexed DP flips the question: instead of "what's the best value for this weight?", it asks "what's the minimum weight to achieve this value?" The answer to the original problem is: the largest p such that dp[n][p] ≤ W.

Profit-Indexed DP: State: dp[i][p] = minimum weight to achieve profit exactly p using items {1,...,i} Base: dp[0][0] = 0 dp[0][p] = ∞ for all p ≥ 1 Transition: dp[i][p] = min( dp[i-1][p], // don't take item i dp[i-1][p - vᵢ] + wᵢ // take item i (if p ≥ vᵢ) ) Answer: max p such that dp[n][p] ≤ W

This DP also solves knapsack exactly. Its runtime is O(n · P) where P = Σ vi. This is also pseudo-polynomial (P can be huge), but it opens the door to the FPTAS trick.

The Key Insight: Scale Down the Values

The profit-indexed DP's runtime depends on the total profit P. If we make the values smaller, the DP table shrinks. Here's the plan:

Step 1
Scale values down
→
Step 2
Run profit DP on small values
→
Step 3
Return same items

We define a scaling factor K and replace each value vi with a scaled value v'i = ⌊vi / K⌋. The floor ensures values stay integers. We solve the knapsack exactly on the scaled values (using the profit-indexed DP), then return the items that the scaled DP chose, but count their original values.

The trade-off: scaling loses information (rounding down), so the result is approximate. But if K is chosen carefully, the error is bounded by ε, and the runtime becomes truly polynomial.

Choosing the Scaling Factor K

The Scaling Factor

Let vmax = maxi vi (the largest item value). We set:

K = (ε · vmax) / n

This choice is not arbitrary. It's precisely calibrated so that:

1. The maximum scaled value is ≤ n/ε, keeping the DP table small.

2. The total rounding error across all selected items is ≤ ε · vmax ≤ ε · OPT.

Why does this keep the table small? The maximum scaled value of any item is:

v'i = ⌊vi / K⌋ ≤ vmax / K = vmax / (ε · vmax / n) = n / ε

So the maximum total scaled profit is at most n · (n/ε) = n²/ε. The profit-indexed DP table has size n × n²/ε, giving runtime O(n³/ε). This is polynomial in both n and 1/ε, making it an FPTAS!

Part 6 / 8 — The Full FPTAS Algorithm + Worked Example

The Complete Algorithm

FPTAS for 0/1 Knapsack: Input: Items (v₁,w₁),...,(vₙ,wₙ), capacity W, parameter ε > 0 1. Discard any item with wᵢ > W (can never be taken) 2. Set v_max = max vᵢ 3. Set K = (ε · v_max) / n 4. For each item i, compute scaled value: v'ᵢ = ⌊vᵢ / K⌋ 5. Run the exact profit-indexed DP on scaled values v'ᵢ with original weights wᵢ and capacity W 6. Let S = the item set chosen by the scaled DP 7. Return S (with original values, not scaled)

That's it. The algorithm is remarkably simple: scale, solve, return. The cleverness is all in the analysis (proving it works and is fast).

What If the FPTAS Doesn't Find the Exact Optimum?

In our example, the FPTAS happened to find the exact optimal (value 130). That's because the scaling didn't change the relative ranking of item sets. But scaling can cause suboptimal choices. Consider:

Example Where Scaling Causes a Suboptimal Choice

Items: (v=100, w=5), (v=55, w=3), (v=52, w=3). Capacity W = 6. ε = 0.5.

K = 0.5 · 100 / 3 ≈ 16.67. Scaled: v'₁ = ⌊6⌋ = 6, v'₂ = ⌊3.3⌋ = 3, v'₃ = ⌊3.12⌋ = 3.

Exact OPT: items {2,3}, value = 55 + 52 = 107, weight = 6.

Scaled DP: items {2,3} have scaled value 3+3 = 6. Item {1} alone has scaled value 6. Tied! The DP might pick item 1 (value 100) instead of {2,3} (value 107).

If it picks item 1: FPTAS value = 100. Ratio = 100/107 ≈ 0.93. Guarantee says ≥ (1−0.5) · 107 = 53.5. We're well within bounds.

The FPTAS guarantee is a worst-case bound. In practice, it usually does much better than (1−ε) · OPT.

Part 7 / 8 — Proving the Guarantees

What We Need to Show

The FPTAS has two guarantees to prove:

1. Approximation ratio: The returned solution S satisfies value(S) ≥ (1−ε) · OPT.

2. Runtime: The algorithm runs in O(n³/ε), polynomial in both n and 1/ε.

Part 1: The (1−ε) Approximation Proof

Let S* be the optimal solution (value OPT), and let S be the FPTAS solution.

Step A: The scaled DP finds the optimal scaled solution

S is optimal for the scaled instance (we solve the scaled DP exactly). So for any feasible set, including S*:

Σi∈S v'i ≥ Σi∈S* v'i

S has the highest total scaled profit among all feasible item subsets.

Step B: Relate scaled values to original values

For any item i, the floor operation gives: v'i = ⌊vi/K⌋ ≥ vi/K − 1.

So for the optimal set S*:

Σi∈S* v'i ≥ Σi∈S* (vi/K − 1) = OPT/K − |S*| ≥ OPT/K − n

(Since S* has at most n items, |S*| ≤ n.)

Step C: Convert the FPTAS scaled profit back to original values

For any item i: vi ≥ v'i · K (since ⌊vi/K⌋ · K ≤ vi). So:

value(S) = Σi∈S vi ≥ K · Σi∈S v'i
Step D: Chain everything together
value(S) ≥ K · Σi∈S v'i (Step C) ≥ K · Σi∈S* v'i (Step A: S optimal for scaled) ≥ K · (OPT/K − n) (Step B) = OPT − n·K = OPT − n · (ε·vmax/n) = OPT − ε·vmax
Final step: Why OPT ≥ vmax

We discarded items with wi > W in Step 1. Every remaining item fits individually. So the optimal solution includes at least one item, meaning OPT ≥ vmax (we can always take just the most valuable item).

value(S) ≥ OPT − ε·vmax ≥ OPT − ε·OPT = (1 − ε) · OPT ✓

Part 2: Runtime Analysis

Step 1: Maximum scaled value per item

v'i = ⌊vi/K⌋ ≤ vmax/K = vmax / (ε·vmax/n) = n/ε

Step 2: Maximum total scaled profit

Pmax = Σ v'i ≤ n · (n/ε) = n²/ε

Step 3: DP table size and total runtime

The profit-indexed DP has n rows and Pmax columns. Each cell is O(1).

Runtime = O(n · Pmax) = O(n · n²/ε) = O(n³/ε)

This is polynomial in both n and 1/ε. ✓

Problem 2
Pseudo-Polynomial vs. Polynomial
Understanding why O(nW) is not polynomial time

(a) An algorithm runs in O(nW) where n is the number of items and W is the knapsack capacity. Is this polynomial time? Explain carefully, making the distinction between input size and input value.

(b) Suppose W = 240 (about 1012) and n = 50. How many operations does the O(nW) DP use? How does this compare to the FPTAS with ε = 0.1?

(c) A student says: "The Knapsack DP runs in O(nW), so Knapsack is in P." What's wrong with this reasoning?

(a) No, O(nW) is not polynomial time. Polynomial time means polynomial in the input size, measured in bits. The number W is encoded in binary using B = ⌈log₂(W)⌉ bits. So W = 2B, and the runtime O(nW) = O(n · 2B) is exponential in the input size B. The total input size is O(n · B) (n items, each value/weight encoded in O(B) bits), and 2B is exponential in B. This is called pseudo-polynomial time: polynomial in the numeric values, but not polynomial in the input length.

(b) Exact DP: n · W = 50 × 240 ≈ 50 × 1012 = 5 × 1013 operations. At 109 operations/sec, that's about 50,000 seconds (roughly 14 hours).

FPTAS with ε = 0.1: Runtime O(n³/ε) = O(50³ / 0.1) = O(125,000 / 0.1) = 1,250,000 operations. That takes a fraction of a second.

The FPTAS is over 40 million times faster on this instance, at the cost of an answer that's within 10% of optimal.

(c) The reasoning confuses "polynomial in the numeric value of the input" with "polynomial in the input size." For a problem to be in P, the algorithm must run in time polynomial in the number of bits needed to write the input. Since W can be written in log₂(W) bits but the DP runs in O(W) time (exponential in that bit count), the DP does not prove Knapsack is in P. Knapsack remains NP-hard because no known algorithm solves it in time polynomial in the bit length of the input.

Problem 4
FPTAS Proof Comprehension
Explain each step of the approximation proof

The FPTAS approximation proof contains this chain:

value(S) ≥ K · Σi∈S v'i ≥ K · Σi∈S* v'i ≥ K · (OPT/K − n) = OPT − nK = OPT − ε·vmax ≥ (1−ε)·OPT

For each of the five "≥" steps, answer:

(a) value(S) ≥ K · Σ v'i: Why does the original value of each item exceed its scaled value times K?

(b) K · ΣS v'i ≥ K · ΣS* v'i: What property of S makes this true?

(c) ΣS* v'i ≥ OPT/K − n: What property of the floor function is used here?

(d) OPT − ε·vmax ≥ (1−ε)·OPT: What assumption about OPT makes this true? When might this assumption fail, and how does the algorithm handle it?

(e) Why does each item contribute at most K error from rounding, and why is nK the total worst-case error?

(a) v'i = ⌊vi/K⌋, so v'i · K = ⌊vi/K⌋ · K ≤ vi. This is because the floor rounds down, so multiplying back by K gives at most the original value. Therefore vi ≥ v'i · K, and summing over all items in S gives value(S) ≥ K · Σi∈S v'i.

(b) S is the optimal solution for the scaled knapsack instance (we solve the profit-indexed DP exactly on the scaled values). Since S* is also a feasible solution for the scaled instance (same weights, same capacity), S must have total scaled profit at least as large as S*. So Σi∈S v'i ≥ Σi∈S* v'i.

(c) The floor function satisfies ⌊x⌋ ≥ x − 1 for any real x. Applied to each item: v'i = ⌊vi/K⌋ ≥ vi/K − 1. Summing over S*: Σi∈S* v'i ≥ Σi∈S* (vi/K − 1) = OPT/K − |S*|. Since S* has at most n items, |S*| ≤ n, giving Σ v'i ≥ OPT/K − n.

(d) OPT − ε·vmax ≥ (1−ε)·OPT rearranges to ε·OPT ≥ ε·vmax, which simplifies to OPT ≥ vmax. This means the optimal solution's value is at least as large as the single most valuable item. This holds because Step 1 of the algorithm discards items with wi > W. After discarding, every remaining item fits by itself, so the optimal solution can always include at least the most valuable item. Without Step 1, an item might have huge value but weight exceeding W, making vmax > OPT.

(e) For each item i, the rounding error is vi − v'i·K = vi − ⌊vi/K⌋·K. Since ⌊x⌋ ≥ x − 1, we get ⌊vi/K⌋ ≥ vi/K − 1, so v'i·K ≥ vi − K. The error per item is at most K. In the worst case, the FPTAS selects all n items, so the total rounding error is at most nK = n · (ε·vmax/n) = ε·vmax. This is exactly the "budget" of error we designed K to allow.

(a) We've seen three approximation paradigms: Greedy (Session 2), LP Rounding (Session 4), and FPTAS (today). For each, explain in one sentence the source of the "lower bound" (or "upper bound" for maximization) that lets us prove the approximation ratio.

(b) Why can't we use LP rounding to get an FPTAS for Knapsack? (Hint: think about the integrality gap.)

(c) Not every NP-hard problem has an FPTAS. Explain intuitively why Knapsack is "special" in a way that allows the scaling trick. What property of the problem makes the DP runtime depend on numeric values rather than combinatorial structure?

(a)

Greedy (EDP): The lower bound comes from a counting argument: the greedy algorithm uses many edge-disjoint paths, and any optimal solution can't use "too many more" because the paths share limited edge resources.

LP Rounding (Vertex Cover/Set Cover): The lower bound is LPOPT, the value of the LP relaxation. Since LPOPT ≤ ILPOPT = OPT (for minimization), the LP relaxation provides a provable bound on the true optimum, and we relate the rounded solution's cost to this LP bound.

FPTAS (Knapsack): The "lower bound" comes from the optimal scaled solution. Since the FPTAS solves the scaled instance exactly, and the scaling introduces bounded rounding error (at most nK = ε·vmax ≤ ε·OPT), the FPTAS value is within (1−ε) of OPT.

(b) LP rounding for Knapsack has a fixed integrality gap. The LP relaxation of Knapsack (the "fractional knapsack") allows taking fractions of items, and its solution can be arbitrarily close to the ILP optimum, but the rounding step to get integers can lose a fixed fraction. More specifically, LP rounding gives a fixed-ratio approximation (roughly a 2-approximation for many problems), but you can't "tune" it to get (1−ε) for any ε. The FPTAS works differently: instead of relaxing integrality constraints, it changes the numbers to make exact integer solving feasible. LP rounding and FPTAS are fundamentally different strategies.

(c) Knapsack's hardness is numeric, not combinatorial. The DP table size depends on the magnitude of values/weights (W or P), not on the graph structure or constraint complexity. This means we can "shrink" the problem by scaling down the numbers, losing some precision but making the DP tractable. Problems like Vertex Cover or 3-SAT have hardness rooted in combinatorial structure (graph connectivity, clause interactions), and there's no "dial to turn" that trades precision for speed. Scaling numbers down doesn't simplify their constraint structure. This is why Knapsack has an FPTAS while many other NP-hard problems don't even have a PTAS.
