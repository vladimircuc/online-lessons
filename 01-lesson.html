OOP Fundamentals

Dynamic dispatch, static context, casting. The rules are unambiguous once you internalize how Java separates declared type from runtime type.

Dynamic dispatch: declared type vs runtime type

Every variable in Java has two types: the declared type (what the compiler sees) and the runtime type (the actual object in memory). Method calls use dynamic dispatch, meaning Java picks the method implementation based on the runtime type, not the declared type.

Animal a = new Dog();  // declared: Animal, runtime: Dog
a.speak();                 // calls Dog.speak() — runtime type wins
The rule: fields are resolved by declared type (static), methods are resolved by runtime type (dynamic). This is why polymorphism works.

Casting does NOT change the object

A cast is a compile-time promise to the compiler: "trust me, this reference points to a subtype." It changes how the compiler treats the reference. It does not mutate the actual object.

Light light = new TunableWhiteLight("living-room", 2700, 100);
((DimmableLight) light).turnOn();
// Runtime type is STILL TunableWhiteLight.
// If TunableWhiteLight overrides turnOn(), that's what runs.
Exam trap: students see the cast to DimmableLight and think "oh, so DimmableLight's version runs." Wrong. The cast just tells the compiler "I promise this IS a DimmableLight." The actual object is still whatever you instantiated, so its overridden methods win.

Static methods can't touch instance state

A static method belongs to the class, not to any instance. There is no this in a static method. So it cannot reach instance fields directly, there's no instance to reach into.

public class Example {
    int x = 10;                       // instance field

    public static void printX() {
        System.out.println(x);         // COMPILE ERROR
    }
}

Fix: either make x static (belongs to the class), or accept an Example instance and read instance.x.

QUICK CHECK · 1.1
Light light = new TunableWhiteLight("room", 2700, 100);
((DimmableLight) light).turnOn();
If TunableWhiteLight overrides turnOn(), which version runs?
Answer: D. Casts only adjust what the compiler allows. The actual object is a TunableWhiteLight, so its overridden turnOn() always runs.
QUICK CHECK · 1.2
Why doesn't this compile?
public class Example { int x = 10; public static void printX() { System.out.println(x); } }
Answer: B. There's no this inside a static method, so there's no instance to read x from. Access modifiers don't matter here, it's about instance vs class context.
QUICK CHECK · 1.3
Given Shape s = new Circle(); where both declare a String type field with different values, what does s.type return?
Answer: A. Only methods dispatch dynamically. Fields are resolved by declared type, so s.type gives you Shape's field. This is called field shadowing and it's a classic trap.

Collections & Comparators

The Big-O of lookups, and how Comparator chaining builds multi-key sorts without writing your own compare method.

Big-O you actually need to know

OperationArrayListHashSetHashMapTreeMap
contains / getO(n)O(1)O(1)O(log n)
addO(1)*O(1)O(1)O(log n)
remove by valueO(n)O(1)O(1)O(log n)
iterationO(n)O(n)O(n)O(n) sorted

* ArrayList add is amortized O(1); occasional resizes are O(n).

Membership test matters: testing "is this in the collection?" over an ArrayList with a for-each loop is O(n). Using a HashSet with .contains() is O(1). If you do many membership tests, that's the difference between fast and painfully slow.

Comparator chaining

Build multi-key sorts by composing. Each .thenComparing() is a tiebreaker that kicks in only when the previous keys are equal.

lights.sort(
    Comparator.comparing(DimmableLight::getId)      // primary
        .thenComparingInt(DimmableLight::getBrightness) // tiebreaker
);

If two lights have the same id (say both "bedroom"), brightness breaks the tie, ascending. So bedroom/50 comes before bedroom/80. Input order is irrelevant once you sort.

Reversing + natural order

  • Comparator.naturalOrder() uses Comparable's compareTo
  • Comparator.reverseOrder() flips it
  • .reversed() flips a custom comparator
  • nullsFirst(cmp) / nullsLast(cmp) handle nulls
QUICK CHECK · 2.1
Version A uses ArrayList<Device> with a for-each membership check. Version B uses HashSet<Device> with contains(). Big-O for membership?
Answer: A. Linear scan vs hash lookup. The JIT claim in D is a made-up trap, JIT does not convert data structures.
QUICK CHECK · 2.2
Given three lights: bedroom/80, kitchen/30, bedroom/50, sorted by id then brightness ascending. What's the order?
Answer: A. "bedroom" < "kitchen" alphabetically, so bedrooms come first. Within the bedrooms, 50 < 80.
QUICK CHECK · 2.3
Which statement about TreeMap is correct?
Answer: B. TreeMap is a red-black tree, logarithmic operations but naturally sorted iteration. HashMap allows null keys, TreeMap doesn't (needs to compare).

Specifications & Nullability

Why you can use complex classes without reading their code, and how modern Java codebases prevent NullPointerExceptions at compile time.

Specifications enable chunking

You can use HashMap without reading its 2000-line implementation. Why? Because its Javadoc tells you exactly what every method does: preconditions, postconditions, complexity guarantees. You treat the whole class as a single mental unit.

Abstraction is leverage. A good spec lets anyone use your code as a black box. A bad spec forces every caller to read your implementation, which doesn't scale.

@NullMarked and @Nullable

In a @NullMarked package, the default flips: every type is assumed non-null unless explicitly annotated @Nullable. This is the opposite of vanilla Java.

// package is @NullMarked
public String formatIngredient(@Nullable String prefix, String name) {
    if (prefix == null) {
        return name;
    }
    return prefix + " " + name;
}
  • prefix is explicitly @Nullable, callers may pass null
  • name has no annotation, under @NullMarked it is assumed non-null
  • Passing null for name triggers a checker warning/error
  • The return type has no annotation, so it's non-null too
Exam trap: don't assume annotations "automatically throw NPE." They drive a static checker, they don't add runtime checks by themselves.
QUICK CHECK · 3.1
Why can you use HashMap without reading its implementation?
Answer: B. Specs enable abstraction. D is wrong because TreeMap and HashMap have very different performance and ordering guarantees.
QUICK CHECK · 3.2
In a @NullMarked package, a method has @Nullable String prefix and String name. Which is true?
Answer: A. @NullMarked flips the default to non-null. Only the explicitly-annotated parameter is nullable.
QUICK CHECK · 3.3
A complex lambda with 4 conditions inside a .filter(). Best refactor for testability?
Answer: C. A named method can be unit tested directly with varied inputs. Lambdas embedded in streams cannot be called in isolation.

Testing & Testability

Mockito mechanics, and the code patterns that make testing impossible.

Mockito: when vs verify

when(...).thenReturn(...)

Stubs an input. Tells the mock: "if this method is called with these args, return this value."

verify(mock).method(...)

Asserts an output. Checks: "this method was called on the mock, exactly once, with these args."

TemperatureSensor mockSensor = mock(TemperatureSensor.class);
HVACService mockHVAC = mock(HVACService.class);

when(mockSensor.readTemperature("livingRoom")).thenReturn(65.0);  // stub

ThermostatController controller = new ThermostatController(mockSensor, mockHVAC);
controller.adjustToTargetTemperature(72.0, "livingRoom");

verify(mockHVAC).activate("livingRoom");  // assertion

Things that break testability

Static Calls

Cannot be swapped with a test double. TimeUtils.isBusinessHours() inside your class? Your test is stuck with real wall-clock time.

Private Methods

Can't be called directly from a test. You must invoke them through their public caller, which may be hard to set up.

new Inside Methods

new Database() inside a method means the test can't substitute a fake. Pass dependencies into the constructor instead.

Singletons

Global state across tests means test order matters. Flaky, brittle, avoid.

Controllability vs Observability. Controllability = can the test drive the class into the state we want? Observability = can the test see the result? Both must be good for a test to be useful. Widgets (buttons, labels) are a classic example: low controllability (needs TestFX) and low observability (must read widget state after).

Dependency Injection makes things testable

// BAD: controller creates its own HVAC, can't swap in a test
class Controller {
    HVACService hvac = new RealHVAC();
}

// GOOD: dependency injected, test passes a mock
class Controller {
    private final HVACService hvac;
    public Controller(HVACService hvac) { this.hvac = hvac; }
}
QUICK CHECK · 4.1
What does verify(mockHVAC).activate("livingRoom") assert?
Answer: A. Verify is an assertion about past calls. Stubbing is when().thenReturn().
QUICK CHECK · 4.2
A class calls TimeUtils.isBusinessHours() (a static method). Why is testing a branch that depends on this hard?
Answer: B. You can't swap out a static call without special tools (Mockito inline, PowerMock). Standard Mockito can't intercept statics.
QUICK CHECK · 4.3
Best way to make a class with an HVAC dependency testable?
Answer: D. DI lets tests inject a mock. Everything else locks the class to a specific implementation.

SOLID & Design Patterns

Five principles, two patterns you must know (Strategy, Builder), plus the fragile base class problem.

SOLID in one line each

PrincipleIn one lineSmell when violated
SRPOne class, one reason to changeGiant classes doing 5 things
OCPOpen for extension, closed for modificationSwitch statement you re-edit for every new case
LSPSubclasses must honor the parent's contractSubclass throws unexpected exceptions or weakens postconditions
ISPClients shouldn't depend on methods they don't use"Fat" interfaces forcing empty implementations
DIPDepend on abstractions, not concretionsnew RealHVAC() hard-coded in consumers

Fragile Base Class

Subclassing a class you don't own (like ArrayList) is risky because the parent's internal implementation can change between Java versions. Your overrides may stop satisfying their specification.

class LoggingList<E> extends ArrayList<E> {
    private int addCount = 0;
    @Override public boolean add(E e) {
        addCount++;
        return super.add(e);
    }
    @Override public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return super.addAll(c);
    }
}
The danger: if a future ArrayList.addAll is reimplemented to call add internally, you'd double-count every element. Your spec ("count of items added") would silently break. You're coupled to invisible internals.

Strategy Pattern

Replace a switch over "what kind of thing am I?" with a set of classes that all implement a common interface. Consumers hold the interface, not a switch.

// Before (violates OCP, SRP):
switch (format) {
    case "json": return parseJson(source);
    case "xml":  return parseXml(source);
    // add new format = edit this class
}

// After (Strategy):
interface RecipeParser {
    Recipe parse(String source);
    boolean supports(String format);
}

return parsers.stream()
    .filter(p -> p.supports(format))
    .findFirst().orElseThrow()
    .parse(source);

Builder Pattern

For objects with many optional or sequentially-added parameters. Chain .setX() / .addY() calls, finalize with .build().

Recipe recipe = new RecipeBuilder()
    .title("Chocolate Cake")
    .addIngredient(new MeasuredIngredient("flour", 2, "cups"))
    .addIngredient(new MeasuredIngredient("sugar", 1, "cup"))
    .instructions(List.of("Mix", "Bake at 350F"))
    .build();
No-args builder problem: a new RecipeBuilder() with no arguments lets you call .build() before setting a required field like title. Safer: new RecipeBuilder(title) forces required fields at construction.
QUICK CHECK · 5.1
LoggingList extends ArrayList, counts calls to add and addAll. What is the core problem?
Answer: B. Fragile base class: if addAll in the parent ever calls add internally, your count doubles. You're coupled to implementation details you don't own.
QUICK CHECK · 5.2
Switch-over-format in RecipeImporter. Which SOLID principles does it violate?
Answer: A. OCP because every new format requires modifying the class. SRP because parsing JSON, XML, binary, OCR are four distinct responsibilities.
QUICK CHECK · 5.3
A spec promises Recipe has non-empty title and ingredients. OcrRecipeParser sometimes returns empty ingredients. Which principle is violated?
Answer: C. Liskov: anywhere you use a RecipeParser you expect the promised postconditions to hold. Breaking them in a subclass means callers can't rely on the interface.

MVC & MVVM

Where logic lives, and why MVVM is easier to test than MVC.

MVC: Model, View, Controller

  • Model, data + business logic. Recipes, scaling rules, what a device is.
  • View, widgets the user sees. Buttons, labels, sliders.
  • Controller, glue. Listens to the View, delegates to the Model, updates the View.
Rule of thumb: if you can describe it as "when a bell goes off, do a calculation," the bell belongs in the View, the calculation belongs in the Model, and the wiring belongs in the Controller. 
// BAD: Controller doing Model's work
submitButton.setOnAction(e -> {
    for (Ingredient ing : recipe.getIngredients()) {
        ing.scale(factor);   // scaling logic inline in Controller
    }
});

// GOOD: Controller delegates to Model
submitButton.setOnAction(e -> {
    model.scale(servings);    // Model owns the operation
    view.refresh();
});

MVVM: ViewModel replaces Controller

The ViewModel holds observable properties bound to the View. Crucially, the ViewModel does not reference the View. That makes it fully unit-testable: a test can set properties and read results without simulating button clicks or starting the UI.

AspectMVCMVVM
Glue layerController (often references View)ViewModel (no View reference)
View ↔ glueManual event handlersData binding on observable properties
TestabilityNeed TestFX or UI stand-inPure unit tests on ViewModel
QUICK CHECK · 6.1
Version A's handler iterates ingredients and calls ing.scale(factor). Version B's handler calls model.scale(servings). Which follows MVC better?
Answer: B. Business logic lives in the Model. The Controller's job is wiring, not computation.
QUICK CHECK · 6.2
In MVVM, why can a test set a ViewModel property and verify the result without starting the UI?
Answer: D. The defining MVVM property: the ViewModel is ignorant of the View. So you can exercise it in plain JUnit.
QUICK CHECK · 6.3
A Controller directly manipulates pixels on a Canvas and also writes to a database. What's the most likely SOLID/MVC smell?
Answer: C. Rendering and persistence are two distinct responsibilities. That's a Single Responsibility red flag.

Concurrency

Race conditions, deadlocks, and why i++ is not atomic. Includes an interactive simulator.

Why i++ is not atomic

deviceCount++ looks like one step but actually runs as three JVM instructions:

  1. Read deviceCount into a register
  2. Increment the register
  3. Write back to memory

Two threads can read the same value before either writes, then each writes back "their" incremented value. One increment is lost. Run it in the widget below to see it happen.

Deadlock: inconsistent lock ordering

Deadlock requires two threads, two locks, and one acquiring them in opposite order.

// Thread A
synchronized (room) {
    synchronized (device) { ... }
}

// Thread B
synchronized (device) {
    synchronized (room) { ... }
}

If A grabs room while B grabs device, neither can proceed. A waits for device, B waits for room. Forever.

Prevention: always acquire locks in a consistent global order. If everyone grabs room first, the deadlock can't form.
QUICK CHECK · 7.1
Ten threads each call registerDevice() which does deviceCount++. Why might the final count be < 10?
Answer: D. The increment is three operations, and any two threads can interleave those three steps, overwriting each other's writes.
QUICK CHECK · 7.2
Thread A locks room then device. Thread B locks device then room. What can happen?
Answer: C. Two locks, opposite order, classic four-conditions-of-deadlock setup.
QUICK CHECK · 7.3
Safest fix for the deviceCount++ race?
Answer: B. volatile gives visibility but not atomicity for read-modify-write. AtomicInteger gives both. synchronized is also acceptable.

JavaFX Threading

The single biggest conceptual cluster on this exam. Only the FX Application Thread can touch UI; don't block it with slow work. BackgroundTaskRunner is the pattern.

The three rules

Rule 1

Only the FX Application Thread may read or write UI widgets.

Rule 2

Never block the FX thread with slow work (DB, network, big computations). The UI will freeze.

Rule 3

Do slow work on a background thread, then hop back to the FX thread to update the UI.

BackgroundTaskRunner pattern

BackgroundTaskRunner.<String>run(
    () -> zigbeeController.getStatus(area),  // Callable — runs on bg thread
    status -> showInfo(status),                // onSuccess — FX thread
    err    -> showError("Unable to retrieve status")  // onError — FX thread
);

The run() method returns immediately. The callable starts on a background thread. When it completes, onSuccess (or onError) fires on the FX thread so it can safely update the UI.

Setting up an event handler

    System.out.println("Submitted");

QUICK CHECK · 8.1
Order of "Button registered" vs "Submitted"?
Answer: A. setOnAction registers the callback; the event loop calls it later when the click happens.
QUICK CHECK · 8.2
A button handler does a 2-second DB query directly (no background thread). What happens?
Answer: B. The FX thread runs one thing at a time. A 2-second block = 2 seconds of a frozen UI.
QUICK CHECK · 8.3
Why does onSuccess run on the FX thread rather than the background thread that produced the result?
Answer: C. Rule 1 again: only the FX thread may touch UI. onSuccess usually updates labels/tables, so it must run there.

Architecture & Ops

Modular monoliths, FaaS, distributed fallacies, licensing, ADRs. The "one-question-each" grab bag.

Modular monolith vs microservices

A modular monolith runs as one process but enforces strict module boundaries, modules only talk through public APIs, never via shared tables. Compared to an unstructured monolith, the main win is extractability: you can lift a module into its own service later without months of untangling.

Trap: modular monoliths do NOT give independent deploys (one process = one deploy) or network isolation (still in-process). Don't pick options that claim those benefits.

FaaS: what you control vs what the platform does

You controlPlatform controls
The algorithm inside your functionServer hardware
Function input/output contractOS configuration
Which runtime / languageScaling under load

Fallacies of distributed computing

  1. The network is reliable
  2. Latency is zero
  3. Bandwidth is infinite
  4. The network is secure
  5. Topology doesn't change
  6. There is one administrator
  7. Transport cost is zero
  8. The network is homogeneous

Batching 100 results into one call instead of 100 calls combats fallacy #2 directly, 100 round trips cost 100× latency.

The GPL risk: if you distribute software that links a GPL library, your whole product must be released under the GPL. For proprietary software, this is usually a deal-breaker.

ADR format

An Architecture Decision Record documents a choice. The standard skeleton:

  • Context, the situation and forces at play
  • Decision, what you chose and why (reference at least one driver)
  • Consequences, benefits AND drawbacks (both required)
QUICK CHECK · 10.1
Primary benefit of a modular monolith over an unstructured monolith?
Answer: C. It's still one process and one deploy. The real gain is the option to extract later with minimal pain.
QUICK CHECK · 10.2
Using a startup must-ship GPL library, what's the risk?
Answer: B. GPL is copyleft: derivative works inherit the license.
QUICK CHECK · 10.3
Batching 100 test results into one API call instead of 100 calls. Which fallacy does this address?
Answer: A. Each call has irreducible latency. Cutting 100 round-trips to 1 is a direct latency win.
