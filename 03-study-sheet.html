§ 01

Java OOP Rules

Dynamic Dispatch (Instance Methods)

Non-static method calls are dispatched on the runtime type of the object, not the declared type. Overridden methods in subclasses take over.

Animal a = new Dog();
a.speak();  // runs Dog.speak(), not Animal.speak()

Static Methods (No Dispatch)

Static calls are resolved by the compile-time type. Subclass static methods hide, not override.

Animal a = new Dog();
a.taxonomy();  // runs Animal.taxonomy(), NOT Dog.taxonomy()

Field Access

Fields also use compile-time type. Never polymorphic. Always prefer methods for overridable behavior.

Casting

  • Upcast (sub to super): implicit, always safe.
  • Downcast (super to sub): explicit, can throw ClassCastException.
  • Use instanceof before downcasting, or pattern match: if (a instanceof Dog d) { ... }.
Common trap: a static method called on an instance reference still resolves via the reference's declared type, not the object's runtime type.

Collections Big-O

Collection add / put get / contains remove Ordered?
ArrayList O(1) amortized (end) O(1) by index O(n) Insertion order
LinkedList O(1) head/tail O(n) by index O(1) given node Insertion order
HashMap O(1) avg O(1) avg O(1) avg None
HashSet O(1) avg O(1) avg O(1) avg None
TreeMap / TreeSet O(log n) O(log n) O(log n) Sorted by key
LinkedHashMap O(1) avg O(1) avg O(1) avg Insertion order
Gotcha: ArrayList.remove(int index) vs .remove(Object o). Calling .remove(1) on ArrayList<Integer> removes by index, not value. Use .remove(Integer.valueOf(1)) to remove by value.

Picking the Right One

  • Need fast index access? ArrayList.
  • Frequent head/tail ops (queue, deque)? ArrayDeque or LinkedList.
  • Membership tests without order? HashSet.
  • Key-value, order doesn't matter? HashMap.
  • Need sorted iteration? TreeMap / TreeSet.
§ 03

Nullability Annotations

The Rule With @NullMarked

When a package or class is @NullMarked, every reference is non-null by default. You must explicitly mark nullable ones.

@NullMarked
package com.example.app;

// inside this package:
String name;              // non-null (guaranteed)
@Nullable String middle; // may be null

Quick Rules

  • Parameters: if unmarked and package is @NullMarked, you cannot pass null.
  • Returns: if unmarked and package is @NullMarked, caller doesn't need to null-check.
  • Fields: same rules. Non-null fields must be initialized before first read.
  • @Nullable on a return means the caller must handle null.
Benefit: compile-time checking catches nulls before they become NullPointerException at runtime.

Comparator Patterns

Basic Chaining

Comparator<Person> byAgeThenName =
    Comparator.comparingInt(Person::getAge)
              .thenComparing(Person::getName);

Reversal & Null Handling

// descending by age
Comparator.comparingInt(Person::getAge).reversed();

// nulls first
Comparator.nullsFirst(Comparator.naturalOrder());

Which Method to Use

Trap: using comparing(Person::getAge) autoboxes every int to Integer. Prefer comparingInt for primitives.
§ 05

Testing & Mockito

Mockito Core Syntax

// stubbing a return value
when(mockRepo.findById("42")).thenReturn(Optional.of(user));

// stubbing an exception
when(mockRepo.findById(any())).thenThrow(new DbException());

// verifying an interaction
verify(mockRepo).save(user);
verify(mockRepo, times(2)).save(any());
verify(mockRepo, never()).delete(any());

Testability Red Flags

Flag 01
Static method calls

Can't mock easily. LocalDateTime.now() inside logic forces clock-dependent tests.

Flag 02
new inside constructor

Hard-coded dependencies. No way to substitute a fake.

Flag 03
Private collaborators

If a dependency is created privately and never injected, it can't be mocked.

Flag 04
God methods

Long methods that do I/O, computation, and persistence can't be tested in isolation.

Fix: Constructor Injection

// before (hard to test)
class Service {
  private Repo repo = new Repo();  // baked in
}

// after (testable)
class Service {
  private final Repo repo;
  public Service(Repo repo) { this.repo = repo; }
}
§ 06

SOLID

LetterPrincipleOne-liner
S Single Responsibility A class should have one reason to change.
O Open / Closed Open for extension, closed for modification. Add behavior without editing existing code.
L Liskov Substitution Subtypes must be usable wherever the supertype is expected, without breaking contracts.
I Interface Segregation Many small, focused interfaces beat one fat interface.
D Dependency Inversion Depend on abstractions, not concrete classes. Inject dependencies.

Classic Violations

  • SRP violated: one class does parsing, validation, persistence, and UI.
  • OCP violated: a switch on type that grows every time a new type is added.
  • LSP violated: Square extends Rectangle where setWidth also changes height.
  • ISP violated: a Printer interface forcing fax() and scan() on every implementation.
  • DIP violated: a high-level class new-ing a concrete DB driver.
§ 07

Strategy & Builder

Strategy Pattern

Encapsulates interchangeable algorithms behind a common interface. Lets the caller swap behavior at runtime.

interface PricingStrategy {
  double price(Cart cart);
}
class RegularPricing implements PricingStrategy { ... }
class BlackFridayPricing implements PricingStrategy { ... }

class Checkout {
  private final PricingStrategy pricing;
  Checkout(PricingStrategy p) { this.pricing = p; }
}
When to use: you have a growing if/switch chain selecting between algorithm variants. Strategy turns that into polymorphism. Satisfies OCP.

Builder Pattern

Constructs objects step by step. Great for objects with many optional params or where order of setters matters.

Pizza p = new Pizza.Builder("large")
    .withCheese("mozzarella")
    .withTopping("pepperoni")
    .withTopping("mushrooms")
    .build();
  • Usually a static nested Builder class.
  • Each setter returns this for chaining.
  • Terminal build() returns the immutable product.
  • Avoid telescoping constructors: new Pizza("large", "mozzarella", "pepperoni", null, null, false).

MVC vs MVVM

MVCMVVM
Model Domain + data Domain + data
Middle layer Controller: receives input, mutates model ViewModel: exposes observable state, commands
View Renders model, sends events to controller Binds declaratively to ViewModel observables
Communication Imperative: view asks controller, controller pushes to view Reactive: binding keeps view in sync automatically
Testability Controller testable, view usually not ViewModel highly testable (just data + commands, no UI)
JavaFX angle: FXML + Controller classes are MVC. Observable properties (StringProperty, IntegerProperty) with bindings push you toward MVVM.
§ 09

JavaFX Threading Rules

Rule 01
Only FX Thread touches UI

Any read/write of scene graph nodes must happen on the JavaFX Application Thread.

Rule 02
Never block the FX Thread

Long work (I/O, sleeps, big computation) freezes the UI. Offload to background.

Rule 03
Hop back with runLater

After background work finishes, publish results to UI via Platform.runLater or equivalent.

The BackgroundTaskRunner Shape

// 1. event handler fires on FX thread
button.setOnAction(e -> {
    status.setText("Loading...");          // OK, on FX

    backgroundTaskRunner.run(() -> {
        // 2. this runs on a background thread
        Data d = slowNetworkCall();
        return d;
    }, result -> {
        // 3. callback runs back on FX thread
        label.setText(result.summary());      // OK, back on FX
    });
});
Wrong: calling label.setText() from inside the background lambda. That's a UI mutation off the FX thread, and JavaFX will throw.
Also wrong: calling Thread.sleep(5000) or doing blocking I/O directly in the event handler. Freezes every animation and input until it returns.
§ 10

Concurrency

Why i++ is a Race

i++ is not atomic. It's three steps: read i, add 1, write i. Two threads can each read the same value, both add 1, both write the same result. One increment is lost.

// two threads each do 1000 iterations of counter++
// expected: 2000
// actual: something less, nondeterministic

Fixes

Fix 01
synchronized

Mutual exclusion on a block or method. Only one thread holds the lock at a time.

Fix 02
AtomicInteger

Lock-free atomic ops like incrementAndGet(). Fast for simple counters.

Fix 03
ReentrantLock

Explicit lock/unlock. More flexible (tryLock, fair ordering) but requires try/finally.

Deadlock

Two threads hold locks the other needs, and neither releases. Classic pattern: thread A locks X then Y, thread B locks Y then X.

  • Prevent: always acquire locks in the same global order.
  • Detect: thread dumps show blocked threads waiting on held locks.
  • Avoid holding multiple locks when possible.

Visibility (happens-before)

Without synchronization, a write on one thread may never be seen by another. Use volatile, synchronized, or java.util.concurrent classes to establish visibility.

§ 13

The 8 Fallacies of Distributed Computing

Assumptions naive developers make that aren't true in distributed systems:

#FallacyReality
1The network is reliablePackets drop, connections die.
2Latency is zeroRound trips cost milliseconds to seconds.
3Bandwidth is infinitePipes have limits; big payloads throttle.
4The network is secureAssume eavesdroppers and attackers. Encrypt.
5Topology doesn't changeNodes join, leave, move, reroute.
6There is one administratorMultiple orgs, different policies, different uptime.
7Transport cost is zeroSerialization, encryption, routing all cost CPU and money.
8The network is homogeneousDifferent protocols, platforms, vendors.
Exam tip: if a question asks what assumption a buggy distributed design is making, it's almost always one of these 8.

Open-Source Licensing

License Copyleft? Attribution Commercial use Key obligation
MIT No Required Yes Include copyright + license text. That's it.
Apache 2.0 No Required Yes MIT-ish, plus explicit patent grant and NOTICE file.
GPL (v2/v3) Strong Required Yes (but viral) Any distributed derivative must also be GPL, with source available.
LGPL Weak Required Yes Modifications to the library are GPL, but linking apps aren't forced open.
GPL viral effect: if you ship a product that statically links GPL code, the whole product must be GPL and you must provide source. This is why companies often avoid GPL dependencies.
Permissive (MIT/Apache): safe for closed-source products. Just credit upstream.
§ 15

ADR (Architecture Decision Record) Template

A short document capturing one architectural decision and why. Lives in version control next to the code.

Skeleton

# Title: short noun phrase naming the decision

## Status
proposed | accepted | deprecated | superseded by ADR-NNN

## Context
The forces at play. What problem are we solving? What constraints exist?
Technical, business, team, timeline. Be honest about tradeoffs.

## Decision
What we're doing. One clear sentence, then supporting detail.

## Consequences
What becomes easier, what becomes harder, what risks we accept,
what we'll need to revisit.

What Makes a Good ADR

  • Short. One page, not ten.
  • Written at decision time, not reverse-engineered later.
  • Records alternatives considered, not just the choice.
  • Immutable once accepted. Supersede with a new ADR instead of editing.
  • Numbered sequentially (ADR-001, ADR-002, ...) for easy cross-reference.
MediQueue-style answer pattern: state the decision, list 2-3 alternatives rejected with reasons, name concrete consequences (positive and negative).
