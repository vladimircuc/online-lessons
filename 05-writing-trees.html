Part 1 of 7 · The foundation

The universal template

This is the hardest topic on his exam, but it's not because the problems are complex. It's because most students don't have a template. Once you give them one, every T.2 problem becomes fill-in-the-blank. This lesson drills that template until it's automatic.

The universal tree-function template

Every recursive function over a BTree looks like this:

funcName :: BTree a -> ResultType
funcName Empty                = baseCase
funcName (BNode v left right) = combine v (funcName left) (funcName right)

Two equations. Always. For every T.2 problem your student sees.

  • Equation 1: the Empty case. What's the "nothing" answer? A number like 0, a boolean like False, a list like [], or an empty tree Empty.
  • Equation 2: the BNode case. Destructure into v, left, right. Combine the current value with the recursive results.

Why it works for every problem

A BTree is either Empty or a BNode, those are the only two shapes. Two equations = exhaustive coverage. And because a BNode contains two more BTrees (left and right), recursion is the natural tool. The template encodes exactly this structure.

Concrete example: sumTree

Sum all the integer values in a tree.

sumTree :: BTree Integer -> Integer
sumTree Empty                = 0
sumTree (BNode v left right) = v + sumTree left + sumTree right
  • Empty case: an empty tree sums to 0.
  • BNode case: the sum is v (current value) plus the sum of the left subtree plus the sum of the right subtree. Plus-combine.

Another: treeDepth

Compute the depth (height) of a tree.

treeDepth :: BTree a -> Int
treeDepth Empty                = 0
treeDepth (BNode _ left right) = 1 + max (treeDepth left) (treeDepth right)
  • Empty case: no depth, 0.
  • BNode case: the current node adds 1, plus whichever subtree is deeper. (We use _ because we don't need the value itself.)

Another: mirror

Build a new tree with left and right children swapped at every node.

mirror :: BTree a -> BTree a
mirror Empty                = Empty
mirror (BNode v left right) = BNode v (mirror right) (mirror left)
  • Empty case: mirror of nothing is nothing.
  • BNode case: rebuild with v, and the recursive-mirrored subtrees swapped.

What makes the bodies different?

Look at the three examples side by side. Same skeleton, different combinations:

FunctionEmptyCombine
sumTree0v + left + right
treeDepth01 + max left right
mirrorEmptyBNode v right left (swapped!)

The skeleton is identical. What changes is (1) what the empty answer is and (2) how you combine the current value with the recursive results.

Your teaching superpower

If your student can write this template from memory, 80% of T.2 problems are solved. The rest is figuring out two small pieces: the base case value, and the combining logic. Drill the template first, then practice filling it in.

The two variations you'll actually see

His exam adds two wrinkles to the basic template. Both are covered in later parts:

  1. Extra pattern for leaves: sometimes you need a third equation for "BNode v Empty Empty" (a leaf), to handle it specially. The smallLeaf problem uses this.
  2. Guards to decide between two behaviors: sometimes the function prunes or transforms based on the value. The prune problem uses this.

We'll see both in Parts 4 and 5. Both are small additions to the base template, not replacements.

Part 2 of 7 · Warmup

Warmup: filling in the template

Before the exam problems, let's practice plugging into the template. Four small functions, each showing how the base case and combine differ. If these feel easy, the exam will feel easy.

Function 1: countNodes

Count the total number of BNodes in a tree.

Thinking it through An empty tree has 0 nodes. A non-empty tree has 1 node (this one) plus however many are in the left subtree, plus however many in the right. So combine is 1 + left + right, independent of the node's value. 
countNodes :: BTree a -> Int
countNodes Empty                = 0
countNodes (BNode _ left right) = 1 + countNodes left + countNodes right

Notice the _: we don't need the value, we're just counting.

Function 2: countLeaves

Count only leaf nodes (nodes where both children are Empty).

Thinking it through Empty contributes 0 leaves. But wait, we need a special case for leaves themselves: a BNode with both children Empty counts as 1. An internal BNode (at least one non-empty child) contributes 0 for itself, but recurses. This is our first time using the "extra pattern for leaves" trick. 
countLeaves :: BTree a -> Int
countLeaves Empty                  = 0
countLeaves (BNode _ Empty Empty)  = 1
countLeaves (BNode _ left right)   = countLeaves left + countLeaves right

Three equations now. Haskell tries them top to bottom:

  1. If Empty → 0.
  2. Else if BNode with both children Empty (a leaf) → 1.
  3. Else (internal node) → recurse.
Order matters

The leaf pattern must come before the general BNode pattern. Otherwise the general pattern catches everything and the leaf case never fires.

Function 3: hasValue

Return True if the tree contains a specific target value.

Thinking it through Empty tree contains nothing, so False. For a BNode: either this node IS the target (True), or we need to look in either subtree. Combine with OR. 
hasValue :: Integer -> BTree Integer -> Bool
hasValue _ Empty = False
hasValue target (BNode v left right) =
  v == target || hasValue target left || hasValue target right

Here the function takes two arguments: the target and the tree. Note the _ on the first argument in the Empty case, since we don't care about the target when the tree is empty.

The combine is OR-combine: True if any of the three parts is True.

Function 4: doubleAll

Return a new tree where every integer has been doubled.

Thinking it through Empty maps to Empty (nothing to double). For a BNode, we return a new BNode with v * 2 and recursively-doubled subtrees. This is a rebuild, the shape stays the same but we create a new tree. 
doubleAll :: BTree Integer -> BTree Integer
doubleAll Empty                = Empty
doubleAll (BNode v left right) = BNode (v * 2) (doubleAll left) (doubleAll right)

When the return type is BTree, the combine is usually BNode (some v) (recurse left) (recurse right). Rebuild the tree with transformed parts.

Rebuild, don't modify

Critical Haskell concept: trees are immutable. doubleAll doesn't change the original tree; it constructs a brand new one. The original is untouched. This idea is central to the prune problem in Part 5.

Part 3 of 7 · The method

The 3 questions

Before writing any T.2 function, your student should ask himself three questions. The answers determine everything. This is the meta-skill: a checklist that converts a word problem into a completed function, every time.

The checklist

The 3 questions for every T.2 problem
  1. What's the Empty answer? If the tree is empty, what should the function return? (A number, a boolean, a list, an Empty tree?)
  2. What data do I need from the current node? Do I need v? Do I need both left and right? Or just one?
  3. How do I combine the value with the recursive results? Addition? OR? Rebuild with BNode? Conditional on a guard?

Answer those three, and the function writes itself. Let's see it in action.

Applying the checklist

Example · treeMax

Find the largest value in a non-empty tree.

Answering the 3 questions
  1. Empty answer: problem says non-empty, so we don't need to handle Empty cleanly. We'll still need base cases, though, for leaves where children are Empty.
  2. Data needed: v (the candidate max), plus recurse on left and right.
  3. Combine: take the max of v and the subtree maxes.

Based on those answers, we write:

treeMax :: BTree Integer -> Integer
treeMax (BNode v Empty Empty) = v
treeMax (BNode v Empty right) = max v (treeMax right)
treeMax (BNode v left Empty)  = max v (treeMax left)
treeMax (BNode v left right)  = max v (max (treeMax left) (treeMax right))

Four cases because "treeMax Empty" isn't defined. For each shape of BNode (leaf, one child missing, both children), we compute the right max. No Empty equation at all.

When to skip Empty

Usually you always handle Empty. The rare exception is when the problem explicitly says "non-empty input" (like treeMax). Even then, you still need leaf-level base cases.

Example · insertLeft

Insert a new value as a leaf on the leftmost path of a tree, replacing the leftmost Empty.

Answering the 3 questions
  1. Empty answer: create a new leaf holding the target value. BNode target Empty Empty.
  2. Data needed: v and left. Right stays put.
  3. Combine: rebuild the node with the same v, the recursively-updated left, and right unchanged.
insertLeft :: Integer -> BTree Integer -> BTree Integer
insertLeft target Empty                = BNode target Empty Empty
insertLeft target (BNode v left right) = BNode v (insertLeft target left) right

Clean template. Empty case creates the new leaf. BNode case rebuilds, recursing only on the left.

Example · sumEvens

Sum only the even-valued nodes in a tree.

Answering the 3 questions
  1. Empty answer: 0 (no evens in nothing).
  2. Data needed: v (to check if even), plus recurse on both.
  3. Combine: if v is even, add it; otherwise don't. Either way, always sum the subtree sums. Guard needed.
sumEvens :: BTree Integer -> Integer
sumEvens Empty = 0
sumEvens (BNode v left right)
  | even v    = v + sumEvens left + sumEvens right
  | otherwise = sumEvens left + sumEvens right

This introduces guards on the BNode case. Same idea as guards everywhere: branch based on a condition, here we branch on whether we include v in the sum or not. The recursion on both subtrees happens regardless.

Spotting each pattern

Over enough problems, you'll see the same handful of combine patterns repeat:

Result typeTypical combineExample problem
Integerv + left + right or similar arithmeticsum all values
Int1 + left + rightcount nodes
Boolf v ∣∣ left ∣∣ rightdoes any node match?
Boolf v && left && rightdo all nodes match?
[a]left ++ [v] ++ rightin-order traversal

BTree aBNode v right leftmirror

Memorize this table. When your student sees a problem, he matches the result type, finds the typical combine, and adjusts. Speeds everything up.

The meta-strategy

Ask your student to state the 3 answers out loud before writing any code. "Empty returns X. I need Y from the node. I combine with Z." Once he can do that verbally, writing the code is trivial.

Part 4 of 7 · Exam problem 1

Exam: smallLeaf

First of the two exam functions. smallLeaf asks whether the tree has at least one leaf whose label is a lowercase letter. We'll apply the 3 questions, handle the "leaf" pattern trick, and verify against his examples.

The problem statement

smallLeaf tr determines whether there is at least one leaf node in tr whose label is a lowercase letter.

Signature:

smallLeaf :: BTree Char -> Bool

The examples we must match

Recall the example trees:

pear = BNode 'A' (BNode '?' Empty Empty)
                 (BNode 'c' (BNode 'b' Empty Empty) (BNode 'G' Empty Empty))

fig  = BNode 'r' (BNode 'c' (BNode 'E' Empty Empty) (BNode '#' Empty Empty))
                 (BNode 'M' (BNode 'Q' Empty Empty) Empty)
      pear                            fig
       A                              r
      / \                            / \
     ?   c                          c   M
        / \                        / \  /
       b   G                      E   #  Q

Expected:

  • smallLeaf pear = True, because leaf 'b' is lowercase.
  • smallLeaf fig = False, because leaves are 'E', '#', 'Q', none lowercase.
Critical definition

A leaf is a node with BOTH children being Empty. In pear, the letter 'c' is NOT a leaf (it has children b and G). Only '?', 'b', and 'G' are leaves. Internal node labels don't count, even if they're lowercase.

Applying the 3 questions

Answering the 3 questions
  1. Empty answer: there are no leaves in an empty tree, so return False.
  2. Data from current node: we need v (to check if lowercase), and whether this node is a leaf (both children Empty). So we'll need a special pattern for leaves.
  3. Combine: for a leaf with lowercase v, return True. For any other BNode, recurse on both subtrees and combine with OR. ("True if either side finds one.")

Building the function

We need three equations:

  1. smallLeaf Empty = False (Empty case)
  2. smallLeaf (BNode v Empty Empty) = isLower v (leaf case)
  3. smallLeaf (BNode _ left right) = smallLeaf left || smallLeaf right (internal case)

Here's the full function:

import Data.Char    -- for isLower

smallLeaf :: BTree Char -> Bool
smallLeaf Empty = False
smallLeaf (BNode v Empty Empty) = isLower v
smallLeaf (BNode _ left right)  = smallLeaf left || smallLeaf right

Reading each equation:

  • Line 1 (Empty): no leaves, no lowercase.
  • Line 2 (leaf): this is a leaf (both children are Empty). Check if its label is lowercase with isLower.
  • Line 3 (internal): not a leaf. Ignore the value (_), check both subtrees with OR.
Order of equations matters

The leaf pattern (line 2) must come before the general BNode pattern (line 3). Otherwise the general pattern catches all BNodes and the leaf check never fires.

Common mistakes on this problem

Missing the leaf pattern

Writing only 2 equations (Empty, general BNode) and checking isLower v in the general case. That would count any lowercase node, not just leaves. Problem requires leaf specifically.

Wrong combine for internal nodes

Using && instead of ||. The problem says "at least one" leaf, which means OR. Using AND would require every subtree to have a small leaf.

Order flipped

Putting the general BNode equation before the leaf equation. The general pattern catches every BNode including leaves, so the leaf case never fires.

Strong pattern: "leaf-aware recursion"

Any time a problem treats leaves differently from internal nodes, you need 3 equations: Empty, leaf (BNode v Empty Empty), general BNode. Memorize this shape, it's reusable.

Part 5 of 7 · Exam problem 2

Exam: prune

The harder of the two. prune removes nodes below a threshold AND their descendants, returning a new tree. This one showcases the "rebuild" pattern: instead of returning a number or boolean, you return a restructured tree. Master this and you can handle any tree-transformation problem.

The problem statement

prune num tr returns the tree obtained from tr by removing any node (plus its descendants) whose label is less than num.

Recall that removing a node is equivalent to replacing it with an empty tree.

Signature:

prune :: Integer -> BTree Integer -> BTree Integer

The example tree

lime :: BTree Integer
lime = BNode 53 (BNode 24 (BNode 14 Empty Empty) (BNode 67 Empty Empty))
                (BNode 23 (BNode 50 Empty Empty) (BNode 18 Empty Empty))
            53
          /    \
        24      23
       /  \    /  \
      14  67  50  18

Expected outputs

CallExpected result
prune 25 limeBNode 53 Empty Empty
prune 20 limeBNode 53 (BNode 24 Empty (BNode 67 Empty Empty)) (BNode 23 (BNode 50 Empty Empty) Empty)

Understanding "remove node plus descendants"

When a node's value is below num, we throw away that entire subtree, not just the one node. Example: if 24 < 25, then removing node 24 also removes its children 14 and 67, even though 67 > 25.

This is counterintuitive. Even a "valid" node can be removed if its ancestor is too small.

Applying the 3 questions

Answering the 3 questions
  1. Empty answer: pruning nothing gives Empty.
  2. Data from current node: we need v (to compare against num), plus left and right to recursively prune them.
  3. Combine: if v < num, return Empty (kill this subtree). Otherwise, rebuild a BNode with v and the pruned versions of both subtrees.

Writing the function

prune :: Integer -> BTree Integer -> BTree Integer
prune _   Empty = Empty
prune num (BNode v left right)
  | v < num   = Empty
  | otherwise = BNode v (prune num left) (prune num right)

Reading each equation:

  • Line 1: pruning an empty tree gives an empty tree. Nothing to do. Use _ for num since it's unused here.
  • Line 2-3: if the current value is below the threshold, nuke the whole subtree by returning Empty. Descendants disappear because we never recurse into them.
  • Line 4: otherwise, keep this node. Rebuild with v unchanged, and pruned versions of both subtrees.

The critical insight: "kill by returning Empty"

How removal works in a functional language

We don't "delete" nodes, we construct a new tree that doesn't contain them. When v < num, we return Empty, and the caller (the parent node's rebuild) uses that Empty in place of the old subtree. The descendants exist in the original tree but not in the new one we're constructing.

Why this is the hardest problem

What makes prune hard for students

Three reasons: (1) the return type is BTree, not a simple value, so the combine is a BNode construction; (2) the "kill subtree" branch doesn't recurse, breaking the usual pattern of "always recurse on both sides"; (3) thinking about "remove node plus descendants" requires understanding that returning Empty at one level propagates up. If your student gets this one, he can handle any tree-transformation problem.

Part 6 of 7 · Deep concepts

Rebuilding, not mutating

One concept separates students who "get" Haskell trees from those who don't: you never modify a tree, you construct a new one. This part cements the mental model and shows patterns that come up constantly.

The core principle

Haskell is immutable. You cannot change a BTree value. What you can do is construct a brand new BTree and return it. The original still exists unchanged. Your function just describes the new tree.

Example: doubleAll revisited

doubleAll :: BTree Integer -> BTree Integer
doubleAll Empty                = Empty
doubleAll (BNode v left right) = BNode (v * 2) (doubleAll left) (doubleAll right)

In Python, you'd probably write code that walks the tree and mutates each node in place. Here, we instead:

  1. Recursively describe a new left subtree where every value is doubled.
  2. Recursively describe a new right subtree.
  3. Build a fresh BNode combining them with the doubled current value.

If you do let newTree = doubleAll oldTree in ghci, oldTree is still the original. Nothing was modified.

Shape preservation vs shape change

Two useful patterns:

Shape preserving (doubleAll, mirror)

preserveShape :: BTree a -> BTree b
preserveShape Empty                = Empty
preserveShape (BNode v left right) = BNode (transform v) (preserveShape left) (preserveShape right)

Every node in the original has a corresponding node in the new tree. Only the values change.

Shape changing (prune)

possiblyRemove :: BTree a -> BTree a
possiblyRemove Empty = Empty
possiblyRemove (BNode v left right)
  | shouldRemove v = Empty
  | otherwise      = BNode v (possiblyRemove left) (possiblyRemove right)

Some branches of the original might not exist in the new tree (replaced with Empty). Shape can shrink.

What about growing?

You can also grow trees. Example: insertLeft from Part 3 extends the leftmost path by one leaf.

insertLeft :: Integer -> BTree Integer -> BTree Integer
insertLeft target Empty                = BNode target Empty Empty
insertLeft target (BNode v left right) = BNode v (insertLeft target left) right

When we hit an Empty, instead of returning Empty, we return BNode target Empty Empty. That grows the tree.

The returning-a-tree pattern cheat sheet

When your function's result type is BTree, here's what the body looks like:

IntentBNode body
Transform every valueBNode (f v) (recurse left) (recurse right)
Swap left/rightBNode v (recurse right) (recurse left)
Conditionally removeguard on v: if kill then Empty else BNode v (recurse) (recurse)
Only recurse on one sideBNode v (recurse left) right (or other side)
Grow (replace Empty)Empty case returns a new BNode instead of Empty

Equational reasoning: why this is safer than mutation

The mental model

In Haskell, prune 20 lime and lime are two values. The first is derived from the second by applying prune. Both exist independently. If you write let x = prune 20 lime; y = prune 100 lime, you now have three values: lime, x, y. No one was modified. This is called equational reasoning, it's why Haskell programs are easier to reason about.

Practical: what about performance?

You might worry: "doesn't constructing a new tree every time use tons of memory?" Answer: Haskell's runtime shares structure cleverly. If you rebuild a tree and most of it stays the same, the new tree shares those subtrees with the original. Only changed parts are actually new.

Your student doesn't need to know this for the exam. Just tell him: "Haskell handles it. Focus on correctness."

One more worked example: replaceValue

Replace every occurrence of old with new in a tree of integers.

3 questions
  1. Empty answer: Empty.
  2. Data needed: v, left, right, the old value, the new value.
  3. Combine: rebuild. If v == old, use new as the value; otherwise use v. Recurse on both sides either way.
replaceValue :: Integer -> Integer -> BTree Integer -> BTree Integer
replaceValue _ _ Empty = Empty
replaceValue old new (BNode v left right)
  | v == old  = BNode new (replaceValue old new left) (replaceValue old new right)
  | otherwise = BNode v   (replaceValue old new left) (replaceValue old new right)

Or more compactly using a helper value:

replaceValue :: Integer -> Integer -> BTree Integer -> BTree Integer
replaceValue _ _ Empty = Empty
replaceValue old new (BNode v left right) =
  BNode (if v == old then new else v)
        (replaceValue old new left)
        (replaceValue old new right)

Both work. The first uses guards, the second uses an inline if. For his exam, guards are more idiomatic.

Part 7 of 7 · Consolidation

Practice & cheat sheet

Eight fresh problems in the exam's style. Apply the 3 questions, use the template, check your answer. Cheat sheet at the bottom. If your student can do all 8 without peeking, he's ready.

Practice problems

Problem 1 · Basic count

Write countOdds :: BTree Integer -> Int that counts how many values in the tree are odd.

Show answer & explanation

Empty: 0. Combine: if v odd, 1 + subtree counts; otherwise 0 + subtree counts. Guards.

countOdds :: BTree Integer -> Int
countOdds Empty = 0
countOdds (BNode v left right)
  | odd v     = 1 + countOdds left + countOdds right
  | otherwise = countOdds left + countOdds right
Problem 2 · All-nodes predicate

Write allPositive :: BTree Integer -> Bool that returns True if every value in the tree is positive. Empty tree is considered True (vacuously).

Show answer & explanation

Empty: True. Combine: current node positive AND both subtrees all positive. Use &&.

allPositive :: BTree Integer -> Bool
allPositive Empty = True
allPositive (BNode v left right) = v > 0 && allPositive left && allPositive right

Note: && short-circuits. If v > 0 is False, the recursive calls don't even run.

Problem 3 · Leaf-aware

Write maxLeaf :: BTree Integer -> Integer that returns the largest value among the leaves of a non-empty tree. You can assume the tree is non-empty.

Show answer & explanation

No Empty case needed (non-empty tree). Leaf case: return v. Internal: handle up-to-3 possibilities (one or both children Empty).

maxLeaf :: BTree Integer -> Integer
maxLeaf (BNode v Empty Empty) = v
maxLeaf (BNode _ Empty right) = maxLeaf right
maxLeaf (BNode _ left Empty)  = maxLeaf left
maxLeaf (BNode _ left right)  = max (maxLeaf left) (maxLeaf right)

Note: internal nodes' values are ignored (the _), we only care about leaves.

Problem 4 · Transform each value

Write addOne :: BTree Integer -> BTree Integer that returns a new tree where every value is incremented by 1.

Show answer & explanation

Empty → Empty. BNode → rebuild with v + 1 and recursed children.

addOne :: BTree Integer -> BTree Integer
addOne Empty                = Empty
addOne (BNode v left right) = BNode (v + 1) (addOne left) (addOne right)
Problem 5 · Collect all values to a list

Write flatten :: BTree a -> [a] that returns all values in the tree as a list, left-to-right (in-order traversal).

Show answer & explanation

Empty → empty list. BNode → left list ++ [v] ++ right list.

flatten :: BTree a -> [a]
flatten Empty                = []
flatten (BNode v left right) = flatten left ++ [v] ++ flatten right

For tiny = BNode 5 (BNode 3 Empty Empty) (BNode 7 Empty Empty) → [3, 5, 7].

Problem 6 · Conditional removal (prune-style)

Write dropNegatives :: BTree Integer -> BTree Integer that removes any node with a negative value AND all its descendants.

Show answer & explanation

Same shape as prune. Empty → Empty. Negative node → Empty. Otherwise → rebuild.

dropNegatives :: BTree Integer -> BTree Integer
dropNegatives Empty = Empty
dropNegatives (BNode v left right)
  | v < 0     = Empty
  | otherwise = BNode v (dropNegatives left) (dropNegatives right)
Problem 7 · Leaf predicate

Write anyLeafPositive :: BTree Integer -> Bool that returns True if at least one leaf has a positive value.

Show answer & explanation

Empty: False. Leaf: v > 0. Internal: OR the subtrees.

anyLeafPositive :: BTree Integer -> Bool
anyLeafPositive Empty = False
anyLeafPositive (BNode v Empty Empty) = v > 0
anyLeafPositive (BNode _ left right)  = anyLeafPositive left || anyLeafPositive right

Same shape as smallLeaf. Drill this pattern, you'll see it again.

Problem 8 · Transform with rebuild

Write capLeaves :: Integer -> BTree Integer -> BTree Integer that returns a new tree where every leaf's value is capped at n (i.e., if a leaf's value exceeds n, replace it with n). Internal node values stay unchanged.

Show answer & explanation

Empty → Empty. Leaf → cap value. Internal → rebuild with unchanged value, recurse on both sides.

capLeaves :: Integer -> BTree Integer -> BTree Integer
capLeaves _ Empty = Empty
capLeaves n (BNode v Empty Empty) = BNode (min v n) Empty Empty
capLeaves n (BNode v left right)  = BNode v (capLeaves n left) (capLeaves n right)

Three-equation shape (Empty, leaf, internal). Same structure as smallLeaf but returning a BTree instead of a Bool.
