Why Bother With Randomness?

Randomness lets the algorithm avoid worst cases. An adversary who knows your algorithm can craft a bad input for a deterministic algorithm. But if the algorithm flips coins, the adversary can't predict which choices it will make, so there's no single "bad input" anymore.

The classic example: deterministic QuickSort with "always pick the first element as pivot" has worst-case O(n²) on sorted input. Randomized QuickSort picks a random pivot and gets expected O(n log n) on every input. We'll see this in Part 3.

Warm-Up: When would you prefer Las Vegas vs Monte Carlo?▼

Las Vegas is better when correctness is non-negotiable (medical diagnosis, financial calculations, sorting data). You always get the right answer; sometimes it just takes longer.

Monte Carlo is better when you need a guaranteed time budget and can tolerate a small error probability (network routing, approximate counting). You always finish on time; occasionally the answer is wrong.

The beautiful thing: if a Monte Carlo algorithm has even a tiny probability of success, you can boost it by running many independent trials. This is Part 6's main topic.

Part 2 / 8 — Fundamentals of Probability
The Toolkit You Need for Analyzing Randomized Algorithms

Before we can analyze any randomized algorithm, we need some probability fundamentals. This section covers exactly the tools used in Parts 3–7, nothing extra.

Sample Spaces, Events, and Probability

A sample space Ω is the set of all possible outcomes of a random experiment. An event A ⊆ Ω is a subset of outcomes. The probability Pr[A] is a number between 0 and 1 measuring how likely A is.

Example: Two Coin Flips

Sample space: Ω = {HH, HT, TH, TT}. Each outcome has probability 1/4.

Event "at least one head" = {HH, HT, TH}. Probability = 3/4.

Event "both heads" = {HH}. Probability = 1/4.

Independence

Events A and B are independent if knowing one happened doesn't change the probability of the other:

Independence
A and B are independent iff Pr[A ∩ B] = Pr[A] · Pr[B] For n independent events: Pr[A₁ ∩ A₂ ∩ ... ∩ Aₙ] = Pr[A₁] · Pr[A₂] · ... · Pr[Aₙ]

This is the key property for boosting: if we run an algorithm multiple times with fresh random coins, the runs are independent, and we can multiply failure probabilities.

Conditional Probability

Conditional Probability & Chain Rule
Pr[A | B] = Pr[A ∩ B] / Pr[B] Chain rule: Pr[A₁ ∩ A₂ ∩ ... ∩ Aₙ] = Pr[A₁] · Pr[A₂|A₁] · Pr[A₃|A₁∩A₂] · ...

The chain rule is exactly what we'll use to analyze Karger's algorithm in Part 5: each contraction succeeds given that all previous contractions succeeded.

Expected Value and Linearity of Expectation

The expected value E[X] of a random variable X is its weighted average outcome: E[X] = Σₓ x · Pr[X = x].

Linearity of Expectation (The Power Tool)

E[X₁ + X₂ + ... + Xₙ] = E[X₁] + E[X₂] + ... + E[Xₙ]

No independence required! This works for ANY random variables, correlated or not. This single fact is behind most of the elegant proofs in randomized algorithms.

Union Bound

Union Bound
Pr[A₁ ∪ A₂ ∪ ... ∪ Aₙ] ≤ Pr[A₁] + Pr[A₂] + ... + Pr[Aₙ]

Useful when you want to bound "the probability that something bad happens." Bound each bad event individually and add them up.

Markov's Inequality

Markov's Inequality
If X ≥ 0 and a > 0, then: Pr[X ≥ a] ≤ E[X] / a

Part 3 / 8 — Las Vegas Algorithms
Always Correct, Sometimes Slow

Formal Definition

Las Vegas Algorithm
A randomized algorithm is Las Vegas if: 1. It ALWAYS returns the correct answer 2. Its runtime is a random variable 3. We analyze the EXPECTED runtime

A Las Vegas algorithm never lies. It might take a while, but the answer is guaranteed correct.

Example: Randomized QuickSort

Randomized QuickSort

RandomizedQuickSort(A): if |A| ≤ 1: return A Pick pivot p uniformly at random from A Partition into L (< p), {p}, R (> p) return QS(L) + {p} + QS(R)

Why Randomness Defeats the Adversary

Deterministic QuickSort (always pick first element): adversary gives sorted array. Every partition is maximally unbalanced. Total: O(n²).

Randomized QuickSort: on any input, the random pivot is equally likely to be any element. Expected comparisons: O(n log n) on every input.

The Analysis (Linearity of Expectation)

Let sorted elements be a₁ < a₂ < ... < aₙ. Define Xᵢⱼ = 1 if aᵢ and aⱼ are ever compared. By linearity:

E[comparisons] = Σᵢ<ⱼ Pr[aᵢ and aⱼ compared]

Key Insight

aᵢ and aⱼ are compared iff one of them is the first element from {aᵢ, aᵢ₊₁, ..., aⱼ} chosen as pivot. If some aₖ with i < k < j is chosen first, aᵢ and aⱼ are separated and never compared.

Pr[aᵢ and aⱼ compared] = 2/(j − i + 1)

Summing: E[comparisons] = Σ Σ 2/(j-i+1) ≤ 2n · Hₙ = O(n log n), where Hₙ ≈ ln n is the harmonic number.

Self-Check: Why can't an adversary force O(n²)?▼

The adversary chooses the input before seeing random choices. No matter what array they pick, the random pivot is equally likely to be any element. The adversary would need to predict which element gets chosen, but that's decided after the input is fixed. Expected comparisons: O(n log n) on every input.

Part 4 / 8 — Monte Carlo Algorithms
Always Fast, Sometimes Wrong

Formal Definition

Monte Carlo Algorithm
A randomized algorithm is Monte Carlo if: 1. Its runtime is deterministic (worst-case bounded) 2. It may return an INCORRECT answer 3. Pr[correct] ≥ some bound (e.g., ≥ 2/3)

Sounds worse than Las Vegas, but the guaranteed runtime is a huge advantage. And as we'll see, even a tiny success probability can be boosted to near-certainty.

One-Sided vs Two-Sided Error

One-Sided Error

If true answer is YES → always says YES
If true answer is NO → says NO with prob ≥ 1/2
Only "false positives," never "false negatives."

Two-Sided Error

Can err in either direction
Pr[correct] ≥ 1/2 + ε for some ε > 0
Right more often than wrong, but can make any type of mistake.

Converting Between Las Vegas and Monte Carlo

Las Vegas → Monte Carlo

Run the Las Vegas algorithm with a time limit T. If it finishes, return its (correct) answer. If not, return a default. Result: Monte Carlo with runtime ≤ T.

Error prob = Pr[LV takes longer than T]. Use Markov's inequality: set T = 2·E[runtime] to get error ≤ 1/2.

Monte Carlo → Las Vegas (when verification is cheap)

Run Monte Carlo, then verify the output. If correct, return it. If not, run again with fresh coins. Expected trials until success: 1/p.

Part 5 / 8 — Karger's Min-Cut Algorithm
Monte Carlo in Action: The Most Elegant Randomized Algorithm

The Global Min-Cut Problem

Given an undirected graph G = (V, E), find a cut (S, V\S) minimizing the number of crossing edges. No source/sink needed.

You could solve this via n−1 max-flow computations. Karger's algorithm is simpler and beautifully illustrates randomized thinking.

Edge Contraction

Contracting Edge (u, v)

1. Merge u and v into a single supernode
2. Keep all edges to other vertices (parallel edges OK)
3. Remove self-loops

A cut survives if the contracted edge was NOT a cut edge. A cut is destroyed if the contracted edge crosses it.

The Algorithm

Karger's Min-Cut Algorithm
Input: Undirected (multi)graph G = (V, E), |V| = n 1. While more than 2 vertices remain: a. Pick edge e uniformly at random b. Contract e (merge endpoints, remove self-loops) 2. Two remaining supernodes define a cut 3. Return this cut
Interactive: Karger's Algorithm Step-by-Step

6 vertices. Min-cut edges shown dashed in orange (size 2). Click to contract random edges.

The Analysis

Fix a minimum cut δ(S*) of size k. The algorithm finds it iff none of the n−2 contracted edges belong to δ(S*).

Step 1: Edge count lower bound

Every vertex has degree ≥ k (otherwise that vertex alone gives a smaller cut). So |E| ≥ nk/2.

Step 2: First contraction

Pr[avoids cut] ≥ 1 − k/(nk/2) = 1 − 2/n

Step 3: Cascade via conditional probability

After i contractions (all avoiding the cut), we have n−i vertices. Same argument:

Pr[contraction i+1 avoids | all previous avoided] ≥ 1 − 2/(n−i)

Step 4: The telescoping product

Success Probability

Pr[success] = ∏ᵢ₌₀ⁿ⁻³ (1 − 2/(n−i)) = (n−2)/n · (n−3)/(n−1) · (n−4)/(n−2) · ... · 1/3 = 2/(n(n−1)) ≥ 2/n²

The factors telescope! Most terms cancel, leaving just the boundary terms.

Part 6 / 8 — Boosting the Probability of Success
From Slim Chance to Near-Certainty

The Core Idea

Run a Monte Carlo algorithm T independent times. The probability that ALL T runs fail:

Boosting by Repetition
Pr[all T fail] = (1 − p)ᵀ Pr[≥ 1 success] = 1 − (1 − p)ᵀ To get failure ≤ δ: T ≥ (1/p) · ln(1/δ) Key fact: (1 − 1/x)ˣ ≤ 1/e for all x ≥ 1
Interactive: (1 − 1/x)ˣ ≤ 1/e

Watch the function converge to 1/e ≈ 0.368 from below as x grows.

Applied to Karger's

Boosted Karger's

p = 2/n², run T = (n²/2)·c·ln(n) times Pr[all fail] = (1 − 2/n²)ᵀ ≤ (1/e)^(c·ln n) [using key fact] = 1/nᶜ

With T = O(n² log n) runs × O(n²) each = O(n⁴ log n) total, error ≤ 1/nᶜ for any constant c!

General Boosting Recipes

Optimization (e.g., Min-Cut)

Run T times, keep the best answer. If any run succeeds, you'll pick up its result.

One-Sided Error Decisions

Run T times. If ANY says YES, output YES. Otherwise NO. (YES is always trustworthy.)

Two-Sided Error Decisions

Run T times, take majority vote. With Pr[correct] ≥ 1/2 + ε, error drops exponentially in T.

Karger-Stein (Brief Mention)

Observation: early contractions are safe (prob 2/n is tiny); the danger is in the last few. Karger-Stein contracts to n/√2 vertices, then branches into two independent runs and recurses. Runtime improves to O(n² log³ n).

Exam Tip

You won't need Karger-Stein details. The important pieces: (1) basic Karger algorithm, (2) success prob ≥ 2/n², (3) boosting argument giving O(n⁴ log n) with high probability.

Cheat Sheet for Exam

• Las Vegas: always correct, expected runtime. Example: randomized QuickSort, E[O(n log n)].

• Monte Carlo: fixed runtime, may err. Example: Karger's, Pr[success] ≥ 2/n².

• Boosting: T = (1/p) · ln(1/δ) independent runs drive failure to δ.

• Linearity of expectation: E[X+Y] = E[X] + E[Y], no independence needed.

• Key inequality: (1 − 1/x)ˣ ≤ 1/e.

• Karger's: contract random edges → 2 remain. Pr[finds min-cut] ≥ 2/n². One run O(n²). Boosted: O(n⁴ log n).

Problem 1Linearity of Expectation: Random Cuts
Expected cut edges under random 2-coloring

A graph has 15 edges. You randomly 2-color each vertex: each is red or blue independently with probability 1/2. An edge is "cut" if its endpoints differ.

(a) Define indicator Xₑ for each edge. What is E[Xₑ]?

(b) Using linearity of expectation, what is E[total cut edges]?

(c) Argue some 2-coloring must cut ≥ 8 edges. (The probabilistic method!)

Show Solution▼

(a) Xₑ = 1 if edge e is cut. Pr[cut] = Pr[different colors] = 1/2. So E[Xₑ] = 1/2.

(b) E[total] = Σ E[Xₑ] = 15 · 1/2 = 7.5. (Linearity applies even though Xₑ's are correlated!)

(c) If E[X] = 7.5, some outcome has X ≥ 7.5. Since X is an integer, some coloring cuts ≥ 8 edges. This is the probabilistic method: proving existence without explicit construction.

Problem 2Las Vegas vs Monte Carlo Conversions
Converting between algorithm types

(a) You have a Las Vegas algorithm with expected runtime O(n³). Convert it to a Monte Carlo algorithm with worst-case runtime O(n³) and error ≤ 1/2.

(b) You have a one-sided-error Monte Carlo for a decision problem: never wrong on YES, correctly says NO with prob ≥ 2/3 on NO-instances. Runtime O(T(n)). Design a new algorithm with error ≤ 1/2ⁿ and state its runtime.

Show Solution▼

(a) Run LV for at most 2n³ steps. If it finishes, return the correct answer. Otherwise output a default. By Markov's: Pr[runtime ≥ 2n³] ≤ E[runtime]/(2n³) = n³/(2n³) = 1/2. ✓

(b) Run n times independently. If ANY run says YES → output YES. If ALL say NO → output NO.

On YES-instances: always correct (zero error). On NO-instances: each run wrongly says YES with prob ≤ 1/3. All n runs wrong: (1/3)ⁿ < 1/2ⁿ. Runtime: O(n · T(n)). ✓

Problem 4Boosting Calculation
Trials and runtime for boosted algorithms

A two-sided-error Monte Carlo succeeds with prob ≥ 2/3 on every input. You want error ≤ 1/n³.

(a) Using majority-vote boosting with T = O((1/ε²)·ln(1/δ)) where ε = 1/6, δ = 1/n³, how many trials?

(b) Each trial is O(n²). Total runtime?

(c) Still polynomial? Justify.

Show Solution▼

(a) T = O((1/(1/6)²)·ln(n³)) = O(36 · 3 ln n) = O(log n).

(b) Total = O(log n) · O(n²) = O(n² log n).

(c) Yes! O(n² log n) is polynomial. Boosting from constant error to 1/n³ cost only a log n factor, negligible. ✓
