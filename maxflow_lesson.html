The Setup
What is a Network?
Real World First

Think of Water Pipes 🚰

Imagine a city's water system. Water flows from a reservoir (source) through a bunch of pipes to your house (sink). Each pipe has a max capacity — it can only carry so much water per second.

A network in your notes is exactly this. It's just a directed graph where:

s = Source
Where everything starts. Water comes out of here. Think: reservoir, power plant, data center.
t = Sink (target)
Where everything ends up. Water arrives here. Think: your house, city, end user.
📐 Formal definition: A network is G = (V, E) with a source s ∈ V, a sink t ∈ V, and a capacity c(e) on every edge e. The capacity says how much can flow through that pipe.

Here's the example from your lecture notes. The numbers on each edge are flow, capacity — so "3, 10" means 3 units are currently flowing through a pipe that can handle 10:

Reading the graph

Edge s→A has capacity 8 and carries 8 units of flow — it's full! Edge A→B has capacity 10 but only carries 3. Edge s→B has capacity 1 and carries 1. Edge A→t carries 5/5. Edge B→t carries 4/8 — could carry more!

🎯 The goal (Flow Problem): Find the assignment of flows to edges that maximizes total flow arriving at t, while respecting capacities and conservation.
02
The Constraints
Rules a Flow Must Follow
Two Laws of Flow

You Can't Break Physics 🔬

A "flow" isn't just any assignment of numbers — it has to follow two laws. Think of water in a pipe system:

Law #1 — Capacity Constraint

You can't overfill a pipe

For every edge e: 0 ≤ f(e) ≤ c(e)

Flow on an edge must be between 0 (nothing flowing) and c(e) (pipe is full). You literally cannot push more water than the pipe can hold.

If pipe A→t has capacity 5, you can send 0, 1, 2, 3, 4, or 5 units through it. Not 6. Never 6.
Law #2 — Conservation Constraint

Water doesn't disappear in the middle

For every node v (except s and t): flow IN = flow OUT

Whatever enters a middle node must leave it. Nothing gets created or destroyed in transit. This is like a junction box — all water that comes in must go out.

If 8 units flow INTO node A, then exactly 8 units must flow OUT of A (split however you want across A's outgoing edges).

s and t are exceptions — s only produces, t only consumes.

Why these rules matter

Together, these two rules define what a valid flow is. The max-flow problem asks: among all valid flows, which one makes the total leaving s as large as possible?

💡 Key insight: total flow out of s = total flow into t. Because of conservation, everything that leaves s must eventually arrive at t (it can't just vanish at a middle node).
03
The Algorithm
Ford-Fulkerson
The Core Idea

Keep Finding Paths, Push Flow

Ford-Fulkerson is beautifully simple in concept. Here's the plain English version:

1. Start with zero flow everywhere.
2. Find ANY path from s to t that still has leftover capacity.
3. Push as much flow as possible through that path.
4. Repeat until no such path exists.
5. You're done — that's the max flow!

That path in step 2 is called an "augmenting path." It's just a route from source to sink with spare capacity.

The Algorithm in Code
FordFulkerson(G, s, t):
  // start with 0 flow on all edges
  for each edge e: f(e) = 0

  while there exists a path P from s to t in residual graph:
    bottleneck = min capacity on path P
    for each edge e in P:
      f(e) += bottleneck // push flow
      f(reverse of e) -= bottleneck // update residual

  return total flow out of s
⚠️ What's a "residual graph"? That's the next chapter — it's the key trick that makes this work. Don't worry about it yet.
Bottleneck

The Weakest Link on a Path 🔗

When you find a path s → A → B → t, you can only push flow equal to the minimum capacity edge on that path. That smallest edge is the bottleneck.

Path has edges with capacities 8, 3, 5? The bottleneck is 3. You can push at most 3 units through this path. The edge with capacity 3 is now full.
04
The Key Trick
Residual Graph
Why we need it

What if Your First Choice Was Wrong? 😬

Here's the problem with a naive approach: what if you pick a path early on that blocks a better solution later? You need a way to undo mistakes.

The residual graph solves this. For every edge, it tracks two things:

Forward residual

c(e) − f(e)
Remaining capacity — how much MORE you can push forward.

Edge has cap 10, flow 3 → forward residual = 7. You can still push 7 more units forward.
Backward residual

f(e)
Flow already sent — how much you can "take back" (undo).

Edge carries 3 units → backward residual = 3. You can cancel up to 3 units (redirect them).

How Fast Is It?
Complexity + Wrap-Up
Two Versions

It Depends How You Find Paths

Ford-Fulkerson's speed depends on HOW you find augmenting paths in step 2. Your notes mention two approaches:

Ford-Fulkerson (DFS)
O(C · E)

C = max capacity value, E = edges. Uses DFS to find paths.

⚠️ If capacities are huge (like C = 1 billion), this can be very slow. It might need a billion iterations.
Edmonds-Karp (BFS)
O(VE²)

V = vertices, E = edges. Uses BFS to always find the SHORTEST path.

✅ Polynomial time — always fast regardless of capacity values. This is the better algorithm.
Why BFS is better

BFS always finds the shortest augmenting path (fewest edges). This guarantees the number of iterations is bounded by O(VE), making the total O(VE²). It doesn't matter if capacities are a trillion — it still terminates quickly.

🧠 Same idea as in your DFS lecture — switching from DFS to BFS changes the complexity from potentially infinite to guaranteed polynomial.
