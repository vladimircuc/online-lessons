Part 1 of 7 · Trees

The BTree data type

Before we trace anything, your student needs to see trees clearly. Haskell represents trees as recursive ADTs, a direct application of Lesson 2's material. Read the definition carefully and everything else in this lesson follows.

The definition

data BTree a = Empty
             | BNode a (BTree a) (BTree a)
             deriving (Show)

Three things to notice:

  1. BTree a is a generic type. The a is a type variable, like [a] for lists. BTree Integer means a tree of integers, BTree Char means a tree of characters.
  2. Two constructors. Empty (leaf-like placeholder, no data) and BNode a (BTree a) (BTree a) (a node holding a value of type a plus two more trees).
  3. The definition is recursive. A BNode contains BTrees inside itself. That's what makes it a tree.

Anatomy of a BNode

Every BNode has exactly three pieces:

BNode value left right
--    ^^^^^ ^^^^ ^^^^^
--      |    |    |
--      |    |    right child (itself a BTree)
--      |    left child (itself a BTree)
--      the value stored at this node

If a child is missing (a leaf with no children below it), you use Empty.

Example: maple

His study guide defines:

maple :: BTree Integer
maple = BNode 33 (BNode 44 (BNode 55 Empty Empty) Empty) (BNode 66 Empty Empty)

That one-liner is a tree. Reading it visually:

            33
           /  \
          44    66
         / \  
        55  Empty

Let me peel back the nesting, you'll need to do this for every trace:

maple = BNode 33 LEFT_SUBTREE RIGHT_SUBTREE

  where
    LEFT_SUBTREE  = BNode 44 (BNode 55 Empty Empty) Empty
    RIGHT_SUBTREE = BNode 66 Empty Empty

And peel once more:

LEFT_SUBTREE of 44:
  left  = BNode 55 Empty Empty   -- leaf with value 55
  right = Empty                  -- no right child

RIGHT_SUBTREE:
  BNode 66 Empty Empty           -- leaf with value 66
Teaching a student to read trees

Most students panic when they see BNode 33 (BNode 44 (BNode 55 Empty Empty) Empty) (BNode 66 Empty Empty). Have him draw the tree on paper first, always. The visual makes the structure obvious. Only after drawing should he attempt a trace.

Terminology

TermMeaning
RootThe topmost node. In maple, it's 33.
LeafA node where both children are Empty. In maple: 55 and 66.
Internal nodeAny non-leaf node. 33 and 44.
SubtreeA whole tree rooted at any node (including children).
Depth / heightLength of the longest path from root to any leaf. maple's height is 3.

Another example: pear

From his T.2 section:

pear :: BTree Char
pear = BNode 'A' (BNode '?' Empty Empty)
                 (BNode 'c' (BNode 'b' Empty Empty) (BNode 'G' Empty Empty))
            A
           / \
          ?   c
             / \
            b   G

Every leaf here (?, b, G) is a BNode with both children Empty. The ASCII art just doesn't show the Empties because they'd clutter the diagram.

Part 2 of 7 · The code

Reading a recursive tree function

Before we trace, let's understand what we're tracing. His exam's charade function is a small recursive function on a tree with a guard. If your student can read it correctly, the trace is just careful bookkeeping.

The function

charade :: BTree Integer -> [Integer]
charade Empty = [0]
charade (BNode v left right)
  | even v    = [1] ++ charade left  ++ [v]
  | otherwise = [2] ++ charade right ++ [v]

Three equations. Let's read each line carefully.

Line 1 · Type signature

charade :: BTree Integer -> [Integer]

Takes a tree of integers, returns a list of integers.

Line 2 · Empty case

charade Empty = [0]

If we hit an Empty node, return [0]. This is the base case, recursion stops here.

Line 3 · Pattern match on BNode

charade (BNode v left right)

Match any BNode. Bind:

  • v = the value at this node
  • left = the left subtree (itself a BTree Integer)
  • right = the right subtree

Lines 4-5 · Guards

  | even v    = [1] ++ charade left  ++ [v]
  | otherwise = [2] ++ charade right ++ [v]

Two guards. Haskell evaluates them top to bottom.

  • If v is even: recurse on the left subtree. Return [1] ++ (result of left) ++ [v].
  • Otherwise (v is odd): recurse on the right subtree. Return [2] ++ (result of right) ++ [v].

The critical observation

This function is NOT a full traversal

Based on whether v is even or odd, only one subtree gets recursed into. The other is ignored entirely. Students often assume recursive tree functions visit every node, this one doesn't. Pay attention to which branch the guard takes.

Mechanically, what does each equation produce?

When the function hits any node, it produces a list built from three parts concatenated with ++:

[1] or [2]    (a tag based on parity)
++
charade child    (the recursive result, a LIST of integers)
++
[v]            (the current node's value, as a singleton list)

Why ++ matters

The operator ++ concatenates two lists.

[1, 2] ++ [3, 4]       -- [1, 2, 3, 4]
[0] ++ [5]             -- [0, 5]
[1] ++ [] ++ [7]       -- [1, 7]

Every charade call returns a list. When we see [1] ++ charade left ++ [v], we're:

  1. Computing the recursive call (getting a list back),
  2. Then concatenating [1], that list, and [v] into one big list.

What we need to do in the trace

For each call to charade, the trace must show:

  1. What call is happening (e.g. charade (BNode 33 ...)).
  2. What pattern matched.
  3. What bindings came out of the pattern (v, left, right).
  4. Which guard was taken (or base case fired).
  5. The expression to evaluate (e.g. [2] ++ charade right ++ [33]).
  6. Recurse to resolve the inner call.
  7. Substitute the inner result back in.
  8. Compute the final list for this call.

That's a lot of bookkeeping per call. The template in Part 3 makes it manageable.

Part 3 of 7 · The template

The trace template

Every trace uses the same structure. Drill this template into your student's muscle memory. Follow it and the trace writes itself, skip any step and he'll lose points or get tangled.

The 6-line trace block

For every recursive call, write six lines, always in this order:

The universal call block
1.  CALL:     charade (BNode v_n left_n right_n)
2.  PATTERN:  matched (BNode v left right)
3.  BINDINGS: v = ..., left = ..., right = ...
4.  GUARD:    even v = True/False  →  take this branch
5.  EXPR:     [1] ++ charade left ++ [v]
6.  RETURN:   ... (after recursion completes)

Every call repeats this. If the call hits the base case (Empty), lines 4 and 5 collapse into "base case returns [0]."

Full template for tracing a whole function

The full workflow
  1. Draw the tree on paper. Don't skip this. Even professionals draw the tree.
  2. Write out the function code at the top of your scratch paper for reference.
  3. For each call, write the 6-line block. Indent deeper calls so you can see nesting.
  4. When you recurse, pause the current call ("need charade left first"), start a new block below it indented further.
  5. When the recursive call returns a concrete list, go back to the paused call and substitute.
  6. Compute the final list by concatenating the three pieces.
  7. State the final answer at the bottom.

Visual: what a trace looks like on paper

Here's the shape your student's written trace should take:

charade maple
  CALL: charade (BNode 33 L R)      where L = ..., R = ...
  BINDINGS: v=33, left=L, right=R
  GUARD: even 33 = False → otherwise branch
  EXPR: [2] ++ charade right ++ [33]
  Need: charade right first.

      charade right = charade (BNode 66 Empty Empty)
        BINDINGS: v=66, left=Empty, right=Empty
        GUARD: even 66 = True → even branch
        EXPR: [1] ++ charade Empty ++ [66]
        Need: charade Empty first.

            charade Empty = [0]     (base case)

        BACK: [1] ++ [0] ++ [66] = [1, 0, 66]
        RETURN: [1, 0, 66]

  BACK: [2] ++ [1, 0, 66] ++ [33]
  RETURN: [2, 1, 0, 66, 33]

FINAL: [2, 1, 0, 66, 33]

Notice the indentation. Each inner call is indented further. The outer call is "paused" until the inner one returns. When the inner returns, we go back (outdent) and plug in the result.

Why indentation saves you

Without clear indentation, students mix up which call they're in. They substitute the wrong list, they lose track of bindings, they skip back to the wrong level. Indent religiously, even on a paper exam. It's the difference between full credit and half credit.

The three kinds of calls you'll see

Base case call

CALL: charade Empty
PATTERN: matched Empty (base case)
RETURN: [0]

Short. No bindings, no guard, no recursion.

Recursive call (even branch)

CALL: charade (BNode 4 L R)
PATTERN: matched (BNode v left right)
BINDINGS: v=4, left=L, right=R
GUARD: even 4 = True → even branch
EXPR: [1] ++ charade left ++ [4]
(recurse...)
RETURN: [1] ++ (result) ++ [4]

Recursive call (odd / otherwise branch)

CALL: charade (BNode 7 L R)
PATTERN: matched (BNode v left right)
BINDINGS: v=7, left=L, right=R
GUARD: even 7 = False → otherwise branch
EXPR: [2] ++ charade right ++ [7]
(recurse...)
RETURN: [2] ++ (result) ++ [7]

Minimum a full-credit answer shows

His study guide says explicitly:

"for each recursive call, explicitly indicate the result of the pattern matching (i.e., the bindings to variables), how the result is evaluated, and the final result of that function call."

Three things must be visible per call:

  1. The bindings (what v, left, right got set to).
  2. The evaluation (which guard fired, what expression formed).
  3. The final return value of that call.

Skip any of these and he'll lose partial credit. The 6-line block covers all three with room to spare.

Part 4 of 7 · Warmup

Warmup: a tiny trace

Before we tackle his actual exam, let's do a smaller trace to feel out the template. We'll use a simpler function on a tiny tree. Once you can do this one smoothly, the big one is just more of the same.

The setup

A simple recursive tree function:

sumTree :: BTree Integer -> Integer
sumTree Empty                = 0
sumTree (BNode v left right) = v + sumTree left + sumTree right

Reading it: for Empty, return 0. For a node, return v plus the sum of the left subtree plus the sum of the right subtree. This is a classic full traversal, it visits every node.

Our test tree: tiny

tiny :: BTree Integer
tiny = BNode 5 (BNode 3 Empty Empty) (BNode 7 Empty Empty)
       5
      / \
     3   7

We'll trace sumTree tiny. Expected answer: 5 + 3 + 7 = 15.

The full trace

Trace · sumTree tiny
CALL #1 · sumTree tiny
CALL: sumTree (BNode 5 (BNode 3 Empty Empty) (BNode 7 Empty Empty))
PATTERN: matched (BNode v left right)
BINDINGS:
v = 5
left = BNode 3 Empty Empty
right = BNode 7 Empty Empty
EXPR: 5 + sumTree left + sumTree right
Need sumTree left first...
CALL #2 · sumTree left
CALL: sumTree (BNode 3 Empty Empty)
PATTERN: matched (BNode v left right)
BINDINGS:
v = 3
left = Empty
right = Empty
EXPR: 3 + sumTree Empty + sumTree Empty
CALL #3 · sumTree Empty (left of 3)
PATTERN: matched Empty (base case)
returns 0
CALL #4 · sumTree Empty (right of 3)
PATTERN: matched Empty (base case)
returns 0
BACK TO CALL #2
Substitute: 3 + 0 + 0 = 3
returns 3
CALL #5 · sumTree right
CALL: sumTree (BNode 7 Empty Empty)
BINDINGS: v=7, left=Empty, right=Empty
EXPR: 7 + sumTree Empty + sumTree Empty
CALL #6 · sumTree Empty
returns 0
CALL #7 · sumTree Empty
returns 0
BACK TO CALL #5
Substitute: 7 + 0 + 0 = 7
returns 7
BACK TO CALL #1
Substitute: 5 + 3 + 7 = 15
returns 15

What to notice

  1. Each call has its own bindings. Call #1 has v = 5. Call #2 also has v, but it's 3. The names get re-used, the values are local to each call.
  2. Indentation shows depth. Call #1 is leftmost. Calls #2 and #5 are its direct children, indented once. Calls #3, #4, #6, #7 are grandchildren, indented twice.
  3. Base case is one line. Empty just returns 0. No need for bindings or guards.
  4. BACK TO CALL #X markers are critical. They show when you're resuming a paused call after recursion.
  5. Substitute, then compute. Once both sub-calls return concrete numbers, you plug them into the paused expression.
Practice before the real one

Have your student write out this trace by hand on paper, copying the structure. Muscle memory matters. By the time he gets to charade maple, the structure should feel automatic.

How many calls total?

For a tree with n internal nodes, sumTree makes roughly 2n + (n+1) calls (each internal node triggers 2 recursive calls, all leaves have 2 Empty children). For tiny, that's 3 internal nodes × 2 = 6 Empty calls + 3 node calls = 9 total... close to our 7 (the root only has 2 children as Empty-wrapping leaves). Point being: tree traces multiply fast. Write small, indent carefully.

Part 5 of 7 · The real deal

Exam: charade maple

The full trace from his study guide. We'll walk the whole thing using the template. Notice how the guard makes this trace much shorter than a full traversal, only one subtree per node gets visited.

Setup recap

charade :: BTree Integer -> [Integer]
charade Empty = [0]
charade (BNode v left right)
  | even v    = [1] ++ charade left  ++ [v]
  | otherwise = [2] ++ charade right ++ [v]

maple :: BTree Integer
maple = BNode 33 (BNode 44 (BNode 55 Empty Empty) Empty) (BNode 66 Empty Empty)
            33
           /  \
          44    66
         / \  
        55  Empty

Prediction before tracing

Before going step-by-step, let's mentally predict the path charade will take. For each node, the guard picks left (even) or right (odd):

  • 33 is odd → go right → 66.
  • 66 is even → go left → Empty.
  • Empty → base case, returns [0].

So we only visit three things: 33, 66, and one Empty. The whole left subtree of the root (44 and 55) never gets visited. Keep this in mind as we trace, it should match.

The full trace

Trace · charade maple
CALL #1 · charade maple
CALL: charade (BNode 33 (BNode 44 ...) (BNode 66 Empty Empty))
PATTERN: matched (BNode v left right)
BINDINGS:
v = 33
left = BNode 44 (BNode 55 Empty Empty) Empty
right = BNode 66 Empty Empty
even 33 = False → take otherwise branch
EXPR: [2] ++ charade right ++ [33]
Need charade right first...
CALL #2 · charade right
CALL: charade (BNode 66 Empty Empty)
PATTERN: matched (BNode v left right)
BINDINGS:
v = 66
left = Empty
right = Empty
even 66 = True → take even branch
EXPR: [1] ++ charade left ++ [66]
Need charade left (= charade Empty) first...
CALL #3 · charade Empty
PATTERN: matched Empty (base case)
returns [0]
BACK TO CALL #2
Substitute: [1] ++ [0] ++ [66]
Compute: [1, 0, 66]
returns [1, 0, 66]
BACK TO CALL #1
Substitute: [2] ++ [1, 0, 66] ++ [33]
Compute: [2, 1, 0, 66, 33]
returns [2, 1, 0, 66, 33]

Final answer

Result
charade maple = [2, 1, 0, 66, 33]
Verify in ghci

Paste charade and maple into practice.hs, save, :r, then:

charade maple
-- [2,1,0,66,33]

Observations

  1. Only 3 calls, not 9. Because each node branched only one way, we visited 2 nodes (33, 66) plus 1 Empty base case. The 44 and 55 nodes were never touched.
  2. The prediction matched. We said we'd visit 33 → 66 → Empty. We did.
  3. Final list structure mirrors the call path. [2, 1, 0, 66, 33]: the 2 came from the odd branch at 33, the 1 from the even branch at 66, the 0 from the Empty base case, then the values appended in return order (66 returned first, 33 last).
Why the full-credit answer is long

His rubric explicitly asks for every binding, every guard evaluation, and every return. Even this "small" trace fills a full page in handwriting. Tell your student to leave plenty of space and indent generously. Cramming will cost points.

A sanity check: what if maple had even at the root?

Imagine maple's root was 32 instead of 33. Now:

  • 32 is even → go left → 44.
  • 44 is even → go left → 55.
  • 55 is odd → go right → Empty.
  • Empty → [0].

Now we'd visit 32, 44, 55, Empty. The 66 subtree would be skipped instead. Different path, same mechanical process.

Takeaway: the guards determine the path. Trace the guards carefully, every node.

Part 6 of 7 · Exam answer format

Writing the exam answer

The trace is one thing, writing it down neatly for a grader is another. This part shows your student the exact format his grader expects: what to put on the page, in what order, with examples of what full credit looks like vs partial credit.

What a full-credit handwritten answer looks like

Your student's exam answer for charade maple should look roughly like this on paper:

Tracing charade maple:

Call 1: charade (BNode 33 L R)
  where L = BNode 44 (BNode 55 Empty Empty) Empty
        R = BNode 66 Empty Empty

  Match (BNode v left right):
    v = 33, left = L, right = R

  even 33 = False, so take otherwise branch.
  Result = [2] ++ charade right ++ [33]
         = [2] ++ charade (BNode 66 Empty Empty) ++ [33]

  Need charade (BNode 66 Empty Empty):

    Call 2: charade (BNode 66 Empty Empty)

      Match (BNode v left right):
        v = 66, left = Empty, right = Empty

      even 66 = True, so take even branch.
      Result = [1] ++ charade Empty ++ [66]

      Need charade Empty:

        Call 3: charade Empty
          Matches Empty. Returns [0].

      Back to Call 2:
      Result = [1] ++ [0] ++ [66] = [1, 0, 66]
      Returns [1, 0, 66].

  Back to Call 1:
  Result = [2] ++ [1, 0, 66] ++ [33] = [2, 1, 0, 66, 33]

Final answer: charade maple = [2, 1, 0, 66, 33]

What each call MUST have

Per-call rubric
  1. Label the call (Call 1, Call 2, or a clear identifier).
  2. State the call expression (what function applied to what arguments).
  3. State which pattern matched (Empty? BNode?).
  4. State the bindings (v = ..., left = ..., right = ...).
  5. State which guard fires (or "base case").
  6. Write out the expression to be evaluated (with the recursive call(s) still unresolved).
  7. Resolve the recursive call(s) in separate indented blocks.
  8. Substitute and compute the final list for this call.
  9. State "returns X" explicitly.

Common mistakes that cost points

Skipping bindings

Writing "we recurse on the right subtree" without stating what v, left, right are explicitly. The grader can't verify your reasoning without the bindings.

Skipping the guard evaluation

Writing "[2] ++ charade right ++ [v]" without first writing "even 33 = False, take otherwise". You must show the decision explicitly.

Losing track of which call you're in

Substituting into the wrong paused expression. Indent religiously and use "Back to Call N" markers.

Arithmetic errors in ++

[2] ++ [1, 0, 66] ++ [33] is [2, 1, 0, 66, 33], not [2, [1, 0, 66], 33]. Concatenation flattens, it doesn't nest.

Forgetting the base case returns a list, not a number

charade Empty = [0]. Singleton list containing zero, not the number zero. When you substitute, it's [1] ++ [0] ++ [66], with the [0] remaining a list.

Abbreviations that are okay

For a long trace, some shortcuts are acceptable:

  • Naming big subtrees with letters (L, R, T1, T2) and defining them once at the top.
  • Writing "match (BNode v left right)" once and "same as Call 1 pattern" in subsequent calls.
  • For repeated Empty-returns: "Call N: charade Empty = [0]" on one line.
Practice format: 3 full credit signs

When grading his own practice traces, your student should look for: (1) every call has bindings listed, (2) every guard evaluation is shown, (3) every return is stated. If any of those is missing on any call, the trace loses points.

Time management

A trace of charade maple takes about 5-8 minutes written carefully. If your student is spending 15 minutes on one, he's either overwriting or panicking. Practice until he can do a 3-call trace in under 6 minutes.

Practice & cheat sheet

Two fresh trace problems. Work them fully using the template, then check against the answer. At the end, the full cheat sheet for quick reference.

Practice setup

Paste this into practice.hs for verification:

data BTree a = Empty | BNode a (BTree a) (BTree a) deriving (Show)

sumTree :: BTree Integer -> Integer
sumTree Empty                = 0
sumTree (BNode v left right) = v + sumTree left + sumTree right

zigzag :: BTree Integer -> [Integer]
zigzag Empty                = []
zigzag (BNode v left right)
  | v > 0     = [v] ++ zigzag right
  | otherwise = [v] ++ zigzag left

collect :: BTree Integer -> [Integer]
collect Empty                = [-1]
collect (BNode v left right)
  | even v    = collect left ++ [v] ++ collect right
  | otherwise = [v]

Tree shape:

         3

Trace:

Call 1: zigzag (BNode 3 L R)

Problem 3 · Guard with both recursive calls

Trace collect called on:

blend = BNode 4 (BNode 2 Empty Empty) (BNode 3 Empty Empty)

Show every call, all bindings, every guard evaluation, and the final result.

Show trace & answer

Tree shape:

     4
    / \

Trace:

Call 1: collect (BNode 4 (BNode 2 Empty Empty) (BNode 3 Empty Empty))
  Match (BNode v left right):
    v = 4, left = BNode 2 Empty Empty, right = BNode 3 Empty Empty
  Guard: even 4 = True → even branch
  Expr: collect left ++ [4] ++ collect right
  Need collect left first:

    Call 2: collect (BNode 2 Empty Empty)
      Match: v = 2, left = Empty, right = Empty
      Guard: even 2 = True → even branch
      Expr: collect Empty ++ [2] ++ collect Empty

        Call 3: collect Empty → [-1] (base case)
        Call 4: collect Empty → [-1] (base case)

      Back to Call 2: [-1] ++ [2] ++ [-1] = [-1, 2, -1]
      Returns [-1, 2, -1].

  Need collect right:

    Call 5: collect (BNode 3 Empty Empty)
      Match: v = 3, left = Empty, right = Empty
      Guard: even 3 = False → otherwise branch
      Expr: [3]
      Returns [3].
      (Notice: no recursion in this branch!)

  Back to Call 1: [-1, 2, -1] ++ [4] ++ [3] = [-1, 2, -1, 4, 3]
  Returns [-1, 2, -1, 4, 3].

Notice: the odd branch returns [v] with no recursion at all. That's why collect (BNode 3 ...) stops immediately, the Empty children of 3 are never visited.
