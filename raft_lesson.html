What is Raft?

Understanding the problem before writing a single line of code

The Problem

Imagine you run a bank with one server storing all account balances. If that server crashes, everything is lost. So you run 3 servers that all keep the same copy of the data. Now if one dies, the other two still have everything.

But this creates a new problem: if a customer sends "deposit $100," which server handles it? What if two servers process different requests in different orders? Now they have different data, and you're in trouble.

Raft's Solution

Raft is a set of rules (a protocol) that solves this. The core idea is simple:

  1. One server is elected as the leader
  2. ALL client requests go through the leader
  3. The leader tells all other servers (followers) what to do, in what order
  4. If the leader dies, the remaining servers hold an election and pick a new leader

Three Server States

StateWhat it doesWhen
FollowerListens to the leader, does what it saysDefault state, where everyone starts
CandidateTrying to become leader, asks others for votesWhen a follower's election timer expires
LeaderSends heartbeats, handles all client requestsAfter winning an election with majority votes

Terms (Election Rounds)

Raft divides time into terms, numbered 1, 2, 3, etc. Each term starts with an election. If a leader is elected, it serves for the rest of that term. If the election fails (split vote), a new term starts immediately.

Terms are crucial because they prevent old leaders from causing problems. If server 0 was leader in term 3, then crashed and came back, but server 1 is now leader in term 5, server 0 sees the higher term and immediately steps down.

Go Quick Review

The essentials you'll use in this project, mapped to Python and Java

Variables

Go gives you two ways to declare variables. The short form := is used inside functions. The long form var is used for struct fields or package-level variables.

// Short form (type inferred, like Python)
x := 5
name := "Andrew"
isLeader := true

// Long form (explicit type, like Java)
var x int = 5
var name string = "Andrew"

Functions

Return type comes after the parameters. Go can return multiple values, which is used heavily in Raft.

// Single return
func add(a int, b int) int {
    return a + b
}

// Multiple returns (used by GetState, Start, etc.)
func GetState() (int, bool) {
    return 5, true
}

// Calling it:
term, isLeader := GetState()

For Loops

Go only has for. No while keyword. But for can do everything:

// Classic (like Java)
for i := 0; i < 10; i++ {
    fmt.Println(i)
}

// While-style (just a condition)
for x < 100 {
    x = x * 2
}

// Infinite loop
for {
    // runs forever, like while(true)
}

// Range loop (like Python's "for i in range")
for i, val := range mySlice {
    fmt.Println(i, val)
}

If/Else

Same as Java but no parentheses around the condition. Braces are mandatory.

if term > currentTerm {
    currentTerm = term
    state = 0  // become follower
} else if term == currentTerm {
    // same term
} else {
    // stale term, ignore
}

Slices (Dynamic Arrays)

Like ArrayList in Java or list in Python.

nums := []int{1, 2, 3}

Maps

Like HashMap in Java or dict in Python.

Pointers & Structs

How Go organizes data, and why pointers matter for Raft

Structs (Go's version of classes)

Go doesn't have classes. Instead, it has structs, which are just collections of fields. Then you attach methods to them separately.

// Define a struct (like a class without methods)
type Server struct {
    Name    string
    Term    int
    IsAlive bool
}

// Create one
s := Server{Name: "alpha", Term: 1, IsAlive: true}
fmt.Println(s.Name)    // "alpha"
s.Term = 2             // modify a field

In the Raft project, the Raft struct holds all the state for one server: its current term, who it voted for, whether it's the leader, its log, etc.

Methods on Structs

In Java, methods live inside the class. In Go, you define them separately with a receiver:

// Java style:
// class Server {
//     public String status() { return this.Name; }
// }

// Go style:
func (s *Server) status() string {
    return s.Name    // s is like "this" in Java or "self" in Python
}

The (s *Server) part is the receiver. It says "this method belongs to Server, and s is the instance."

Pointers: The & and *

In Java, when you do Dog d = new Dog(), d is automatically a reference (pointer). You're already using pointers, Java just hides the syntax.

Go makes it explicit:

x := 5          // x is an int, value is 5
p := &x         // p is a pointer to x (& = "address of")
fmt.Println(*p) // *p = "value at this address" = 5
*p = 10         // change the value through the pointer
fmt.Println(x)  // x is now 10
SymbolMeaningExample
*intType: "pointer to an int"var p *int
&xGet the address of xp = &x
*pGet the value p points tofmt.Println(*p)

Why *Raft in method receivers?

func (rf *Raft) GetState() (int, bool) {
    return rf.currentTerm, rf.state == 2
}

The *Raft means "rf is a pointer to a Raft struct." This lets the method read and modify the actual server's data. Without the *, Go would give you a copy, and any changes wouldn't affect the original.

Key Insight

Every time you see rf.currentTerm or rf.state in the Raft code, it's accessing the real server's data through a pointer. rf = this in Java = self in Python.

Why &args and &reply in RPCs?

reply := RequestVoteReply{}                          // create empty struct
ok := rf.sendRequestVote(server, &args, &reply)     // pass addresses
// now reply.VoteGranted has the answer

You create an empty reply struct, then pass its address with &. The RPC function fills it in. If you passed the struct directly (without &), the function would fill in a copy, and you'd never see the result.

Think of it as: "here's a blank form on MY desk, write the answer on it" vs "here's a photocopy, write on that" (you never see the photocopy).

Goroutines & Channels

Running code in parallel and communicating between parallel tasks

Why Goroutines?

In Raft, the leader needs to send heartbeats to all followers at the same time. If it sent them one by one, and one server was slow or dead, the whole system would stall.

// WITHOUT goroutines (slow, sequential)
sendHeartbeat(server1)   // wait 100ms...
sendHeartbeat(server2)   // wait 100ms...
sendHeartbeat(server3)   // wait 100ms...
// total: 300ms

// WITH goroutines (fast, parallel)
go sendHeartbeat(server1)   // fires immediately
go sendHeartbeat(server2)   // fires immediately
go sendHeartbeat(server3)   // fires immediately
// total: ~100ms, all running at the same time

The go keyword starts a function in the background. Your code keeps running immediately.

Anonymous Goroutines

Often you define the function right there inline instead of making a separate named function:

// Named function
go sendHeartbeat(server1)

// Anonymous function (same idea, just inline)
go func() {
    fmt.Println("running in background")
}()   // the () at the end CALLS the function

That () at the end is important. Without it, you're just defining the function but never running it.

The (i) Pattern in Loops

for i := 0; i < 5; i++ {
    go func(server int) {
        fmt.Println("sending to", server)
    }(i)   // passes current value of i into "server"
}

Why pass (i)? Because the goroutine runs in the background, and by the time it actually starts, the loop has already moved on. The variable i might be 5 by then. Passing (i) captures the value at that moment.

Common Bug

Forgetting to pass (i) is one of the most common Go mistakes. All goroutines end up using the same (final) value of the loop variable.

Channels

Goroutines run in the background, but how do you get results back? You use a channel, which is a pipe between goroutines.

ch := make(chan bool)    // create a pipe that carries booleans

// Goroutine sends a value INTO the pipe
go func() {
    ch <- true          // send true into channel
}()

// Main code receives FROM the pipe (waits until something arrives)
result := <-ch           // result is true

The <- operator shows direction:

SyntaxDirectionWhat it does
ch <- valueSend INTO channelPuts a value into the pipe
result := <-chReceive FROM channelWaits and pulls a value out

Key detail: receiving from a channel blocks (waits) until someone sends. This is how goroutines synchronize.

Real Raft Example: Collecting Votes

A candidate needs to ask all peers for votes in parallel and count the results:

voteCh := make(chan bool)    // create the pipe

// Send vote requests to all peers in parallel
for i := range rf.peers {
    if i != rf.me {              // skip myself
        go func(server int) {
            reply := RequestVoteReply{}
            ok := rf.sendRequestVote(server, &args, &reply)
            voteCh <- (ok && reply.VoteGranted)
        }(i)
    }
}

// Sit and collect results
votes := 1                           // count my own vote
for i := 0; i < len(rf.peers)-1; i++ {
    if <-voteCh {                    // wait for next result
        votes++
    }
}
// now "votes" has the total count

Mutexes & Locking

Preventing goroutines from corrupting shared data

The Problem

Multiple goroutines access the same Raft struct at the same time. The ticker goroutine might be reading rf.currentTerm while an RPC handler is writing to it. This causes a race condition, which means unpredictable behavior.

// Goroutine A (election)          // Goroutine B (heartbeat received)
rf.currentTerm = 5                  rf.currentTerm = 10
rf.state = 1                        rf.state = 0
// Who wins? Unknown. Data corrupted.

The Solution: Mutex

A mutex (mutual exclusion) is like a bathroom lock. Only one goroutine can hold it at a time:

rf.mu.Lock()           // "I'm using the data, everyone else wait"
rf.currentTerm = 5
rf.state = 1
rf.mu.Unlock()         // "Done, next person can go"

If another goroutine tries to Lock() while it's already locked, it just waits until the first one calls Unlock().

The defer Pattern

defer means "run this when the function returns." It guarantees the unlock happens even if you return early:

func (rf *Raft) GetState() (int, bool) {
    rf.mu.Lock()
    defer rf.mu.Unlock()    // runs automatically when function exits
    return rf.currentTerm, rf.state == 2
}

This is the cleanest pattern and you'll use it constantly.

Critical Rule

Never hold the lock while sending an RPC. RPCs can take a long time or never return (dead server). If you hold the lock while waiting, no other goroutine can access the struct. Everything freezes. This is a deadlock.

// BAD: deadlock risk
rf.mu.Lock()
rf.sendRequestVote(server, &args, &reply)   // might wait forever
rf.mu.Unlock()

// GOOD: lock, copy what you need, unlock, then send
rf.mu.Lock()
args := RequestVoteArgs{Term: rf.currentTerm}
rf.mu.Unlock()

rf.sendRequestVote(server, &args, &reply)   // no lock held

rf.mu.Lock()
// process the reply safely
rf.mu.Unlock()

The Raft Struct

The data every Raft server needs to keep track of

What We Need to Add

Based on the Raft paper's Figure 2, every server needs:

type Raft struct {
    mu    sync.Mutex
    peers []*labrpc.ClientEnd
    me    int
    dead  int32

    // Persistent state (Figure 2)
    currentTerm int        // latest term this server has seen
    votedFor    int        // who we voted for in current term (-1 = nobody)
    log         []LogEntry // the log of commands

    // Volatile state
    commitIndex int        // highest log index known to be committed
    lastApplied int        // highest log index applied to state machine

    // Election state
    state         int       // 0=follower, 1=candidate, 2=leader
    lastHeartbeat time.Time // when we last heard from leader/granted vote

    // Leader-only state (Part 3B)
    nextIndex  []int        // for each peer, next log index to send
    matchIndex []int        // for each peer, highest log index replicated

    // Channel
    applyCh chan ApplyMsg    // where to send committed entries
}

What Each Field Does

FieldPurposeExample
currentTermWhich election round we're inStarts at 0, goes up each election
votedForWho we voted for this term-1 means "haven't voted yet"
stateAm I a follower, candidate, or leader?0, 1, or 2
logList of commands in orderMinimal for 3A, full for 3B
lastHeartbeatLast time we heard from the leaderUsed to detect election timeout
applyChChannel to report committed commandsTest listens on this channel

The LogEntry Struct

type LogEntry struct {
    Command interface{}    // the actual command (any type)
    Term    int            // term when the leader received it
}

Election Flow

Step by step, how a leader gets elected

The Full Sequence

Click "Next Step" to walk through an election with 3 servers:

Step 1 All servers start as followers. Each one sets a random election timeout (say 350ms, 420ms, 380ms). They all start sleeping in the ticker() loop.
Step 2 Server 0's timer expires first (350ms). It hasn't heard from any leader, so it decides to run for leader. It transitions from follower to candidate.
Step 3 Server 0 prepares for election. It increments currentTerm from 0 to 1, votes for itself (votedFor = 0), and resets its election timer.
Step 4 Server 0 sends RequestVote RPCs to server 1 and server 2 in parallel (using goroutines). The message says: "I'm server 0, term 1, please vote for me."
Step 5 Server 1 receives the RequestVote. It checks: Is term 1 >= my term? Yes. Have I voted for someone else in term 1? No. So it grants the vote, updates its own term to 1, and resets its election timer.
Step 6 Server 2 does the same thing. Grants the vote to server 0.
Step 7 Server 0 counts votes. It has 3 votes (itself + server 1 + server 2). That's a majority of 3. Server 0 becomes the leader!
Step 8 Server 0 immediately starts sending heartbeats (empty AppendEntries RPCs) to servers 1 and 2 every ~100ms. This tells them: "I'm still alive, don't start your own election."
Step 9 Servers 1 and 2 receive heartbeats and reset their election timers each time. As long as heartbeats keep coming, they stay as followers. The system is stable.

What Happens When a Leader Dies?

Step 1 Server 0 (leader) gets disconnected by the test. Servers 1 and 2 stop receiving heartbeats.
Step 2 Server 2's election timer expires. It becomes a candidate, increments to term 2, votes for itself.
Step 3 Server 2 sends RequestVote to servers 0 and 1. Server 0 is disconnected, so that RPC fails/times out. Server 1 votes yes.
Step 4 Server 2 has 2 votes out of 3 (itself + server 1). That's a majority. Server 2 becomes the new leader and starts sending heartbeats.
Step 5 Server 0 reconnects. It still thinks it's leader of term 1. But when it receives any message with term 2, it sees the higher term and immediately steps down to follower.

The Key Rule About Terms

Most Important Rule in Raft

If a server EVER sees a term higher than its own (in any RPC request or response), it immediately: (1) updates its term to the higher one, (2) becomes a follower, (3) resets votedFor to -1. This applies to leaders too. This is how old leaders step down.

1. Make() Initialization

func Make(peers []*labrpc.ClientEnd, me int,
          applyCh chan ApplyMsg) *Raft {
    rf := &Raft{}
    rf.peers = peers
    rf.me = me

    // Initialize state
    rf.currentTerm = 0
    rf.votedFor = -1
    rf.state = 0              // start as follower
    rf.log = []LogEntry{{}}   // dummy entry at index 0
    rf.lastHeartbeat = time.Now()
    rf.applyCh = applyCh

    go rf.ticker()
    return rf
}

2. GetState()

func (rf *Raft) GetState() (int, bool) {
    rf.mu.Lock()
    defer rf.mu.Unlock()
    return rf.currentTerm, rf.state == 2
}

3. ticker()

func (rf *Raft) ticker() {
    for rf.killed() == false {
        time.Sleep(50 * time.Millisecond)

        rf.mu.Lock()
        state := rf.state
        elapsed := time.Since(rf.lastHeartbeat)
        rf.mu.Unlock()

        if state == 2 {
            // Leader: send heartbeats
            rf.sendHeartbeats()
        } else {
            // Check election timeout (random 300-500ms)
            timeout := time.Duration(300+rand.Intn(200)) *
                       time.Millisecond
            if elapsed > timeout {
                rf.startElection()
            }
        }
    }
}

4. startElection()

This is the most complex function. The candidate must:

  1. Increment term, vote for itself, reset timer
  2. Send RequestVote RPCs to all peers in parallel
  3. Count votes
  4. If majority, become leader and send immediate heartbeats

5. RequestVote Handler

When another server asks you to vote:

  1. If their term < your term: reject
  2. If their term > your term: update your term, become follower
  3. If you haven't voted yet (or already voted for them): grant vote
  4. Reset election timer when granting a vote

6. AppendEntries (Heartbeat)

This RPC doesn't exist in the skeleton, you create it from scratch:

  1. Define AppendEntriesArgs and AppendEntriesReply structs
  2. Write the handler: check term, reset election timer, reply
  3. Write a sendHeartbeats() function that sends to all peers

Common Pitfalls

The bugs that trip up almost every student

1. Holding Lock During RPC

This is the #1 cause of deadlocks. RPCs can take forever if the server is dead.

// BAD
rf.mu.Lock()
rf.peers[i].Call("Raft.AppendEntries", &args, &reply)
rf.mu.Unlock()

// GOOD
rf.mu.Lock()
args := AppendEntriesArgs{Term: rf.currentTerm, LeaderId: rf.me}
rf.mu.Unlock()
rf.peers[i].Call("Raft.AppendEntries", &args, &reply)
rf.mu.Lock()
// handle reply
rf.mu.Unlock()

2. Not Checking Term After RPC Returns

While your RPC was in flight, the world might have changed. Maybe a new election happened and your term is stale now.

// After any RPC returns, always check:
rf.mu.Lock()
if rf.currentTerm != savedTerm || rf.state != 2 {
    rf.mu.Unlock()
    return  // things changed while we were waiting, bail out
}
// ... safe to proceed
rf.mu.Unlock()

3. Forgetting to Reset Election Timer

You should only reset the election timer when:

  • You grant a vote to a candidate
  • You receive a valid AppendEntries from a current leader
  • You start a new election (as candidate)

Do NOT reset it just because you received any RPC. If you get a stale message from an old term, don't reset.

4. All Servers Electing at the Same Time

If all election timeouts are the same, all servers become candidates simultaneously, all vote for themselves, nobody gets a majority. The fix is randomization:

// Each check uses a DIFFERENT random timeout
timeout := 300 + rand.Intn(200)  // 300ms to 500ms

5. Heartbeats Too Slow or Too Fast

  • Too slow: followers time out and start unnecessary elections
  • Too fast: the test requires no more than 10 heartbeats per second

Good range: send heartbeats every 100-150ms. Election timeout: 300-500ms.

6. Not Updating Term When You See a Higher One

Never Forget This

In EVERY RPC handler and EVERY RPC response, check if the other server's term is higher than yours. If it is: update your term, become follower, reset votedFor to -1. Missing this check in even one place can cause split-brain (two leaders).

7. Goroutine Loop Variable Bug

// BUG: all goroutines might use same i
for i := range rf.peers {
    go func() {
        sendTo(i)    // i might be wrong by now
    }()
}

// FIXED: pass i as parameter
for i := range rf.peers {
    go func(server int) {
        sendTo(server)
    }(i)
}

The Race Detector

The -race flag is your best friend. If you forgot to lock the mutex somewhere, it catches it:

WARNING: DATA RACE
Write at 0x00c0000b4088 by goroutine 19:
  raft.(*Raft).startElection()
      raft.go:142

Previous read at 0x00c0000b4088 by goroutine 22:
  raft.(*Raft).ticker()
      raft.go:189

This tells you exactly which lines have a race condition. Add mutex locks around those accesses.

Pro Tip

Run the tests at least 10 times in a row. Concurrency bugs are often intermittent, they might pass 9 times and fail once. If it passes all 10, you're in good shape.
