1. Procedures (Functions)

A procedure is a named block of reusable code. You define it once, then call it whenever you need it. AP pseudocode uses the keyword PROCEDURE.

Defining a Procedure

PROCEDURE greet()
{
    DISPLAY("Hello!")
}

← Calling it:
greet()     ← Displays "Hello!"
greet()     ← Displays "Hello!" again

The code inside the procedure doesn't run until you call it. Defining it just saves it for later.

Why Use Procedures?

The exam will ask about benefits of procedures. There are four main reasons:

1. Reusability  ← Write once, use many times
2. Readability  ← Code is easier to understand
3. Debugging   ← Fix a bug in one place, fixed everywhere
4. Abstraction ← Hide complex details behind a simple name

2. Parameters vs Arguments

Procedures become powerful when they accept inputs.

Parameter = the variable name in the definition

Argument = the actual value you pass in

PROCEDURE greetUser(name)   ← "name" is the PARAMETER
{
    DISPLAY("Hello, " + name)
}

greetUser("Alice")   ← "Alice" is the ARGUMENT
greetUser("Bob")     ← "Bob" is the ARGUMENT

When you call greetUser("Alice"), the parameter name takes on the value "Alice" for that call.

Multiple Parameters

PROCEDURE add(a, b)
{
    DISPLAY(a + b)
}

add(3, 5)     ← Displays 8. a=3, b=5
add(10, 20)   ← Displays 30. a=10, b=20

Parameters are matched by position. The first argument goes to the first parameter, second to second, etc.

The parameter only exists inside the procedure. You can't use name outside of greetUser. It's local to that procedure. The exam tests this.

Quick Check: Parameters

PROCEDURE mystery(x, y) { DISPLAY(x * y + 1) } mystery(4, 3) What does it display?
x=4, y=3 4 * 3 + 1 = 13 DISPLAY → 13
PROCEDURE printDouble(n) { DISPLAY(n * 2) } x ← 7 printDouble(x + 3) What displays?
The argument is x + 3 = 7 + 3 = 10 So n = 10 inside the procedure. 10 * 2 = 20 DISPLAY → 20 The argument expression is evaluated FIRST, then the result is passed in.

3. RETURN Values

So far our procedures just DISPLAY things. But procedures can also send a value back to wherever they were called. That's what RETURN does.

RETURN sends a value back

PROCEDURE square(n)
{
    RETURN(n * n)
}

result ← square(5)    ← result = 25
DISPLAY(result)         ← Displays 25
DISPLAY(square(3))     ← Displays 9

The procedure computes n*n and sends it back. You can store the returned value in a variable or use it directly.

RETURN immediately exits the procedure

PROCEDURE check(n)
{
    IF (n > 0)
    {
        RETURN("positive")
    }
    ELSE
    {
        RETURN("not positive")
    }
    DISPLAY("this never runs")  ← DEAD CODE
}

Once RETURN executes, the procedure is done. Any code after RETURN never runs.

DISPLAY vs RETURN are NOT the same. DISPLAY prints to screen but doesn't send anything back. RETURN sends a value back to the caller but doesn't print anything. The exam loves testing this distinction.

Quick Check: RETURN

PROCEDURE max(a, b) { IF (a > b) { RETURN(a) } ELSE { RETURN(b) } } x ← max(8, 12) y ← max(x, 20) DISPLAY(y)
max(8, 12): 8>12? No → RETURN 12. x = 12. max(12, 20): 12>20? No → RETURN 20. y = 20. DISPLAY → 20 You can use procedure calls AS arguments to other procedure calls. Inner evaluates first.
PROCEDURE mystery(n) { IF (n MOD 2 = 0) { RETURN(n / 2) } RETURN(n * 3 + 1) } DISPLAY(mystery(6)) DISPLAY(mystery(7))
mystery(6): 6 MOD 2=0? Yes → RETURN 6/2 = 3 mystery(7): 7 MOD 2=0? No → RETURN 7*3+1 = 22 DISPLAY → 3, then 22 (This is the Collatz conjecture step function!)

4. Procedural Abstraction & Libraries

Abstraction = Hiding Complexity

When you use a procedure, you don't need to know HOW it works inside. You just need to know:

1. What it's called (the name)
2. What it takes in (the parameters)
3. What it gives back (the return value)

This is procedural abstraction. The internal details are hidden. You use it like a black box.

Libraries / APIs

A library is a collection of pre-built procedures someone else wrote. You use them without knowing (or caring) how they work internally.

← You don't need to know HOW these work:
temperature ← getWeather("Dallas")
sendEmail("hello@test.com", "Hi!")
distance ← calculateDistance(pointA, pointB)

Libraries are a form of abstraction. The exam asks: "What is the benefit of using a library?" Answer: you don't need to write everything from scratch, and you can use tested, reliable code.

API (Application Programming Interface) is the set of procedures, their parameters, and their return values that a library exposes. It's the "user manual" for the library. You don't need the source code, just the API.

Quick Check: Abstraction

A programmer uses a procedure called sortList(myList) from a library. The programmer doesn't know how the sorting algorithm works inside. Is this a problem? Why or why not?
No, this is NOT a problem. This is the whole point of abstraction. The programmer only needs to know: - Name: sortList - Input: a list - Output: the list, sorted How it sorts (bubble sort, merge sort, etc.) is an internal detail that doesn't matter to the person using it.

5. Linear Search vs Binary Search

Two ways to find a value in a list. The exam always tests the difference.

Linear Search

Check every element, one at a time, from start to end.

PROCEDURE linearSearch(list, target)
{
    FOR EACH item IN list
    {
        IF (item = target)
        {
            RETURN(true)
        }
    }
    RETURN(false)
}

Works on ANY list (sorted or unsorted). But slow for large lists because you might check every element.

Binary Search

Cut the list in half each time. Check the middle element, then decide whether to search the left half or the right half.

← List MUST be sorted: [1, 3, 5, 7, 9, 11, 13]
← Looking for 9:

Step 1: Check middle (7). 9 > 7 → search RIGHT half
Step 2: Right half is [9, 11, 13]. Middle is 11. 9 < 11 → LEFT
Step 3: Left is [9]. Found it!

Only 3 steps instead of 5 (linear would need 5 checks to reach 9).

Comparison

Linear SearchBinary Search
Works on unsorted lists?YesNo, must be sorted
Speed (worst case)Check ALL n elementsCheck ~log₂(n) elements
1,000 elementsUp to 1,000 checks~10 checks
1,000,000 elementsUp to 1,000,000 checks~20 checks
Simple to code?YesMore complex
Binary search REQUIRES a sorted list. If the list is unsorted, binary search gives wrong results. This is the most tested fact about binary search. If the exam says "the list is not sorted," binary search cannot be used.

Quick Check: Search

A list has 2,048 sorted elements. What is the maximum number of comparisons binary search needs to find a value?
log₂(2048) = 11 2048 → 1024 → 512 → 256 → 128 → 64 → 32 → 16 → 8 → 4 → 2 → 1 11 comparisons maximum. (Linear search would need up to 2,048!)
A list of student names is stored in random (unsorted) order. Which search can be used? Why?
Only LINEAR search can be used. Binary search REQUIRES the list to be sorted. Since the names are in random order, binary search would not work correctly. You could sort the list first and THEN use binary search, but that adds extra work.

6. Algorithmic Efficiency

The exam asks about how algorithms scale as input gets bigger. This isn't about speed in seconds, it's about how the number of steps grows.

Reasonable vs Unreasonable Time

ReasonableUnreasonable
Growth ratePolynomial: n, n², n³Exponential: 2ⁿ, 10ⁿ, n!
n = 100n² = 10,000 steps2¹⁰⁰ = more than atoms in universe
Practical?Yes, finishes in timeNo, takes too long
ExamplesLinear search, binary search, sortingTrying every combination

How to Recognize on the Exam

Reasonable:
  - Single loop over n items → n steps
  - Nested loop (loop inside loop) → n² steps
  - Binary search → log₂(n) steps
  - All of these are polynomial or better

Unreasonable:
  - Checking ALL subsets of n items → 2ⁿ
  - Checking ALL orderings → n! (factorial)
  - "Doubles with each additional element" → 2ⁿ
The key exam question: "If the input size doubles, what happens to the runtime?" If it roughly doubles → linear (n). If it roughly quadruples → quadratic (n²). If it becomes impossibly large → exponential (2ⁿ). This phrasing shows up constantly.

Heuristic Approach

When a problem has no reasonable algorithm, you can use a heuristic, an approximate solution that's "good enough" but not guaranteed to be perfect. The exam tests this as: "a heuristic may not find the optimal solution, but it runs in reasonable time."

Quick Check: Efficiency

An algorithm checks every possible pair of items in a list of n items. How many checks for n=100? Is this reasonable or unreasonable?
Every pair = n * (n-1) / 2 For n=100: 100 * 99 / 2 = 4,950 checks This is approximately n², which is polynomial. REASONABLE. Even though n² grows fast, it's still polynomial and finishable. 2ⁿ would not be.
A password is 10 characters long using only lowercase letters (26 options each). A brute-force algorithm tries every possible password. How many tries? Reasonable or not?
26¹⁰ = 141,167,095,653,376 (141 trillion) This is exponential growth (26ⁿ). UNREASONABLE. This is why longer passwords are more secure, each additional character MULTIPLIES the work.

7. Random Values & Simulations

RANDOM(a, b)

Generates a random integer between a and b, inclusive (both endpoints included).

RANDOM(1, 6)    ← Random number 1-6 (like a die roll)
RANDOM(1, 100)  ← Random number 1-100
RANDOM(0, 1)    ← Either 0 or 1 (coin flip)

Using RANDOM for Decisions

← Simulating a coin flip:
flip ← RANDOM(1, 2)
IF (flip = 1)
{
    DISPLAY("heads")
}
ELSE
{
    DISPLAY("tails")
}

← Picking a random element from a list:
list ← ["red", "blue", "green"]
idx ← RANDOM(1, LENGTH(list))
DISPLAY(list[idx])

Simulations

A simulation is a program that models a real-world process using randomness and rules. The exam tests why simulations are useful:

1. Cheaper than real experiments
2. Safer (no physical risk)
3. Faster (can compress time)
4. Repeatable (run thousands of trials)
5. Can test "what if" scenarios

But simulations are NOT perfect. They simplify reality and depend on the assumptions built into the model. Different random runs give different results.

Simulations are approximations. They do not guarantee exact results. The exam will ask "which is a limitation of simulations?" and the answer is always about simplification or randomness causing variation.

Quick Check: Random & Simulations

What are ALL possible values of x? x ← RANDOM(1, 5) * 10
RANDOM(1,5) can be: 1, 2, 3, 4, or 5 Multiply each by 10: 10, 20, 30, 40, 50 Those are the 5 possible values of x.
A simulation models traffic flow in a city. After running 1000 trials, it predicts an average commute of 25 min. Does this GUARANTEE that a real commute will be exactly 25 minutes?
No. A simulation is an approximation. Reasons it might differ from reality: - The model simplifies real traffic - Random variations in each trial - It may not account for all factors (weather, accidents, construction) The result is a useful ESTIMATE, not a guarantee.

8. Undecidable Problems

What's an Undecidable Problem?

An undecidable problem is one where no algorithm can ever be written that solves it correctly for ALL possible inputs. It's not that we haven't found the algorithm yet, it's mathematically proven to be impossible.

The Halting Problem

The classic example: "Given any program and its input, will the program eventually stop (halt) or run forever?"

It has been proven that no general algorithm can answer this question for ALL possible programs. Some specific cases can be determined, but no single algorithm works for every case.

Decidable vs Undecidable

Decidable: "Is this number even?" → Always solvable
Decidable: "Is this list sorted?" → Always solvable
Undecidable: "Will this program halt?" → NOT always solvable
Undecidable: "Do these two programs produce
               the same output for ALL inputs?" → NOT always solvable
The exam question is usually: "Which of the following is NOT solvable by an algorithm?" or "Which is an undecidable problem?" The answer involves determining behavior of arbitrary programs for ALL inputs.

Quick Check: Undecidable

Which is undecidable? A. Determining if a number is prime B. Sorting a list of numbers C. Determining if an arbitrary program will produce output for all inputs D. Searching for a value in a list
C. Determining if an arbitrary program will produce output for all inputs. A, B, and D are all decidable. There exist algorithms that solve them for every input. C is a variant of the Halting Problem and is proven to be undecidable.
