Spring, SpEL, and why this version is dangerous

Spring is a Java framework — it handles the boring plumbing of web apps (routing requests, connecting to databases) so you focus on business logic. The assignment scenario uses spring-data-rest-webmvc 2.6.5 — a version with a known critical vulnerability.

UserController.java — normal safe code safe 
@PostMapping("/greet")
public String greet(@RequestParam String name) {
    // safe: just using the string directly, no eval
    return "Hello, " + sanitize(name);
}
UserController.java — vulnerable SpEL code vulnerable 
@PostMapping("/greet")
public String greet(@RequestParam String name) {
    ExpressionParser parser = new SpelExpressionParser();
    // DANGEROUS: evaluating raw user input as an expression!
    Expression exp = parser.parseExpression(name);
    return exp.getValue(String.class);
}

The 7 areas of security — which ones apply here?

The assignment asks you to go through all 7 areas and say which are relevant to this Spring app. Click each card to reveal whether it applies and why.

Input Validation
✓ relevant
SpEL injection happens because user input is processed without validation. This is the #1 concern. Look for @RequestParam values going directly into logic.
Authentication
✓ relevant
It's a web app — it needs to verify who the user is. Check if Spring Security is configured. If endpoints have no auth, that's a finding.
Authorization
✓ relevant
Even after login, users need proper access controls. A regular user shouldn't hit admin endpoints. Look for @PreAuthorize or role checks.
Cryptography
less applicable
Less central here unless the app stores passwords or transmits sensitive data. Mention briefly if passwords aren't hashed.
Error Handling
✓ relevant
Stack traces leaking to the client give attackers a roadmap. Look for bare e.printStackTrace() or exception messages returned directly in responses.
Data Protection
✓ relevant
Any sensitive data the app handles needs protection. Hardcoded credentials in source code are a critical data protection failure.
Code Quality
✓ relevant
Deprecated APIs, unused imports, sloppy patterns — these signal areas where vulnerabilities hide. Spring 2.6.5 itself is a code quality issue (outdated dependency).

Tip: for each relevant area, write 2–3 sentences explaining WHY it matters for this specific Spring app — that's where the rubric points are.

What to find in the code — vulnerabilities explained

When you open the code base in Eclipse, here's exactly what you're hunting for. Tap each one to see the full explanation + how to fix it.

CRITICAL SpEL Injection (CVE-2022-22963) ▶

Spring Expression Language gets evaluated on user-supplied input without sanitization. An attacker can send a crafted request that executes arbitrary code on the server — think "the attacker just got a terminal on your machine."

Look for: ExpressionParser, SpelExpressionParser, StandardEvaluationContext being used with request parameters.

Fix Use SimpleEvaluationContext instead (restricts what expressions can access), or better — remove SpEL evaluation of user input entirely. Validate/whitelist all inputs before they touch any expression evaluator.
HIGH Missing input validation on endpoints ▶

Endpoints accept @RequestParam or @PathVariable values and use them directly in business logic with no null checks, length limits, or format validation.

Look for: Methods annotated @GetMapping / @PostMapping that use request parameters without any if checks or validation annotations.

Fix Add Java Bean Validation annotations: @NotNull, @Size(max=50), @Pattern(regexp="..."). Trigger them with @Valid on method parameters. Spring auto-validates and returns 400 on failure.
MEDIUM Improper error handling ▶

Catch blocks that call e.printStackTrace() or return the raw exception message to the HTTP response. This leaks class names, file paths, and internal details — a roadmap for attackers.

Look for: catch (Exception e) { e.printStackTrace(); } or return e.getMessage(); anywhere in the code.

Fix Use @ControllerAdvice to create a global exception handler. Return generic error messages to clients ("An error occurred"), log details server-side only with SLF4J / Log4j.
HIGH Outdated vulnerable dependency (spring-data-rest-webmvc 2.6.5) ▶

The scenario explicitly tells you the version: spring-data-rest-webmvc 2.6.5. This version is in the CVE database. Your app inherits that vulnerability even if your own code is perfect.

Look for it in: pom.xml — find the <dependency> block with this artifact and check the version tag.

Fix Look up the CVE on nvd.nist.gov to find the patched version. Update the version number in pom.xml. Run mvn dependency:tree to confirm no transitive dependency still pulls in the old version.
HIGH Hardcoded credentials or secrets ▶

Passwords, API keys, or tokens written directly as string literals in Java files. Anyone who can read the source code — or the compiled JAR — gets them instantly.

Look for: String literals that look like passwords, anything assigned to a variable named password, secret, apiKey, token, etc.

Fix Move secrets to environment variables or a config file excluded from version control. Reference them via Spring's @Value("${'{'}db.password{'}'}") pointing to application.properties or environment-level config.

Hunt for vulnerabilities — exactly what to search for

You don't need to read every line. Use Ctrl+H (Find in Files) in Eclipse to search the whole project at once. Here's your hit list:

Search 1 SpelExpressionParser
CRITICAL
▶

What you're looking for: Any file that imports or uses SpelExpressionParser or ExpressionParser with parseExpression() being called on a variable that came from a user request.

What makes it vulnerable: If the argument to parseExpression() is a @RequestParam, @PathVariable, or anything from the HTTP request — that's the vulnerability. User input should never be evaluated as an expression.

When you find it, note: Filename, method name, and line number. Write it down — you'll reference it in Section 2 of the template.
Search 2 spring-data-rest-webmvc
HIGH
▶

Where to look: Open pom.xml directly. Do Ctrl+F inside the file and search for spring-data-rest-webmvc. Find the <version> tag next to it.

What to do: Copy that version number, go to nvd.nist.gov, search for it. You'll find CVEs associated with that version. Note the CVE ID, severity score (CVSS), and the recommended patched version.

When you find it, note: The version number in pom.xml, the CVE ID from NVD, and what version fixes it.
Search 3 printStackTrace
MEDIUM
▶

What you're looking for: Any catch block that calls e.printStackTrace() or returns e.getMessage() directly back in a response. This leaks internal class names and stack details to whoever made the request.

Also search for: return e.getMessage() and ResponseEntity(e.getMessage().

When you find it, note: Which class and method has the bad error handling. One instance is enough to report.
Search 4 password / secret / apiKey
HIGH
▶

What you're looking for: String variables or constants with names like password, secret, apiKey, token, credentials that are assigned a hardcoded string value like "myPassword123".

Also look at: application.properties — if credentials are in there and that file could be committed to git, that's also a finding.

When you find it, note: The variable name, its value (partially), and which file it's in.
Search 5 @RequestParam / @PathVariable with no @Valid
HIGH
▶

What you're looking for: Methods that take @RequestParam String ... or @PathVariable String ... but have no @Valid, @NotNull, @Size, or @Pattern annotations anywhere in sight. This means inputs go into logic completely unchecked.

How to spot it fast: Any controller method (annotated with @GetMapping/@PostMapping) where the parameters are plain String or int with no validation annotations is a finding.

When you find it, note: The endpoint path (like /users/greet), the method name, and what parameters are unvalidated.

Mitigation 1 — SpEL Injection
To mitigate the SpEL injection vulnerability, the development team should replace any use of StandardEvaluationContext with SimpleEvaluationContext, which restricts the expression engine from accessing reflection, class instantiation, and static methods. More broadly, user-supplied input should never be passed directly to an expression parser. All @RequestParam and @PathVariable values should be validated against a strict allowlist using @Pattern(regexp="...") before reaching any expression context. OWASP recommends treating all user input as untrusted and applying input validation at every entry point (OWASP Foundation, 2023).
Mitigation 2 — Outdated Dependency
The development team should upgrade spring-data-rest-webmvc from version [VERSION] to the latest patched release identified on the National Vulnerability Database entry for [CVE-ID]. The version number should be updated in pom.xml and the project rebuilt to confirm the vulnerability is no longer present. Running mvn dependency:tree afterward can confirm no transitive dependency re-introduces the vulnerable version (National Institute of Standards and Technology, [YEAR]).
Mitigation 3 — Error Handling
To address improper error handling, the team should implement a centralized exception handler using Spring's @ControllerAdvice annotation. This handler should catch all unhandled exceptions, log their full details server-side using SLF4J or Log4j2, and return only a generic error message to the client such as "An unexpected error occurred." Removing all e.printStackTrace() calls from production code ensures that internal application details are never exposed through HTTP responses (Manico & Detlefsen, 2015).
Mitigation 4 — Input Validation
All controller endpoints that accept @RequestParam or @PathVariable parameters should be updated to include Java Bean Validation constraints. Adding @NotNull, @Size(max=100), and @Pattern(regexp="[a-zA-Z0-9]+") annotations to method parameters, combined with @Valid or @Validated on the method signature, will cause Spring to automatically reject malformed requests with a 400 Bad Request response before they reach any business logic (OWASP Foundation, 2023).
