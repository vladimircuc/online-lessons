Dependency checks — finding the holes you didn't dig.

Your code can be perfect. But if the libraries it uses have security holes, your app is still vulnerable. This lesson teaches you how to find those holes automatically — then document what you found.

What you'll understand

What Maven and pom.xml actually are, what a dependency check does, how to read the HTML report, and what false positives are.

What you'll do

Add the OWASP plugin to pom.xml, run the scan inside Eclipse, screenshot the report, and fill out the template with your findings.

No Java coding needed

You're editing one XML config file and running one Maven command. The tool does the heavy lifting — you just interpret and document the results.

Concept 1 of 6

What is Maven — and what is pom.xml?

Every Java project needs libraries — other people's code that handles things like web requests, database connections, or security. Maven is the tool that manages all of those for you so you don't have to download and organize them manually.

analogy

Think of Maven like a shopping list manager. Instead of going to the store yourself to buy every ingredient, you write a shopping list (pom.xml) and Maven goes and gets everything for you automatically. If you need a new ingredient, you add it to the list and Maven fetches it.

📄
pom.xml
Tap to expand
Stands for "Project Object Model." It's an XML config file that lives at the root of the project. It tells Maven: what libraries the app needs (dependencies), what version of Java to use, what plugins to run. You never run pom.xml — Maven reads it automatically when you build.
📦
Dependency
Tap to expand
A library your project relies on. Declared as a <dependency> block in pom.xml with a group ID, artifact ID, and version. Maven downloads it from a central repository automatically. Example: Spring itself is a dependency — you don't write Spring, you use it.
🔌
Plugin
Tap to expand
A tool that plugs into Maven's build process. For this assignment, you're adding the OWASP Dependency-Check plugin — it runs as part of Maven and scans all your dependencies against a database of known vulnerabilities (CVEs). You add it once to pom.xml and Maven knows to run it.
🏗️
Build lifecycle
Tap to expand
Maven runs in phases: compile → test → package → verify → install. Plugins hook into these phases. The dependency-check plugin hooks into "verify" — so when you run "mvn verify" it compiles your code AND runs the security scan, producing the HTML report.
pom.xml — what a dependency block looks like 
<dependencies>
  <dependency>
    <groupId>org.springframework.data</groupId>
    <artifactId>spring-data-rest-webmvc</artifactId>
    <version>2.6.5.RELEASE</version>  <!-- this version has known CVEs -->
  </dependency>
</dependencies>

What does a dependency check actually do?

Even if your own code is flawless, your app inherits every vulnerability in every library it uses. A dependency check scans your library versions against a public database of known security holes.

analogy

Imagine the government publishes a list of recalled car parts. A dependency check is like a mechanic running your car's VIN against that list — it cross-references every library version in your pom.xml against the National Vulnerability Database (NVD) and flags any matches. Your job isn't to find the vulnerabilities yourself — the tool does that. You just have to understand and document what it found.

Static testing — no code runs

This is called static testing because it analyzes the code and its dependencies without actually running the app. You don't start a server, you don't send requests — the tool just reads files and compares version numbers against known CVEs.

The OWASP plugin — what you're adding to pom.xml

OWASP (Open Web Application Security Project) makes a free Maven plugin that does the dependency scan. You add one XML block to pom.xml — that's it. No installation, no separate download — Maven fetches the plugin automatically when it sees it in the config.

Before (no plugin)
After (plugin added)
What each line means
<project>
  <groupId>com.snhu.sslserver</groupId>
  <artifactId>ssl-server</artifactId>
  <version>0.0.1-SNAPSHOT</version>

  <dependencies>
    <!-- existing dependencies here -->
  </dependencies>

  <!-- no <build> section yet -->

</project>
<project>
  <groupId>com.snhu.sslserver</groupId>
  <artifactId>ssl-server</artifactId>
  <version>0.0.1-SNAPSHOT</version>

  <dependencies>
    <!-- existing dependencies here -->
  </dependencies>

  <build>
    <plugins>
      <plugin>
        <groupId>org.owasp</groupId>
        <artifactId>dependency-check-maven</artifactId>
        <version>8.4.0</version>
        <executions>
          <execution>
            <goals>
              <goal>check</goal>
            </goals>
          </execution>
        </executions>
      </plugin>
    </plugins>
  </build>

</project>
groupId: org.owasp
Who made the plugin. OWASP = Open Web Application Security Project, the nonprofit that makes this tool.
artifactId: dependency-check-maven
The specific plugin name. Maven uses groupId + artifactId together to look it up in the central repository and download it.
version: 8.4.0
Which version of the plugin to use. 8.4.0 is a recent stable version — the course tutorial may specify a different version, use whatever the tutorial says.
goal: check
The specific action to run. "check" tells the plugin to scan dependencies and generate the report. This runs automatically when you do "mvn verify".

Concept 5 of 6

False positives — when the tool gets it wrong

The dependency-check tool flags anything that might be vulnerable based on version numbers. But sometimes a CVE only applies in very specific configurations — and your app doesn't use that configuration. Those are false positives.

analogy

Imagine a metal detector at an airport beeping because you have a knee replacement — it detects metal, but that metal isn't a threat. The alarm is technically triggered but it's not a real problem. A false positive in dependency scanning is the same — the tool flags a version but the specific vulnerability doesn't apply to how you're using that library.

Examples — real vulnerability vs false positive

REAL
spring-data-rest-webmvc 2.6.5 → CVE-2022-22963
This CVE affects SpEL expression evaluation — which this app actually uses. The vulnerability directly applies. This is a real finding, document it and mitigate it.
FALSE+
tomcat-embed-core 9.0.56 → CVE-2022-23181
This CVE requires local filesystem write access to exploit — it's a local privilege escalation, not a remote attack. If the app runs in a normal container environment without special OS permissions, this vulnerability cannot be triggered. It's a false positive.
FALSE+
spring-core 5.3.x → CVE for a different Spring module
Sometimes a CVE is associated with a jar file by name but actually only affects a specific class or submodule that your app never imports or uses. If you can confirm your code doesn't touch the affected class, it's a false positive.

What to write in the template about false positives

The assignment specifically asks you to discuss why you should filter false positives. The answer: so the dev team focuses on real threats. If every flagged item gets treated as critical, teams waste time fixing things that aren't problems — and might miss the ones that are. Good security practice means reading each CVE description and confirming it actually applies before acting on it.

Assignment Walkthrough · Step 2 of 4

Add the plugin and run the scan

This is the only "coding" in this assignment — pasting an XML block. Then you run one Maven command.

1
Add the OWASP plugin block to pom.xml
Inside the <build><plugins> section, add this block. If there's no <build> section, add the whole thing before the closing </project> tag:
<build> <plugins> <plugin> <groupId>org.owasp</groupId> <artifactId>dependency-check-maven</artifactId> <version>8.4.0</version> <executions> <execution> <goals> <goal>check</goal> </goals> </execution> </executions> </plugin> </plugins> </build>
Use the exact XML from the course tutorial — this is just a reference. The version number might differ.
2
Save pom.xml and update the Maven project
Press Ctrl+S to save pom.xml. Then right-click the project → Maven → Update Project → OK. Eclipse re-reads the pom.xml and downloads the OWASP plugin. You'll see it downloading in the bottom status bar.
3
Run the dependency check
Right-click the project → Run As → Maven build.... In the Goals field, type: verify. Click Run. The Console tab at the bottom will show Maven downloading CVE data and scanning. This takes 5–15 minutes the first time — it has to download the entire NVD database.
Do NOT close Eclipse or cancel — let it finish. You'll see "BUILD SUCCESS" in the console when it's done. If you see "BUILD FAILURE," check the error message — usually it's a network issue or wrong plugin version.
4
Find and open the HTML report
After the build succeeds, the report is at: target/dependency-check-report.html inside your project folder. Navigate to that folder in Windows Explorer, then double-click the HTML file to open it in your browser. You should see the full vulnerability report.
If you can't find the file: right-click the project in Eclipse → Refresh. Then expand the "target" folder in the Project Explorer — the report should appear there.

Assignment Walkthrough · Step 3 of 4

Read the report and take the screenshot

The report is open in your browser. Now you extract the information you need for the template.

1
Screenshot the top of the report (scan information)
The report header shows: project name, scan date, OWASP version, number of dependencies scanned. Press Win + Shift + S → drag to select the top portion of the browser. Paste into your Word template. The rubric specifically requires the scan info to be visible.
2
For each vulnerable dependency — click it and record all details
Click each red/flagged dependency in the report. It expands to show: dependency name and version, CVE ID(s), CVSS score, description of the vulnerability, and links to NVD. Record all of this. You need the dependency name, CVE ID, CVSS score, and description for each one in the template.
3
Click through to NVD for each CVE to get the best solution
In the report, click the CVE link → it takes you to nvd.nist.gov. On the NVD page: read the description, check the CVSS score, look for the "References" section which often links to the fix or patched version. Copy the CVE ID and recommended fix version into your notes.
4
Identify which findings are false positives
For each flagged dependency: read the CVE description carefully. Ask: does this specific vulnerability apply to how this app uses the library? If the CVE requires a specific config or usage pattern your app doesn't have — it's a false positive. Note it with your reason.
When in doubt, assume it's real — it's better to over-report than to miss something in an academic assignment.

In cases where a patched version is not yet available, additional controls such as input validation, network segmentation, or temporary feature disablement should be considered as interim mitigations (National Institute of Standards and Technology, n.d.).
Why filter false positives — required discussion
Filtering false positives from dependency-check results is essential to maintaining an effective security workflow. Dependency-check tools operate by matching library version numbers against CVE records in the National Vulnerability Database. However, a CVE associated with a particular library version may only apply under specific conditions — such as a particular usage pattern, operating system configuration, or application architecture — that may not be present in every project using that version. If all flagged items are treated as confirmed vulnerabilities without verification, development teams risk expending significant time and resources addressing issues that pose no actual threat to their specific application. This can also obscure genuinely critical vulnerabilities by burying them among low-priority alerts, a problem commonly known as "alert fatigue." By evaluating each finding against the application's actual configuration, teams can prioritize real risks and allocate remediation effort more efficiently (OWASP Foundation, 2023).
