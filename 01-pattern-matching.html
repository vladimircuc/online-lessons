Part 1 of 7 · Concept

Pattern matching

Pattern matching is how Haskell destructures data and branches on its shape in a single move. It's the replacement for most if/else, most attribute access, and most "check then extract" logic you'd write in Python or Java. Master this and the rest of Haskell unfolds naturally.

The idea in one sentence

A pattern is a template that describes the shape of some data. When you call a function, Haskell tries to match the input against each pattern top-to-bottom. If it matches, variables in the pattern get bound to the corresponding pieces of the input, and that equation runs.

Why it matters

In Python you'd write:

def describe(n):
    if n == 0:
        return "zero"
    elif n == 1:
        return "one"
    else:
        return "something else"

In Haskell, the same logic:

describe :: Int -> String
describe 0 = "zero"
describe 1 = "one"
describe n = "something else"

Three equations for the same function. Haskell picks the first one that matches. No explicit if. The "condition" is baked into the pattern itself.

Why this is powerful

Pattern matching isn't just about specific values. It can:

  • Match the shape of a list (empty? non-empty? at least three elements?)
  • Match a tuple and pull out its pieces
  • Match a specific constructor from a custom data type
  • Nest, a pattern inside a pattern inside a pattern
  • Use wildcards to ignore parts you don't care about

All with the same unified syntax. That's what makes it the backbone of Haskell programming.

The teaching mental model

Every pattern-matching problem answers one question: "Given this input, which equation matches, and what do the variables bind to?" If your student holds that question in mind, they'll solve every P.1 problem.

The pattern-matching checklist

Walk your student through this every time he sees a pattern:

  1. Identify the shape. Is the data a list? A tuple? A constructor?
  2. Work outside-in. Match the outermost shape first, then what's inside.
  3. Check each binding. For every name in the pattern, what piece of the input does it grab?
  4. Check for failure. If shapes don't line up (list pattern on a non-list, too few elements, etc.), the match fails.

Over the next five parts we'll build this skill up layer by layer: values, lists, tuples, wildcards, constructors. Part 6 walks through all six of his exam problems with this checklist in action.

Part 2 of 7 · Basics

Matching on values

Start simple: patterns that match specific literal values, and the rule that decides which equation runs when you have multiple. Understand this layer and the rest is variation.

Multiple equations, same function

In Haskell you can define the same function multiple times with different patterns for the argument. Haskell tries them in order, using the first one that matches.

describe :: Int -> String
describe 0 = "zero"
describe 1 = "one"
describe 2 = "two"
describe n = "some other number"

Call describe 0, first equation matches, return "zero". Call describe 42, first three equations have specific values that don't match, fall through to the last equation where n binds to 42, return "some other number".

The rules

  1. Top-down. Haskell tries equations in the order you wrote them.
  2. First match wins. Once a pattern matches, that equation runs. Later equations are ignored.
  3. A variable matches anything. n in the last equation is a catch-all, any Int binds to it.
  4. Lowercase = variable, not value. describe n doesn't mean "match the value n", it means "bind the input to the name n."
Order matters

If you reversed the order, putting describe n = "other" first, it would catch everything and the specific cases for 0, 1, 2 would never run. Specific patterns go first, general patterns last.

Literal patterns

Any literal value can be a pattern: numbers, characters, strings, booleans.

reply :: Bool -> String
reply True  = "yes"
reply False = "no"

greet :: Char -> String
greet 'h' = "greeting"
greet 'b' = "farewell"
greet _   = "unknown"

The _ (underscore) is a wildcard, matches anything, doesn't bind a name. We'll see more of it in Part 4.

Put it in practice.hs & test
describe :: Int -> String
describe 0 = "zero"
describe 1 = "one"
describe 2 = "two"
describe n = "something else"

reply :: Bool -> String
reply True  = "yes"
reply False = "no"

Save, in ghci type :r, then:

describe 0
describe 1
describe 42
reply True
reply False

Combining with guards

You've seen guards in Lesson 0. You can combine them with patterns:

classify :: Int -> String
classify 0 = "zero"
classify n
  | n > 0     = "positive"
  | otherwise = "negative"

First equation matches specifically for 0. Second equation catches any other n, then uses guards to branch further.

Key takeaway

"Multiple equations" is just pattern matching with specific values. It's the same mechanism that will later match list shapes, tuples, and constructors. Everything builds on this.

Part 3 of 7 · Lists

Matching on lists

This is the big one. Nearly every Haskell function on lists uses pattern matching. Once you internalize the two basic list patterns, [] and (x:xs), and how they chain, you can solve any list-shape problem.

Lists have two shapes

Recall from Lesson 0: a Haskell list is always either:

  1. Empty, written [].
  2. A head followed by a tail, written (x:xs), read "x cons xs."

That's it. Every list matches one of these two patterns. The pattern [1, 2, 3] is really 1 : 2 : 3 : [] underneath.

The cons pattern (x:xs)

The pattern (x:xs) means: "match a non-empty list. Bind the first element to x, bind the rest (which is itself a list) to xs."

myHead :: [Int] -> Int
myHead (x:_) = x

myLength :: [Int] -> Int
myLength []     = 0
myLength (_:xs) = 1 + myLength xs

mySum :: [Int] -> Int
mySum []     = 0
mySum (x:xs) = x + mySum xs

Let's read each:

  • myHead (x:_): take the first element, ignore the rest. The _ means "I don't need this."
  • myLength []: empty list has length 0. myLength (_:xs): length is 1 plus length of the rest. Classic recursion.
  • mySum: same structure, but uses x because we need the value, not just "there's an element here."
Paste & test

Put these in practice.hs:

myHead :: [Int] -> Int
myHead (x:_) = x

myLength :: [Int] -> Int
myLength []     = 0
myLength (_:xs) = 1 + myLength xs

mySum :: [Int] -> Int
mySum []     = 0
mySum (x:xs) = x + mySum xs

Chaining cons

You can chain : to match deeper into a list.

PatternMeaning
(x:xs)At least 1 element. x = first, xs = rest.
(x:y:xs)At least 2 elements. x = first, y = second, xs = rest.
(x:y:z:xs)At least 3 elements.
(_:y:_)At least 2 elements, bind only the second.

The last name before nothing always grabs "everything else as a list" (which could be empty).

Example

firstThree :: [Int] -> (Int, Int, Int)
firstThree (a:b:c:_) = (a, b, c)

secondElement :: [Int] -> Int
secondElement (_:y:_) = y

Call firstThree [10, 20, 30, 40]: matches with a = 10, b = 20, c = 30, _ = [40]. Returns (10, 20, 30).

Exact-length patterns

Square-bracket patterns match lists of a specific length.

PatternMatches
[]Exactly empty.
[x]Exactly one element.
[x, y]Exactly two elements.
[x, y, z]Exactly three.
Huge distinction

[x, y] means exactly two elements. (x:y:xs) means at least two. They look similar but behave very differently.

f [x, y]     = "exactly 2"
f (x:y:_)    = "at least 2"
-- f [1,2,3] matches the second, NOT the first

What you can't do

You can't pattern match on the end of a list. There's no (xs:y) meaning "everything then last." Lists are front-accessible by design, you'd use drop, last, or reverse for tail access. Every P.1 problem involves front-access, so this limitation won't bite your student.

Can you reuse names?

No. Each name in a pattern must be unique. (x:x:xs) is an error ("conflicting definitions for x"). Use _ if you want to match without binding: (_:_:xs) is fine, _ is the one name allowed to repeat.

Teaching drill

Give your student a pattern and ask: (1) what's the minimum length it matches? (2) what does each variable bind to for input [1,2,3,4,5]? Do this for (x:xs), (x:y:xs), [x, y], (_:y:_). If he gets those four right, he's ready for the exam.

Part 4 of 7 · Tuples & wildcards

Tuples & wildcards

Tuples match by position. The wildcard _ is the universal "I don't care" pattern. Together they let you slice data apart cleanly without naming things you don't need.

Tuple patterns

A tuple pattern matches a tuple of the same size, binding each element by position.

getFirst :: (Int, Int) -> Int
getFirst (a, b) = a

addPair :: (Int, Int) -> Int
addPair (x, y) = x + y

swap :: (a, b) -> (b, a)
swap (x, y) = (y, x)

Notice how clean swap is compared to the fst/snd version from Lesson 0. This is the power of pattern matching.

Larger tuples

middleOfThree :: (Int, Int, Int) -> Int
middleOfThree (_, m, _) = m

sumFour :: (Int, Int, Int, Int) -> Int
sumFour (a, b, c, d) = a + b + c + d

The wildcard _

_ matches anything without binding a name. Use it for parts you don't need.

dropFirst :: (Int, Int, Int) -> (Int, Int)
dropFirst (_, b, c) = (b, c)

secondOfList :: [Int] -> Int
secondOfList (_:y:_) = y

Unlike other names, _ can appear multiple times in a single pattern, because it doesn't actually bind anything.

Critical: [] vs _

This trips up every beginner. They look similar but do opposite things.

PatternMeaningMatches []?Matches [1,2,3]?
[]Specifically the empty list✅ Yes❌ No
_Anything (wildcard)✅ Yes✅ Yes
-- Example: using [] and _ together
describeList :: [Int] -> String
describeList [] = "empty"
describeList _  = "non-empty"

First equation catches empty lists specifically. Second catches everything else (the first didn't match, so we fall through).

Order still matters

If you reversed them, describeList _ would catch all lists including empty ones, and describeList [] would never run. Haskell might even warn you "this pattern is unreachable."

Nested patterns

You can put a pattern inside another pattern.

firstOfFirstPair :: [(Int, Int)] -> Int
firstOfFirstPair ((x, _):_) = x

Let's read this carefully:

  • Outer: (...):_ means "a non-empty list; first element matches (...), rest is ignored."
  • Inner: (x, _) means "the first element is a tuple; first of the tuple binds to x, second is ignored."

Combined: take a list of pairs, grab the first pair, grab the first of that pair.

Call firstOfFirstPair [(1, 2), (3, 4), (5, 6)]: x = 1. Returns 1.

Try it
firstOfFirstPair :: [(Int, Int)] -> Int
firstOfFirstPair ((x, _):_) = x

sumPairs :: [(Int, Int)] -> Int
sumPairs [] = 0
sumPairs ((a, b):rest) = a + b + sumPairs rest
firstOfFirstPair [(1, 2), (3, 4)]
sumPairs [(1, 2), (3, 4), (5, 6)]
-- 1
-- 21

The @ (as-pattern)

Sometimes you want to both destructure a value and keep the whole thing named. Use name@pattern.

describeList :: [Int] -> String
describeList [] = "empty"
describeList all@(x:xs) =
  "the list " ++ show all ++
  " starts with " ++ show x

all@(x:xs) means: match a non-empty list, bind the whole list to all, bind the head to x, bind the tail to xs. You get all three names to work with.

As-patterns show up on his exam (Problem 2 uses one), so don't skip this.

Part 5 of 7 · Constructors

Matching on constructors

His exam introduces custom data types (fully covered in Lesson 2) and then asks P.1 questions about patterns involving them. Here's the minimum you need to know about constructors so the exam problems make sense, the full story comes in P.2.

Custom types have constructors

A custom data type in Haskell is defined by listing its possible "shapes," called constructors. We'll cover this fully in Lesson 2. For now, know the syntax looks like:

data Magic = Charm Integer
           | Spell Char

Read: "A Magic is either a Charm wrapping an Integer, or a Spell wrapping a Char."

So Charm 5 is a Magic. Spell 'J' is also a Magic. Different constructors, same type.

Matching on constructors

Pattern matching on a constructor checks which constructor was used and binds the data inside it.

describe :: Magic -> String
describe (Charm n) = "a charm with number " ++ show n
describe (Spell c) = "a spell with char " ++ [c]

Pattern (Charm n) matches any Charm, binds the Integer inside to n. Pattern (Spell c) matches any Spell, binds the Char inside to c.

Matching specific data inside a constructor

You can narrow further by matching on the specific value inside.

isFive :: Magic -> Bool
isFive (Charm 5) = True
isFive _         = False

(Charm 5) matches only a Charm holding exactly 5. Charm 7, Spell 'x', anything else, all fall through to the wildcard.

Nested constructor patterns

This is where his exam gets interesting. Constructors can contain other constructors, lists, tuples. Patterns nest to match.

data Victory = Self [Integer]
             | Help [Magic]

So Self [2,4,6,8] is a Victory containing a list of integers. Help [Spell 'J', Charm 4] is a Victory containing a list of Magics.

A pattern can peel through multiple layers at once:

firstMagic :: Victory -> Magic
firstMagic (Help (m:_)) = m

Reading (Help (m:_)):

  • Outer: Help (...) means "a Help constructor wrapping something."
  • Inner: (m:_) means "a non-empty list; bind the head to m, ignore the rest."

So it matches a Help whose list has at least one element, and binds that first Magic to m.

Pattern failure

This is crucial for Problem 5 on his exam. A pattern fails when shapes don't match, even at runtime.

Example: the pattern ((Self _):h) expects a list whose first element is a Self. If you give it a bare Self [2,4,6,8] (which is a Victory, not a list), the match fails at the outermost layer. The list cons pattern (...):h demands a list, and a Victory isn't a list.

The shape must match

Before checking names, check the shape. A list pattern won't match a tuple, a tuple pattern won't match a constructor, a constructor pattern won't match a list. Wrong shape = no match.

Summary: four kinds of patterns

Everything on his exam uses some combination of these four:

Pattern kindExampleMatches
Value0, True, 'a'Exactly that literal value
Variablex, xs, nameAnything, binds to name
Wildcard_Anything, doesn't bind
Constructor(x:xs), (a,b), Help m, Self [1,2]Data of that shape, binds interior pieces

All four can nest inside each other.

Problem 1 · Pattern (y:_:r), Data three

Give the resulting value for r.

Step-by-step trace
1
Shape: (y:_:r) is a list pattern with at least 2 elements. three is a list with 3 elements, good, shape matches.
2
Bindings: y = first element, _ = second (ignored), r = rest of the list.
3
three = [Self [10,15], Help [Spell '#'], Self [3,9,8]].
So y = Self [10,15], _ = Help [Spell '#'], r = [Self [3,9,8]].
4
Note: r is the "rest" of the list after removing the first two elements. Since the list had 3 elements and we removed 2, r has 1 element, wrapped in [...].
Answerr = [Self [3,9,8]]
Verify in ghci

Add this to practice.hs:

p1 :: [Victory]
p1 = case three of (y:_:r) -> r

Save, :r, then:

p1

Problem 2 · Pattern y@(d, Self g), Data ([], one)

Give the resulting value for g.

Step-by-step trace
1
Shape: y@(d, Self g) is an as-pattern. The inner pattern (d, Self g) is a tuple of size 2. The data is ([], one), also a 2-tuple, shape matches.
2
Outer bindings: y is the whole tuple, y = ([], one). Then the tuple matches:
d = first element = [].
Second element must match Self g.
3
one = Self [2,4,6,8], which is a Self constructor. Matching Self g against Self [2,4,6,8] binds g = [2,4,6,8].
Answerg = [2,4,6,8]

Step-by-step trace

Problem 5 · Pattern ((Self _):h), Data one

Give the resulting value for h.

Step-by-step trace
1
Shape: Outer is (...):h. That's a list cons pattern, first element is (Self _), rest is h.
2
Check the data's shape: one = Self [2,4,6,8]. That's a Victory, not a list.
3
The pattern demands a list; the data is a Victory. Shapes don't match. Pattern match fails at the outermost layer.

Teaching note

Problem 5 is the classic trap. Students see Self in both the pattern and the data and assume it matches. But the outer shape is wrong, a list pattern (_:h) cannot match a single Victory. Always check the outermost shape first.

Step-by-step trace

2

Teaching summary

Every P.1 problem fits the same template: (1) identify the outer shape, (2) check the data has that shape, (3) peel inward layer by layer, (4) report the bindings. If the outer shape fails, the match fails, don't look deeper.

Give the resulting value for rest.

Show answer & explanation

  • x = 10 (first)
  • y = 20 (second)
  • rest = [30, 40] (everything after the first two)

Problem 2 · Constructor destructuring

Give the resulting value for ms.

Show answer & explanation

alpha = Help [Charm 1, Spell 'A', Charm 9]. Pattern (Help ms) matches a Help constructor, binds the [Magic] inside to ms.

Answer: ms = [Charm 1, Spell 'A', Charm 9]
Problem 3 · Nested list in constructor

Pattern: (Help (_:second:_))   Data: alpha

Give the resulting value for second.

Show answer & explanation

Outer: Help (...) matches the Help constructor. Inner: (_:second:_) matches a list with at least 2 elements.

alpha is Help [Charm 1, Spell 'A', Charm 9]. Inner list has 3 elements.

  • _ = first = Charm 1
  • second = second = Spell 'A'
  • _ = rest = [Charm 9]

Problem 4 · Pattern failure

Pattern: (Self (x:xs))   Data: delta

Give the resulting value for x.

Show answer & explanation

Outer pattern is Self (...). But delta = Help [], a Help constructor. Wrong constructor.

No match. The pattern expects Self but the data uses Help. Pattern match fails at the outermost constructor.

Problem 5 · Another pattern failure

Pattern: (Help (x:xs))   Data: delta

Give the resulting value for x.

Show answer & explanation

Outer: Help (...). delta = Help [], matches the Help constructor ✓.

Inner: (x:xs) demands a non-empty list. But the list inside delta is [], empty. Pattern fails.

No match. The inner pattern (x:xs) cannot match an empty list.

Problem 6 · Tuple-list combo

Pattern: ((m, Self ns):rest)   Data: gamma

Give the resulting values for m and ns.

Show answer & explanation

gamma = [(Charm 7, Self [1,2]), (Spell 'Z', Self [])], a list of 2 tuples.

Outer: (...):rest, non-empty list pattern. First element → inner pattern, rest → rest.

Inner: (m, Self ns), a 2-tuple; first element binds to m, second must be a Self binding its inside to ns.

First element of gamma is (Charm 7, Self [1,2]):

  • m = Charm 7
  • Second tuple element is Self [1,2], matches Self ns, so ns = [1,2]
  • rest = [(Spell 'Z', Self [])]
Answer: m = Charm 7, ns = [1,2]
Problem 7 · As-pattern

Pattern: all@(Self (n:_))   Data: beta

Give the resulting values for all and n.

Show answer & explanation

beta = Self [100, 200, 300].

As-pattern: all binds to the whole thing, the inner pattern still matches too.

  • all = Self [100, 200, 300]
  • Inner: Self (n:_) matches, with the list inside being [100,200,300]. So n = 100.
Answer: all = Self [100, 200, 300], n = 100
Problem 8 · Exact length

Pattern: [a, b, c]   Data: [10, 20]

Give the resulting value for b.

Show answer & explanation

[a, b, c] matches exactly 3 elements. Data has 2.

No match. The exact-length pattern fails because the list doesn't have 3 elements.

(Compare: (a:b:c:_) would also fail here for the same reason, needs at least 3.)
