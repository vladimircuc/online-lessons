Problem 2.1
6 pts — Max Flow on a Specific Graph

Max Flow — The Water Pipes Problem

The Question

You're given a specific graph with edge capacities. Find the maximum flow from s to t and fill in the flow on every edge.

💧

Mental model first: Imagine s is a water reservoir and t is a city. Each edge is a pipe with a maximum capacity. How much water can you push from reservoir to city per second? That's max flow.

The Core Rules (Never Forget These)

Rule 1 — Capacity Constraint

Flow on any edge must be between 0 and its capacity. You can't overfill a pipe.

0 ≤ f(e) ≤ capacity(e)

Rule 2 — Conservation

At every middle node (not s or t): flow in = flow out. Water doesn't appear or disappear in the middle.

Σ flow_in(v) = Σ flow_out(v)

The Max-Flow Min-Cut Theorem

The maximum possible flow equals the minimum cut capacity. A cut is any partition of vertices into two sets (one containing s, one containing t). The capacity of a cut = sum of capacities of edges going from the s-side to the t-side.

So to verify your max flow is actually maximum: find any cut whose capacity equals your flow value. If they match, you're done.

Ford-Fulkerson in Plain English

The algorithm for finding max flow:

  1. Start with zero flow on all edges.
  2. Find any path from s to t with remaining capacity (an augmenting path).
  3. Push as much flow as possible through that path (limited by the bottleneck edge).
  4. Update the residual graph — reduce forward capacity, increase backward capacity.
  5. Repeat until no augmenting path exists.
🔁

Residual graph: after sending flow along an edge, you create a "reverse" edge in the residual graph. This lets the algorithm "undo" mistakes by routing flow backward. The reverse edge capacity = amount of flow already sent on the forward edge.

Problem 2.2
6 pts — Graph Representation for Prim's

Prim's MST — Which Graph Representation?

The Question

You want to run Prim's algorithm on an undirected weighted connected graph with n vertices and m edges, achieving the best possible asymptotic runtime. Should you store the graph as an adjacency list or an adjacency matrix? Justify your choice.

First: What Are These Two Representations?

Adjacency List

Each vertex has a list of its neighbors (and edge weights). Only stores edges that actually exist.

Space: O(n + m)

Check if edge (u,v) exists: O(degree of u)

Find all neighbors of u: O(degree of u)

Great when the graph is sparse (few edges, m ≪ n²).

Adjacency Matrix

An n×n grid. Entry [i][j] = weight of edge i→j, or ∞ if no edge.

Space: O(n²)

Check if edge (u,v) exists: O(1)

Find all neighbors of u: O(n) — must scan entire row

Great when the graph is dense (many edges, m ≈ n²).

What Does Prim's Actually Do? — The Key Operations

At each iteration, Prim's does two things:

  1. Extract-Min: pick the unvisited vertex with the smallest key (cheapest connecting edge).
  2. Decrease-Key: for each neighbor of the extracted vertex, update their key if we found a cheaper connecting edge.

The runtime of Prim's critically depends on how we find neighbors in step 2.

WHICH IS BETTER?

It depends on the graph density:

  • Sparse graph (m ≈ n): List gives O(n log n), Matrix gives O(n²). List wins.
  • Dense graph (m ≈ n²): List gives O(n² log n), Matrix gives O(n²). Matrix wins.
  • General case: The question asks for the "best possible asymptotic runtime." Since O((n+m) log n) is always ≤ O(n² log n), and equals O(n² log n) only when m = n², adjacency list is always at least as good.

📝 EXAM ANSWER

Choice: Adjacency list.

Prim's algorithm with a min-heap priority queue runs in O((n+m) log n) with an adjacency list. At each step, we extract the minimum-key vertex (O(log n)) and update its neighbors' keys (one Decrease-Key per edge, O(log n) each). Since there are n Extract-Min operations and m Decrease-Key operations total, the runtime is O(n log n + m log n) = O((n+m) log n).

With an adjacency matrix, finding neighbors requires scanning an entire row of n entries per vertex, giving O(n²) total. Since O((n+m) log n) ≤ O(n² log n) and equals O(n²) only when m = n² (dense graph) and we drop the log, adjacency list achieves the best asymptotic runtime for Prim's.

Problem 2.3
6 pts — Bellman-Ford and Positive Cycles

Bounded Earnings — Detecting Positive Cycles

The Question

Directed graph, edges have dollar amounts (positive = earn, negative = lose). Starting at a specified vertex, traversing an edge multiple times multiplies the earnings. Are your total earnings bounded (finite maximum) or unbounded (you can earn infinite money)? Design an O(mn) algorithm.

💰

Key insight: earnings are unbounded if and only if there exists a positive cycle reachable from the start vertex. If you can reach a cycle where the sum of edge weights is positive, you just keep going around it forever, earning more each loop.

✅ Bounded

No positive cycle reachable from start. There's some finite maximum you can earn — a best path that you can't improve by looping.

Example: A→B(+5)→C(+3). Best you can earn: $8. No way to do better.

❌ Unbounded

A positive cycle is reachable from start. Just loop around it forever.

Example: A→B(+5)→A(-2). Cycle total = +3. Go around n times: earn 3n dollars → ∞.

Why Bellman-Ford?

Bellman-Ford is famous for detecting negative cycles in shortest-path problems. Here we have the same idea in reverse: we want to detect positive cycles in a longest-path / maximum earnings problem.

The Connection to Bellman-Ford

Think about it this way: finding maximum earnings = finding the longest path (maximizing sum of edge weights). Bellman-Ford for longest paths: instead of relaxing edges by taking minimums, we take maximums.

Negative cycle in shortest-path = you can make paths infinitely short by looping ↔ Positive cycle in longest-path = you can make paths infinitely long by looping.

So we run a maximizing Bellman-Ford variant, then check: if we can still improve earnings after n-1 rounds, a positive cycle exists → unbounded.

The Algorithm

BOUNDED_EARNINGS(G, start):
  // Initialize: dist[v] = max earnings to reach v
  dist[start] = 0
  dist[v] = -∞  for all other v (unreachable initially)

  // Run n-1 rounds of "relaxation" (here: maximization)
  for round = 1 to n-1:
    for each edge (u, v) with weight w:
      if dist[u] + w > dist[v]:
        dist[v] = dist[u] + w   // found better (higher) path

  // Round n: if we can still improve, a positive cycle exists!
  for each edge (u, v) with weight w:
    if dist[u] + w > dist[v]:
      return false  // UNBOUNDED: positive cycle reachable

  return true  // BOUNDED: no positive cycle found
⚡

Why n-1 rounds? Any simple path in a graph with n vertices has at most n-1 edges. So after n-1 rounds of relaxation, dist[v] holds the maximum earnings along any simple path to v. If we can still improve in round n, the improving path must use a cycle — and since it improves earnings, that cycle must be positive.

📍

Reachability matters! We only care about positive cycles reachable from the start vertex. Initializing dist[v] = -∞ for non-start vertices ensures we only propagate earnings along paths reachable from start. A vertex with dist[v] = -∞ after all rounds was never reached.

Runtime: n-1 rounds × m edges per round = O(mn). ✓

Problem 2.4
6 pts — DAG DP: descendantScore

descendantScore — DAG Dynamic Programming

The Question

DAG where each vertex has an integer score. The descendantScore of a vertex is the maximum score among itself and all its descendants (vertices it can reach via directed paths). Compute descendantScore for ALL vertices in O(n+m).

🌳

Mental model: Think of a company org chart. Your "descendantScore" is the highest performer among you and everyone below you in the hierarchy. Every manager's score depends on the scores of their reports.

Key Observation — Process in Reverse Topological Order

Why Topological Order?

A DAG has no cycles, so we can find a topological ordering — a linear order where every edge goes from earlier to later. In topological order, no vertex has a descendant that comes before it.

If we process vertices in reverse topological order (sinks first, sources last), then when we compute descendantScore[v], all of v's children already have their descendantScore computed. So we can just take the max!

descendantScore[v] = max(score[v], max over all children c of: descendantScore[c])

The Algorithm

COMPUTE_DESCENDANT_SCORES(G):
  // Step 1: Topological sort the DAG
  topoOrder ← TOPO_SORT(G)   // O(n+m) via DFS

  // Step 2: Initialize — every vertex's descendantScore starts as its own score
  for each vertex v:
    descendantScore[v] ← score[v]

  // Step 3: Process in REVERSE topological order (sinks first)
  for v in reverse(topoOrder):
    for each child c of v:   // c is a vertex v has an edge to
      descendantScore[v] ← max(descendantScore[v], descendantScore[c])

  return descendantScore
💡

Why does this work? When we process v in reverse topological order, all vertices that v can reach (all descendants) come after v in topological order, meaning they were processed before v in our reverse loop. So descendantScore[c] is already the correct maximum for all of c's descendants when we use it for v.

Runtime

StepOperationRuntime
1Topological sort (DFS)O(n+m)
2Initialize descendantScoreO(n)
3Reverse topo loop — each edge visited onceO(n+m)
—TotalO(n+m) ✓

Step 3 is O(n+m) because the outer loop runs n times and the inner "for each child" loop runs over all edges exactly once total across all vertices.

📝 EXAM ANSWER

Algorithm:

  1. Run DFS to compute a topological ordering of the DAG. O(n+m).
  2. Initialize descendantScore[v] = score[v] for all v.
  3. Process vertices in reverse topological order. For each vertex v, for each outgoing edge (v→c), set descendantScore[v] = max(descendantScore[v], descendantScore[c]).
  4. Return the descendantScore array.

Correctness: Processing in reverse topological order ensures all descendants of v are processed before v. When we update descendantScore[v] using descendantScore[c] for each child c, descendantScore[c] already holds the maximum score in c's entire subtree. Taking the max over all children plus v's own score gives the correct descendantScore[v].

Runtime: Topo sort O(n+m) + one pass over vertices and edges O(n+m) = O(n+m). ✓

Problem 2.5
6 pts — SCC Reachability in O(b²)

SCC Path Reachability — Using the Condensation

The Question

You're given access to the condensation DAG of a graph (b SCCs total) and a function getSCC(v) that returns which SCC vertex v belongs to in O(1). Design an O(b²) algorithm to determine if there's a path from u to v in the original graph.

🔍

Remember: There's a path from u to v in the original graph if and only if getSCC(u) can reach getSCC(v) in the condensation DAG. This is because SCCs compress all the "local" connectivity — all vertices within the same SCC can already reach each other.

Why Work in the Condensation DAG?

Original graph is messy

The original graph could have n vertices and m edges with complex structure. But we can't afford expensive operations here.

Key: there's a path from u to v in G if and only if getSCC(u) can reach getSCC(v) in the condensation DAG.

Condensation DAG is clean

Only b ≤ n vertices. It's a DAG (no cycles). We just need to check reachability from one SCC-node to another in a graph with b vertices.

BFS/DFS on b vertices = O(b + edges of condensation). But the condensation has at most b² edges, so this is O(b²).

The Algorithm

PATH_EXISTS(u, v, condensationDAG, getSCC):
  // Step 1: Find which SCCs u and v belong to
  scc_u ← getSCC(u)   // O(1)
  scc_v ← getSCC(v)   // O(1)

  // Special case: same SCC → they can definitely reach each other
  if scc_u == scc_v:
    return true

  // Step 2: BFS/DFS from scc_u in the condensation DAG
  // Check if scc_v is reachable from scc_u
  reachable ← BFS(condensationDAG, scc_u)   // O(b + edges) = O(b²)

  return scc_v in reachable
⏱

Why O(b²)? BFS/DFS on a graph with b nodes and at most b² edges (since b² is the maximum number of directed edges between b nodes) runs in O(b + b²) = O(b²). The getSCC calls are O(1) each. So the total is O(b²). ✓

Alternative: Precompute All Pairs

If you need to answer many reachability queries, you could run BFS from every SCC-node, giving all-pairs reachability in O(b · b²) = O(b³). Or use a reachability matrix. But the problem only asks about one pair (u,v), so one BFS suffices.

📝 EXAM ANSWER

Algorithm:

  1. Call getSCC(u) and getSCC(v) to get the corresponding SCC-nodes in the condensation DAG. O(1).
  2. If getSCC(u) = getSCC(v), return true (same SCC → mutual reachability guaranteed).
  3. Run BFS from getSCC(u) in the condensation DAG. Return whether getSCC(v) is visited. O(b + edges of condensation) = O(b²).

Correctness: There is a path from u to v in the original graph if and only if getSCC(v) is reachable from getSCC(u) in the condensation DAG. This holds because: (a) within an SCC, all vertices can reach each other; (b) between SCCs, reachability in the original graph exactly corresponds to reachability in the condensation.

Runtime: Two getSCC calls O(1), plus BFS on a graph with b nodes and at most b(b-1) ≤ b² directed edges, which runs in O(b + b²) = O(b²). Total: O(b²). ✓
