Foundations — Start Here

What Even Is a Graph?

Before touching any exam problem, let's build from the ground up. A graph is just a way to represent relationships between things. Think of cities connected by roads, or websites linked to each other, or friends on a social network.

The Two Ingredients

Vertices (nodes) — the "things". Could be cities, people, web pages, computers — anything.

Edges — the "connections" between things. A road between cities. A friendship. A hyperlink.

G = (V, E) where V = vertices, E = edges

Two Flavors

Undirected: edges go both ways. If A connects to B, then B connects to A. (Friendships, roads)

Directed: edges have arrows. A→B doesn't mean B→A. (Web links, one-way streets, Twitter follows)

All of Section 5 uses directed graphs.

Weighted Graphs

Many problems attach a weight (cost, distance, time) to each edge. In Section 5, problems 5.1 and 5.2 use weighted/unweighted edges. The weight is just a number on the arrow.

Example: cities A, B, C. Edge A→B has weight 4 (4 km road). Edge B→C has weight 2. The path cost from A to C via B is 4+2=6.

Key Concepts You Must Know

Path & Distance

A path from u to v is a sequence of vertices u→a→b→...→v following edges. The distance (or shortest path cost) is the minimum total edge weight over all paths from u to v.

If no path exists, distance = ∞.

Reachability

Vertex v is reachable from u if there exists any directed path u→...→v. In an undirected graph, if any vertex is reachable from any other, the graph is connected.

Strongly Connected Components (SCC)

In a directed graph, an SCC is a maximal set of vertices where every vertex can reach every other vertex. Two cities A,B are in the same SCC if you can drive from A to B AND from B to A.

Condensation DAG

If you collapse each SCC into a single "super-node", the result is always a DAG (directed acyclic graph — no cycles). This is called the condensation of the graph.

Algorithms From Scratch

🔵 BFS — Breadth-First Search O(n + m)

What it does: explores a graph level by level, starting from a source vertex. Like ripples in water — first visits all neighbors, then their neighbors, etc.

Finds: shortest paths in unweighted graphs (or graphs where all edges have weight 1). Finds all reachable vertices.

How it works: uses a queue (first-in first-out). Start by putting the source in the queue. Repeatedly dequeue a vertex, look at all its neighbors, enqueue any not yet visited.

🟠 Dijkstra's Algorithm O((n+m) log n)

What it does: finds shortest paths from one source vertex to ALL other vertices in a weighted graph.

Key constraint: only works when all edge weights are non-negative. If there are negative edges, it can give wrong answers.

How it works: like BFS but uses a priority queue (min-heap) instead of a regular queue. Always processes the currently closest unvisited vertex next — the "greedy" choice.

⚠️

Remember: Dijkstra fails on negative edges. This is critical for Problem 5.1 where edges CAN be negative. That's why we need Floyd-Warshall instead.

🟣 Floyd-Warshall O(n³)

What it does: computes shortest paths between every pair of vertices — called "All-Pairs Shortest Paths" (APSP). It handles negative edge weights (but not negative cycles).

The DP idea: for each intermediate vertex k, ask: "Can I improve the path from i to j by going through k?" If dist[i][k] + dist[k][j] < dist[i][j], update it.

dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j]) for each k

Why O(n³): three nested loops — for each k (n choices), for each i (n choices), for each j (n choices).

The diameter connection: once Floyd-Warshall fills in all-pairs shortest paths, the diameter is just the maximum value in the matrix! One pass to find it.

🩵 DFS + Kosaraju's SCC Algorithm O(n + m)

DFS (Depth-First Search) dives as deep as possible before backtracking. Like exploring a maze — always go deeper until you hit a dead end, then backtrack and try another way.

Strongly Connected Components (SCC) decompose a directed graph into maximal clusters where every vertex can reach every other. Kosaraju's algorithm finds all SCCs in O(n+m) using two DFS passes:

  1. Run DFS on original graph G; push vertices to a stack in finish time order.
  2. Transpose G (reverse all edges) to get GT.
  3. Run DFS on GT in the order of the stack. Each DFS tree is one SCC.
💡

Why this works: if vertex A can reach B in G, then B can reach A in GT. The finish-time ordering ensures we always start a new DFS from an SCC that has no incoming edges from unvisited SCCs — so each DFS tree stays within one SCC.

Problem 5.1
10 pts — Graph Diameter

Finding the Diameter of a Graph

The Question

Given a weighted strongly connected directed graph G that may have negative edges (but no negative cycles), find the diameter — the largest shortest-path distance between any two vertices — in O(n³) time.

diameter(G) = max{ distance(u,v) | u,v ∈ V }
🤔

Let's think before coding. What does "diameter" mean intuitively? It's asking: among all pairs of vertices, what's the longest shortest path? The "worst case" travel distance in the graph.

Step 1: Identify What You Need

To find the maximum over all pairs, you first need the shortest path for every pair. That's All-Pairs Shortest Paths (APSP).

Why Not Dijkstra n Times?

Running Dijkstra from every vertex would give APSP in O(n(n+m) log n). But Dijkstra fails on negative edges! The problem says edges can be negative.

So Dijkstra is out.

Why Floyd-Warshall?

Floyd-Warshall handles negative edges (just not negative cycles, which we're told don't exist). Its runtime is exactly O(n³) — which matches what the problem asks for.

Perfect fit. ✓

Step 2: The Algorithm

DIAMETER(G):
  // Step 1: Run Floyd-Warshall to get all-pairs shortest paths
  dist ← n×n matrix, initialized as:
    dist[i][i] = 0  // distance from v to itself = 0
    dist[i][j] = weight(i→j)  // if edge exists
    dist[i][j] = ∞             // if no direct edge

  for k = 1 to n:          // try each vertex as intermediate
    for i = 1 to n:
      for j = 1 to n:
        if dist[i][k] + dist[k][j] < dist[i][j]:
          dist[i][j] = dist[i][k] + dist[k][j]

  // Step 2: Find the maximum over all pairs
  diameter ← -∞
  for i = 1 to n:
    for j = 1 to n:
      diameter = max(diameter, dist[i][j])

  return diameter

Step 3: Understand Why It's Correct

▶ Why does Floyd-Warshall give correct distances even with negative edges?

Floyd-Warshall uses dynamic programming, not a greedy approach like Dijkstra. It considers all possible intermediate vertices systematically. At step k, dist[i][j] holds the shortest path from i to j using only vertices {1, 2, ..., k} as intermediates.

This correctness doesn't depend on edge signs — it works as long as there's no negative cycle (otherwise the "shortest path" would be -∞ because you'd loop forever). The problem guarantees no negative cycles, so we're safe.

▶ Why can we just take the max of the dist matrix?

After Floyd-Warshall, dist[i][j] holds the shortest path distance from i to j for every pair (i,j). The diameter is defined as max{distance(u,v) | u,v ∈ V}. So we just scan all entries of the matrix and take the maximum. This final scan is O(n²), which is dominated by the O(n³) Floyd-Warshall step.

The graph is strongly connected, so no dist[i][j] entry will be ∞ — every vertex can reach every other. If the graph weren't strongly connected, we'd need to handle ∞ entries carefully.

Step 4: Runtime Analysis

StepOperationRuntime
1Initialize dist matrixO(n² + m)
2Floyd-Warshall triple loopO(n³)
3Find max in matrixO(n²)
—TotalO(n³) ✓

O(n³) dominates O(n²) and O(n²+m), so the total is O(n³) — exactly what the problem requires.

📝 Exam Answer Template

Algorithm: Run Floyd-Warshall on G to compute all-pairs shortest paths in O(n³). Then scan the resulting dist[i][j] matrix to find the maximum value; this is the diameter. Total time: O(n³).

Why Floyd-Warshall and not Dijkstra: G may have negative edge weights, and Dijkstra's algorithm is not correct on graphs with negative edges. Floyd-Warshall handles negative edges correctly as long as there are no negative cycles (which we are guaranteed).

Why the max is the diameter: After Floyd-Warshall, dist[i][j] = shortest path from i to j for all i,j. The diameter is by definition max{dist(u,v)} over all pairs, so we simply take the max over all entries.

Problem 5.2
13 pts — Closest Reachable Vertex

The Closest Vertex Reachable by Both x and y

The Question

Unweighted directed graph (all edges weight 1). Given distinct vertices x and y, find the vertex v such that:

  • v is reachable from x
  • v is also reachable by y (y can reach v)
  • Among all such v, the distance from x to v is minimized

Runtime must be O(m). (m = number of edges)

🤔

Read carefully. The problem says v must be reachable FROM x (x→v), and reachable BY y (y→v). So both x and y have paths to v. We want the v closest to x that y can also reach.

Breaking It Down — What Do We Need?

Condition 1: Reachable from x

We need all vertices reachable from x, and their distances. Since the graph is unweighted (weight 1 per edge), BFS from x gives us this in O(n+m) = O(m).

Condition 2: Reachable by y

We need all vertices that y can reach. That's: run BFS from y on the original graph. Any vertex v in this BFS result is reachable from y, so y can "reach" v.

The Key Insight: BFS Order Gives Us the Answer

Why BFS Order Matters

BFS from x visits vertices in non-decreasing order of distance from x. That means the first vertex we encounter that satisfies both conditions (reachable from x AND reachable from y) is automatically the closest such vertex to x.

This is the magic of BFS for unweighted graphs — the order of discovery IS the order of distance.

The Algorithm

CLOSEST_REACHABLE(G, x, y):
  // Step 1: BFS from y — find all vertices y can reach
  reachableByY ← BFS(G, y)  // returns set of reachable vertices

  // Step 2: BFS from x — visit vertices in order of distance
  // The FIRST vertex we see that's also in reachableByY is the answer
  queue ← [x]
  visited ← {x}
  while queue not empty:
    v ← dequeue(queue)
    if v in reachableByY:
      return v          // closest vertex reachable by both!
    for neighbor w of v:
      if w not in visited:
        visited.add(w)
        enqueue(queue, w)

  return null  // problem guarantees this won't happen
💡

Note: x itself might be the answer — if y can reach x, then x is distance 0 from itself, which is minimal. The problem tells us to handle this case.

Runtime Analysis

StepOperationRuntime
1BFS from y → find reachableByYO(n + m)
2BFS from x → scan in distance orderO(n + m)
—TotalO(n + m) = O(m) ✓

Since m ≥ n-1 in a connected graph (and m > 0 is given), O(n+m) = O(m).

Algorithm:

  1. Run BFS from y on G. Record all vertices reachable from y in a set R_y. This takes O(n+m).
  2. Run BFS from x on G. Process vertices in BFS order (i.e., non-decreasing distance from x). Return the first vertex v encountered such that v ∈ R_y. This takes O(n+m).

Correctness: BFS from x visits vertices in non-decreasing order of distance from x (since the graph is unweighted). The first vertex seen that's also in R_y is the closest vertex to x that y can also reach. Checking membership in R_y takes O(1) with a hash set or boolean array.

Runtime: Two BFS passes, each O(n+m) = O(m). Total O(m). ✓

Problem 5.3
17 pts — Somewhat-Connected

Is the Graph "Somewhat-Connected"?

The Question

A directed graph G is "somewhat-connected" if for every pair of vertices u,v, either there's a path from u to v, OR a path from v to u (or both). Check this in O(m + n).

🤔

Understand the definition first. "Somewhat-connected" means no pair of vertices is completely isolated from each other. For any u,v, at least one of them can reach the other. It's weaker than "strongly connected" (where both must reach each other) but stronger than nothing.

Step 1: Build Intuition With Examples

✅ Somewhat-Connected

A→B→C→D. For any pair: A can reach B,C,D. B can reach C,D. C can reach D. For the pair (D,A): D can't reach A, but A can reach D. ✓

A simple directed path through all vertices is always somewhat-connected.

❌ NOT Somewhat-Connected

A→B and C→D (two separate components). The pair (B,C): B can't reach C, and C can't reach B. ✗

Any two vertices with no path in either direction = NOT somewhat-connected.

Step 2: The SCC Insight

Key Observation

Collapse each SCC to a single node. You get the condensation DAG. The original graph is somewhat-connected if and only if the condensation DAG is somewhat-connected.

Why? Because all vertices within the same SCC can reach each other (that's the definition of SCC). So for any pair (u,v) within the same SCC, both can reach each other. The only pairs we need to worry about are those in different SCCs.

Step 3: When Is a DAG Somewhat-Connected?

The Topological Order Observation

A DAG has a topological ordering — a linear ordering of vertices where all edges go "forward" (left to right). The DAG is somewhat-connected if and only if this topological ordering forms a Hamiltonian path — meaning there's an edge between every consecutive pair in the ordering.

DAG is somewhat-connected ⟺ topological order is: v₁→v₂→v₃→...→vₙ (with direct edges)

In other words: the condensation DAG is somewhat-connected iff its topological ordering has an edge from each node to the next one. If any consecutive pair is missing an edge, those two SCC-nodes have no path between them.

Step 4: The Algorithm

SOMEWHAT_CONNECTED(G):
  // Step 1: Find all SCCs using Kosaraju's (or Tarjan's)
  SCCs ← KOSARAJU(G)   // O(n+m)

  // Step 2: Build the condensation DAG
  condDAG ← BUILD_CONDENSATION(G, SCCs)  // O(n+m)

  // Step 3: Topological sort the condensation DAG
  topoOrder ← TOPO_SORT(condDAG)  // O(b + edges_in_DAG) = O(n+m)

  // Step 4: Check if consecutive nodes in topo order have a direct edge
  for i = 0 to len(topoOrder) - 2:
    u ← topoOrder[i]
    v ← topoOrder[i+1]
    if edge (u→v) not in condDAG:
      return false   // this pair has no path in either direction!

  return true
▶ Why does checking consecutive topo-order pairs work?

In a topological ordering of a DAG, edges can only go "forward" (from earlier to later in the order). For any pair (u,v) where u comes before v in the ordering: u can reach v IF AND ONLY IF there's a forward path u→...→v in the DAG.

For "somewhat-connected", we need: for every pair (u,v), either u can reach v or v can reach u. In the topological order, if u comes before v, we need u to be able to reach v (v can't reach u since edges only go forward).

If u and v are consecutive in the topological order (no vertex between them), the ONLY way u can reach v is if there's a direct edge u→v. If that edge is missing, no forward path exists either (since there's no intermediate vertex to route through). So u can't reach v, and v can't reach u → NOT somewhat-connected.

This argument works because we're checking ALL consecutive pairs in the topological order — which covers all possible "gaps" in reachability.

▶ What about non-consecutive pairs in the topo order?

If every consecutive pair (v_i, v_{i+1}) has a direct edge, then by transitivity: v_1 can reach v_2, v_2 can reach v_3, so v_1 can reach v_3, etc. All earlier vertices can reach all later vertices. So all pairs are covered. ✓

Algorithm:

  1. Compute all SCCs of G using Kosaraju's algorithm in O(n+m). Build the condensation DAG C.
  2. Compute a topological ordering of C in O(b + |edges of C|) ≤ O(n+m).

Algorithm Selection Guide

Unweighted, 1 source: BFS → O(n+m)

Weighted ≥ 0, 1 source: Dijkstra → O((n+m)logn)

Weighted (neg ok), all pairs: Floyd-Warshall → O(n³)

Reachability, SCCs: DFS/Kosaraju → O(n+m)

Common Mistakes

❌ Using Dijkstra when there are negative edges

❌ Using Floyd-Warshall when asked for O(m) or O(n+m)

❌ Forgetting BFS gives shortest paths only for unweighted graphs

❌ Not building the condensation DAG for SCC problems
