Randomized Algorithms & Probability

When algorithms flip coins to make decisions, how do we analyze them? A crash course in probability, random variables, expectation, and a beautiful randomized algorithm for Min-Cut.

Why Randomized Algorithms?

Every algorithm you've seen so far in this course has been deterministic: given the same input, it always does the exact same thing. Randomized algorithms are different. They make random choices during execution (like flipping coins or rolling dice), and this randomness can actually help.

Sometimes, a randomized algorithm is simpler, faster, or the only known efficient approach. But it comes with a tradeoff: we need probability to reason about correctness and runtime.

Two Flavors of Randomized Algorithms

There are two fundamental types, and they're named after two cities famous for gambling:

🎰 Las Vegas Algorithms

Always correct. The randomness only affects how long it takes.

Example: QuickSort

  • Pick a pivot at random
  • Partition, then recurse on both halves
  • Always returns the correctly sorted array
  • Runtime varies: expected O(n log n), worst case O(n²)

Key question: how long does it take on average?

🃏 Monte Carlo Algorithms

Might be wrong. Runs in a fixed amount of time, but the answer may be incorrect.

Example: Randomized Min-Cut

  • Make random choices to find a cut
  • Might output a cut that is NOT minimum
  • Runtime is predictable
  • We can boost accuracy by repeating

Key question: what's the probability we get the right answer?

💡 The Key Insight

Both types of randomized algorithms require probability theory to analyze. For Las Vegas, we compute the expected runtime. For Monte Carlo, we compute the probability of correctness. That's why we need to learn probability foundations first.

Quick Check: Which Type?

A randomized algorithm always outputs a valid answer, but sometimes takes 5 seconds and sometimes takes 500 seconds. Is it Las Vegas or Monte Carlo?

Probability Foundations

The basic language of randomness: sample spaces, probabilities, and events.

The Setup: Sample Spaces

Every probability problem starts with a sample space, written as Ω (pronounced "oh-MEH-guh", the Greek capital letter Omega). This is the set of all possible outcomes of your random experiment. In this course, Ω is always finite.

Each individual outcome ω ∈ Ω (lowercase omega) gets a probability P(ω). These probabilities must satisfy two rules:

(1)   P(ω) ≥ 0   for every ω ∈ Ω

(2)   ∑ω ∈ Ω P(ω) = 1

That's it. Probabilities are non-negative and they all add up to 1. Everything else follows from these two axioms.

Events

An event is any subset S ⊆ Ω (⊆ means "is a subset of"). The probability of an event is just the sum of the probabilities of its outcomes:

P(S) = ∑ω ∈ S P(ω)

Concrete Examples

Example 1: Fair Coin

Ω = {H, T}, with P(H) = P(T) = 1/2

Part 3 / 8

Random Variables

Turning outcomes into numbers we can actually compute with.

What is a Random Variable?

A random variable is a function that assigns a real number to each outcome:

X : Ω → ℝ

It takes an outcome from the sample space and maps it to a number. For example, if you flip 3 coins, X could be "the number of heads." The outcome HHT maps to X = 2.

We care about three specific distributions in this course:

1. Bernoulli(p) (pronounced "ber-NOO-lee")

The simplest random variable. A single coin flip with probability p of "success."

Range: {0, 1}

Parameter: p, where 0 < p < 1

P(X = 1) = p       P(X = 0) = 1 − p

Think of it as: 1 = "success" (with probability p), 0 = "failure" (with probability 1−p).

2. Binomial(n, p) (pronounced "by-NO-mee-ul")

Flip n independent coins, each with probability p of heads. Count the total number of heads.

Range: {0, 1, 2, …, n}

Parameters: n (number of trials), p (success probability per trial)

P(X = k) = (nk) · pk · (1 − p)n−k

The (nk) is "n choose k" = n! / (k!(n−k)!), the number of ways to pick which k of the n trials are successes.

Key fact: Binomial(n, p) = X₁ + X₂ + … + Xₙ where each Xᵢ is an independent Bernoulli(p). This decomposition will be crucial later.

3. Geometric(p) (pronounced "jee-oh-MET-rik")

Keep flipping a coin (with probability p of heads) until the first success. X = the number of the trial where the first success occurs.

Range: {1, 2, 3, …} (infinite, but always a positive integer)

Parameter: p (success probability per trial)

P(X = k) = p · (1 − p)k−1

Logic: you need k−1 failures (each with probability 1−p), then 1 success (probability p).

Expectation

The weighted average value of a random variable, and the single most important number to know about a distribution.

Definition

The expectation (or expected value) of a random variable X with range {x₁, x₂, …} is:

E[X] = ∑i xᵢ · P(X = xᵢ)

It's the weighted average of all possible values, where the weights are the probabilities. Think of it as "what you'd get on average if you repeated the experiment forever."

Expectation of Bernoulli(p)

X has range {0, 1}, so:

E[X] = 0 · P(X=0) + 1 · P(X=1) = 0·(1−p) + 1·p = p

Makes sense: if a coin lands heads 30% of the time, the expected value of the indicator is 0.3.

Expectation of Geometric(p)

X has range {1, 2, 3, …}. The derivation uses a calculus trick (differentiating a geometric series):

E[X] = ∑k=1∞ k · p(1−p)k−1

The key step: factor out p, recognize the sum as the derivative of the geometric series ∑(1−p)k = 1/p, and the derivative gives 1/p². So:

E[X] = p · (1/p²) = 1/p

Intuition: if each trial succeeds with probability p = 1/6 (like rolling a 6 on a die), you'd expect to wait about 6 rolls for the first success.

Linearity of Expectation

The single most powerful and frequently used property in all of randomized algorithm analysis.

The Theorem

⭐ Linearity of Expectation

For any two random variables X and Y (they do NOT need to be independent):

E[X + Y] = E[X] + E[Y]

More generally, for any random variables X₁, X₂, …, Xₙ:

E[X₁ + X₂ + … + Xₙ] = E[X₁] + E[X₂] + … + E[Xₙ]

This is remarkable because it works unconditionally. The random variables can be dependent, correlated, whatever. The expectation of the sum is always the sum of the expectations.

Application: E[Binomial(n, p)] = np

Here's where the magic happens. Remember that Binomial(n, p) = X₁ + X₂ + … + Xₙ where each Xᵢ is Bernoulli(p). So:

E[Binomial(n, p)] = E[X₁ + X₂ + … + Xₙ]

= E[X₁] + E[X₂] + … + E[Xₙ]    (linearity!)

= p + p + … + p     (n terms)

= np

No binomial coefficients, no complicated sums. Just break it into simple pieces and add.

Why This Matters for Algorithms

In randomized algorithm analysis, you'll almost always use linearity of expectation in the same pattern:

  1. Define indicator variables: Create Xᵢ ∈ {0, 1} for each "event of interest."
  2. Express the total as a sum: X = X₁ + X₂ + … + Xₙ
  3. Compute each E[Xᵢ] individually: Usually this is easy since Xᵢ is Bernoulli.
  4. Add them up: E[X] = E[X₁] + … + E[Xₙ]

Quick Example: Expected Comparisons in QuickSort

For an array of n elements, let Xᵢⱼ = 1 if elements i and j are ever compared during QuickSort, and 0 otherwise. The total number of comparisons is:

X = ∑i < j Xᵢⱼ

By linearity: E[X] = ∑ E[Xᵢⱼ] = ∑ P(i and j are compared). Computing each P individually and summing gives O(n log n). This is how you prove QuickSort's expected runtime without any complicated argument.

Karger's Min-Cut Algorithm

A beautiful randomized algorithm that finds minimum cuts by randomly contracting edges, no source or sink needed.

Recall: The Min-Cut Problem

You saw Min-Cut before in the context of Max-Flow. Given a graph G = (V, E) with positive edge capacities, a cut is a partition of the vertices into two non-empty groups (L, R). The value of the cut is the total weight of edges crossing from L to R. The min-cut is the cut with the smallest value.

With Max-Flow, you needed to pick a source s and sink t, then apply the Max-Flow Min-Cut theorem. But what if you want to find the global min-cut without specifying s and t?

The Edge Contraction Operation

Before seeing the algorithm, you need one new concept: edge contraction.

Edge Contraction

Given an edge e = (u, v) in graph G, contracting e means:

  1. Merge vertices u and v into a single "super-vertex"
  2. All edges that were connected to u or v now connect to the super-vertex
  3. Remove any self-loops (edges from the super-vertex to itself)
  4. Keep parallel (multi) edges

The result is a smaller graph with one fewer vertex. We write the contracted graph as Ge.

Karger's Algorithm (The "Watermelon" Algorithm)

The professor's notes call this the "Watermelon" algorithm. Here is the full algorithm:

Algorithm: Karger's Contraction

Input: Graph G = (V, E)

while |V| > 2:
    Pick an edge e uniformly at random
    G ← Ge    // contract e
return G    // 2 super-vertices remain; edges between them = cut

Output: A cut (not necessarily the minimum cut!).

After |V| − 2 contractions, you're left with exactly 2 super-vertices. The edges between them define a cut of the original graph. This cut might be the min-cut, or it might not.

Why Is This a Monte Carlo Algorithm?

The algorithm always runs in polynomial time (exactly n − 2 contraction steps), but it might return the wrong answer. The output is always a valid cut, just not necessarily the minimum one.

If during any contraction step we accidentally contract an edge that belongs to the min-cut, we "destroy" that min-cut and can't find it anymore. So the key question is:

🔑 The Key Question

What is the probability that Karger's algorithm returns the correct min-cut?

We'll answer this fully in Session 8 (tomorrow). Spoiler: the probability is at least 2/(n(n−1)), which seems small, but we can boost it by repeating the algorithm many times.

Problem 1: Probability Basics

You roll two fair 6-sided dice. Let X be their sum.

  1. What is the sample space Ω? How many outcomes are in it?
  2. What is P(X = 7)?
  3. What is P(X = 2)?

(a) Ω = {(1,1), (1,2), …, (6,6)}, all ordered pairs. There are 6 × 6 = 36 outcomes, each equally likely with probability 1/36.

(b) X = 7 happens for: (1,6), (2,5), (3,4), (4,3), (5,2), (6,1). That's 6 outcomes out of 36, so P(X = 7) = 6/36 = 1/6.

(c) X = 2 happens only for: (1,1). That's 1 outcome, so P(X = 2) = 1/36.

Problem 2: Bernoulli and Expectation

A biased coin lands heads with probability 0.7. You flip it once.

  1. What distribution does this follow?
  2. What is E[X], where X = 1 for heads and X = 0 for tails?
  3. If you flip the coin 50 times, what is the expected total number of heads? (Name the property you used.)

(a) Bernoulli(0.7).

(b) E[X] = 0 · P(X=0) + 1 · P(X=1) = 0 · 0.3 + 1 · 0.7 = 0.7.

(c) Let Y = X₁ + X₂ + … + X₅₀ where each Xᵢ ~ Bernoulli(0.7). By linearity of expectation, E[Y] = E[X₁] + … + E[X₅₀] = 50 × 0.7 = 35 heads expected.

(Equivalently, Y ~ Binomial(50, 0.7), so E[Y] = np = 50 × 0.7 = 35.)

Problem 3: Geometric Expectation

You're searching for a specific node in a graph. Each probe has a 1/10 probability of finding it (independent of previous probes).

  1. What distribution models the number of probes until you find it?
  2. What is the expected number of probes?
  3. What is P(you need more than 20 probes)? Hint: the CDF of Geometric is P(X ≤ k) = 1 − (1−p)k.

(a) Geometric(1/10), since we repeat independent trials until the first success.

(b) E[X] = 1/p = 1/(1/10) = 10 probes.

(c) P(X > 20) = 1 − P(X ≤ 20) = 1 − [1 − (1 − 1/10)²⁰] = (9/10)²⁰.

(9/10)²⁰ = 0.9²⁰ ≈ 0.1216, so about a 12.2% chance. Even though the expected number is 10, there's a meaningful chance you need twice as many attempts.

Problem 4: Linearity of Expectation Application

There are 100 students in a class. Each student independently picks one of 7 days of the week (uniformly at random) as their "study day." Let X be the number of students who pick Monday.

  1. What distribution does X follow? What are its parameters?
  2. What is E[X]?
  3. Now let Y be the number of days (out of 7) that have at least one student. Using linearity of expectation and indicator variables, find E[Y]. Hint: define Yⱼ = 1 if day j has at least one student.

(a) Binomial(100, 1/7). Each of the 100 students independently has a 1/7 chance of picking Monday.

(b) E[X] = np = 100 × (1/7) ≈ 14.29 students.

(c) Define Yⱼ = 1 if day j is chosen by at least one student, 0 otherwise. Then Y = Y₁ + Y₂ + … + Y₇.

By linearity: E[Y] = E[Y₁] + … + E[Y₇] = 7 · E[Y₁] (by symmetry).

Now, E[Y₁] = P(day 1 has at least one student) = 1 − P(day 1 has zero students).

P(no student picks day 1) = (6/7)¹⁰⁰. So:

E[Y₁] = 1 − (6/7)¹⁰⁰ ≈ 1 − 0.0000000564 ≈ 0.99999994

E[Y] = 7 × (1 − (6/7)¹⁰⁰) ≈ 7.0 (almost certainly all 7 days are covered).

The important thing here is the technique: define indicator variables, use linearity, compute each probability separately. This pattern shows up constantly.

Problem 5: Karger's Algorithm Concepts

Consider the following graph with 4 vertices and 5 edges:

A ——— B
| \    |
|  \   |
C ——— D

Edges: A-B, A-C, A-D, B-D, C-D. All edges have weight 1.

  1. What is the min-cut of this graph, and what is its value?
  2. How many contraction steps does Karger's algorithm perform?
  3. In the first step, what is the probability that we contract an edge belonging to the min-cut?
  4. Is Karger's a Las Vegas or Monte Carlo algorithm? Why?

(a) Let's check all possible cuts. The minimum degree vertex is B or C (degree 2 each). Separating B from {A, C, D} cuts edges A-B and B-D, value = 2. Separating C from {A, B, D} cuts edges A-C and C-D, value = 2. Both are valid min-cuts. Min-cut value = 2.

(b) |V| − 2 = 4 − 2 = 2 contraction steps.

(c) Consider the min-cut {B} vs {A, C, D}. The cut edges are A-B and B-D. Out of 5 total edges, the probability of hitting one of these 2 cut edges is 2/5.

For the other min-cut {C} vs {A, B, D}, the cut edges are A-C and C-D, giving the same 2/5 probability.

So the probability that the first contraction ruins a specific min-cut is 2/5, and the probability it doesn't ruin it is 3/5.

(d) Monte Carlo. It always runs in a fixed number of steps (n − 2 contractions), but the output may not be the min-cut. The randomness affects correctness, not runtime.

Problem 6: Las Vegas vs Monte Carlo Classification

Classify each algorithm as Las Vegas or Monte Carlo and explain why:

  1. An algorithm that randomly permutes the input, then runs a deterministic sorting algorithm. It always outputs the sorted array.
  2. A primality testing algorithm that, for a given number n, picks random witnesses and declares "probably prime" or "definitely composite." It runs in fixed polynomial time but has a small probability of declaring a composite number as "probably prime."
  3. A search algorithm that picks random starting points. It always returns the correct answer, but the number of restarts varies.

(a) Las Vegas. Always correct (deterministic sort guarantees correctness). Randomness only affects runtime (different permutations might lead to faster/slower sorts).

(b) Monte Carlo. Fixed runtime, but might give the wrong answer (declaring a composite number as "probably prime"). The probability of error can be made small by picking more random witnesses.

(c) Las Vegas. Always returns the correct answer. The randomness affects how long it takes (number of restarts), not whether the answer is right.
