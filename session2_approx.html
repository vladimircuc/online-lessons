Warm-Up A — Reduction Direction
Which direction proves NP-hardness?

A student wants to prove that problem X is NP-hard. They have two options:

(Option A) Show that X reduces to 3-SAT: any instance of X can be transformed into a 3-SAT instance in polynomial time.
(Option B) Show that 3-SAT reduces to X: any 3-SAT instance can be transformed into an instance of X in polynomial time.

Which option proves X is NP-hard? Explain in 2 sentences why the other option does NOT prove hardness.

Solution

Option B proves NP-hardness. 3-SAT ≤ₚ X means "if I can solve X, I can solve 3-SAT." Since 3-SAT is NP-hard, this means X must also be at least as hard as 3-SAT — so X is NP-hard.

Option A (X ≤ₚ 3-SAT) says the OPPOSITE: "if I can solve 3-SAT, I can solve X." This means X is no harder than 3-SAT, placing X inside NP — but it says nothing about whether X is easy or hard. It could be in P for all we know.

Memory trick: to prove B is hard, reduce FROM a known hard problem TO B. The arrow points toward the new problem you want to prove hard.
Warm-Up B — EDP & Min Cut
Max edge-disjoint paths = min cut value

Graph G has vertices {s, a, b, c, d, t} with directed edges: s→a (cap 1), s→b (cap 1), s→c (cap 1), a→d (cap 1), b→d (cap 1), c→t (cap 1), d→t (cap 1).

On your board: Draw the graph. Find max edge-disjoint s→t paths. Find the min cut. Verify they are equal.

Solution

Max flow paths:

• Path 1: s→a→d→t (uses s→a, a→d, d→t)

• Path 2: s→b→d→t — BLOCKED: d→t already used!

• Try s→b→d→t: conflicts on d→t. Try s→c→t: works! Path 2: s→c→t

Max flow = 2. (Can't get 3 — node d has only one outgoing edge d→t with capacity 1, so at most 1 of {s→a→d, s→b→d} can complete.)

Min cut: ({s,a,b,c,d}, {t}). Edges crossing: d→t (cap 1), c→t (cap 1). Total = 2. ✅ Min cut = max flow = 2.

Note: s→b is "wasted" here — b connects only to d, which is already saturated by the s→a→d path. This is exactly the kind of bottleneck the greedy algorithm in today's session must navigate.

Part 2 / 8 — What is an Approximation Algorithm?

The Core Idea: Close Enough, Fast Enough

NP-hard problems can't be solved optimally in polynomial time (assuming P ≠ NP). But in practice we still need answers. The solution: approximation algorithms — polynomial-time algorithms that guarantee a solution within some factor of optimal.

The factor is called α (the approximation ratio). The formal definition from your notes:

Algorithm A is an α-approximation for problem P if for every input I, A outputs a solution that is within a factor of α from the optimal.

The exact meaning of "within factor α" depends on whether P is a maximization or minimization problem. These are different and easy to confuse:

Max vs Min — The Direction Matters

Maximization Problem

We want to maximize. Optimal = OPT (big is good). Our algorithm gives A(I).

OPT / α ≤ A(I) ≤ OPT
A(I) is at least 1/α of optimal

Example: max matching. If OPT=10 and α=2, our algo gives ≥5.

Minimization Problem

We want to minimize. Optimal = OPT (small is good). Our algorithm gives A(I).

OPT ≤ A(I) ≤ α · OPT
A(I) is at most α times optimal

Example: min vertex cover. If OPT=5 and α=2, our algo gives ≤10.

In both cases α ≥ 1. A perfect algorithm would have α=1 (exact optimal). The larger α gets, the worse the guarantee. For NP-hard problems we aim for α as close to 1 as possible.

A Concrete Max Example: Vertex Cover

Vertex Cover (minimize the number of vertices that cover all edges) has a simple 2-approximation: pick any uncovered edge, add BOTH its endpoints to the cover, repeat.

Let's verify this is α=2: when we pick edge (u,v) and add both, the optimal cover must include at least one of u or v for this edge. So we're using 2 vertices where optimal uses at least 1. We never use more than 2× what optimal uses. ✅

This is why the 2-approximation is achievable here — and why getting better (like a 1.5-approximation) is an open problem, believed to require fundamentally different techniques.
Part 3 / 8 — PTAS and FPTAS

Beyond a Fixed Ratio — Approximation Schemes

A single fixed α (like 2-approximation) is fine, but sometimes we need more flexibility. What if you need a 1.1-approximation for a high-stakes application? Or a 1.01-approximation?

An approximation scheme is a family of algorithms parameterized by ε > 0 that gives a (1+ε)-approximation. You can make ε as small as you want — getting arbitrarily close to optimal.

There are two important classes from your notes: PTAS and FPTAS. The difference is about how the runtime scales with ε:

PTAS — Polynomial Time Approximation Scheme

Algorithm A is a PTAS if for every fixed ε > 0, A runs in polynomial time in the input size n and achieves a (1+ε)-approximation.

Runtime: polynomial in n for any FIXED ε. But as ε → 0, the runtime might blow up. For example: O(n^(1/ε)) is a valid PTAS runtime — polynomial in n for fixed ε, but terrible as ε shrinks.

Example PTAS runtime for n=100:

⚠️ PTAS can be practically useless for small ε. O(n^(1/ε)) with ε=0.01 gives O(n^100) — technically polynomial in n but completely infeasible. You need FPTAS for truly practical guarantees.

FPTAS — Fully Polynomial Time Approximation Scheme

Algorithm A is an FPTAS if it runs in polynomial time in BOTH the input size n AND 1/ε, achieving a (1+ε)-approximation.

Runtime example: O(n³/ε²) — polynomial in both n and 1/ε. Doubling n or halving ε doesn't cause explosion. This is genuinely practical for a wide range of ε values.
poly(n) for fixed ε
runtime can explode as ε→0
Example: O(n^(1/ε))
poly(n, 1/ε) for all ε
runtime stays controlled
Example: O(n³/ε²)
Every FPTAS is also a PTAS. But not every PTAS is an FPTAS — FPTAS is strictly stronger. FPTAS algorithms exist for knapsack (Session 5) but NOT for problems like TSP or EDP.

Part 4 / 8 — The Greedy Algorithm for k-EDP

The Algorithm — Beautifully Simple 🌿

Given graph G=(V,E) and k source-sink pairs (s₁,t₁),...,(sₖ,tₖ), the greedy algorithm is:

GreedyEDP(G, (s₁,t₁), ..., (sₖ,tₖ)):
  Z ← ∅   // set of routed pairs
  E' ← E   // available edges
  for i = 1 to k:
    if ∃ path from sᵢ to tᵢ using only edges in E':
      P ← any such path   // e.g. via BFS/DFS
      Z ← Z ∪ {i}   // mark pair i as routed
      E' ← E' \ P   // remove used edges
  return Z
Runtime: O(k · (V+E)) — for each of k pairs, run BFS/DFS once. Since k ≤ |E| (can't have more pairs than edges), this is O(|E|·(V+E)) — polynomial. ✅

After watching the greedy run, notice:

✅ Greedy succeeds on every pair where a path exists in the remaining edges. It never skips a pair it could have routed.
❌ Greedy can fail to match optimal. An earlier routing can "use up" edges that a later pair needed, even when a globally better assignment existed. This is the approximation gap.
💡 Order matters. If we processed pairs in a different order, we might route more or fewer. Finding the optimal order is itself NP-hard. Greedy picks an arbitrary order — and that's where the approximation ratio comes from.
Part 5 / 8 — Proving the √|E| Approximation Ratio

The Theorem from Your Notes

From L3: there is a (greedy!) |E|^(1/2)-approximation algorithm for the k-edge-disjoint paths problem.

Theorem: Let Z be the set of pairs routed by greedy, and I* be the optimal routing. Then:

|I*| ≤ √|E| · |Z|

Equivalently: |Z| ≥ |I*| / √|E|

This means greedy routes at least 1/√|E| of the optimal number of pairs. So the approximation ratio is √|E| — greedy is at most √|E| times worse than optimal.

Note: √|E| depends on the number of edges in the graph, not just the number of pairs k. For a graph with 100 edges, the guarantee is within a factor of 10. For 10,000 edges, factor of 100. It's not a constant — it grows with graph size.

The Proof Idea — Step by Step

Let Z be greedy output (|Z| pairs routed), I* be optimal (|I*| pairs routed). We want to show |I*| ≤ √|E| · |Z|.

Step 1: What do the greedy paths "use"?

Greedy routes |Z| edge-disjoint paths. Since paths are edge-disjoint and all edges come from E, the total number of edges used by greedy paths is at most |E|. Call this m_used ≤ |E|.

Step 2: Why did greedy fail on some pairs?

Consider any pair (sᵢ, tᵢ) in I* that greedy did NOT route. When greedy tried this pair, there was no path from sᵢ to tᵢ in the remaining graph. This means every possible sᵢ→tᵢ path in the original G shares at least one edge with some greedy path. Otherwise greedy would have found and used that path.

Step 3: Count the "blocking"

Each greedy path Pⱼ has some length len(Pⱼ). It can "block" at most len(Pⱼ) pairs from I* (one per edge — a pair is blocked by Pⱼ only if EVERY path for that pair shares an edge with Pⱼ, but the bound counts by edges).

Total blocking capacity of all Z paths:

|I*| ≤ |Z| + Σⱼ len(Pⱼ)   // Z pairs that were routed, plus blocked pairs
Σⱼ len(Pⱼ) ≤ |E|   // paths are edge-disjoint, use ≤ |E| edges total
So: |I*| ≤ |Z| + |E|

This gives us |I*| ≤ |Z| + |E|, but that's not tight enough. We need a better argument using both bounds simultaneously.

Step 4: The AM-GM argument

We have two facts:

|Z| ≥ 1
greedy routed at least 1 pair (else optimal is also 0)
Σ len(Pⱼ) ≤ |E|
total edge usage bounded by graph size

The paths in Z are edge-disjoint, each of length ≥ 1. So:

|I*| ≤ |Z| · (average path length blocked per path) + ...
Each pair in I* not routed by greedy shares ≥1 edge with a Z-path.
|I*| ≤ |Z| + (total edges in Z-paths) ≤ |Z| + |E|
 
Now use: for any x,y > 0: x + y ≥ 2√(xy) (AM-GM).
We want to bound |I*| using √|E| · |Z|.
 
Key bound: |I*| ≤ √|E| · |Z| follows when:
(|Z|) · (|I*|/|Z|) ≤ |E| → |I*| ≤ |E|/|Z| · |Z| = |E| ...
Combined: |I*|² ≤ |E| · |Z| · (something) → |I*| ≤ √|E| · √|Z| ≤ √|E| · |Z|
The full algebraic proof uses the observation that each path in Z uses ≥1 edge and blocks ≥0 pairs from I*, and the total blocked pairs plus routed pairs equals |I*|. The √|E| bound comes from the geometric mean of the two constraints. The details are in L3 — the key takeaway is the √|E| factor and why it comes from the edge budget.

Part 6 / 8 — When and Why Greedy Fails to Hit Optimal

Greedy is NOT Always Optimal — Here's Why

The √|E| approximation ratio tells us greedy can be far from optimal. But when does this actually happen? Let's construct a graph where greedy provably does worse than optimal.

The classic bad example for greedy EDP is when the greedy routing "uses up" a bottleneck edge that blocks multiple later pairs, while a different routing would have avoided that bottleneck.

Step through: greedy routes pair 1 greedily, but blocks optimal routing of pairs 2 and 3

Pairs: (s₁,t₁) and (s₂,t₂). Both need to go through X→Y bottleneck. But there's also a direct edge s₁→t₁. Click Step to see greedy's mistake.

Why This Happens — The Fundamental Tension

Greedy is myopic — it only sees the current pair and picks any available path. It has no way to know that using the bottleneck edge for pair 1 will block pair 2.

Optimal is omniscient — it sees all pairs simultaneously and coordinates. It knows pair 1 has an alternate route (s₁→t₁ direct), so it reserves the bottleneck for pair 2 which has no alternative.

Greedy output: 1 pair routed (pair 1 via bottleneck, pair 2 blocked).
Optimal output: 2 pairs routed (pair 1 via direct edge, pair 2 via bottleneck).
In this example: |E| = 5, |Z| = 1, so √|E| ≈ 2.24. The theorem says |I*| ≤ √5 · 1 ≈ 2.24, and indeed |I*| = 2 ≤ 2.24. The bound is tight here!

Problem 1 — Approximation Ratio Identification
Is this a valid α-approximation? What is α?

Consider the following algorithm for Maximum Independent Set (a maximization problem — find the largest set of vertices with no edges between them):

"Pick any vertex v. Add v to the independent set S. Remove v AND all its neighbors from the graph. Repeat until no vertices remain."

(a) What is the approximation ratio α of this algorithm?

(b) Construct a small graph (5-6 vertices) where this algorithm achieves exactly α times worse than optimal. Draw it.

(c) Is there any graph class where this algorithm IS optimal (α=1)?

Solution

(a) The approximation ratio is Δ (maximum degree of any vertex) — or more loosely, we can say this greedy gives no constant approximation ratio in general. Here's why: when we pick v and remove all neighbors, we add 1 vertex and throw away up to deg(v) vertices. In the worst case, optimal could have picked ALL those neighbors instead. So we might use 1 where optimal uses deg(v), giving ratio up to Δ.

(b) Bad example: Star graph — one center node connected to k leaves. Algorithm picks center (1 vertex added), removes all k leaves. Independent set size = 1. Optimal: pick all k leaves (no edges between them). Ratio = k = n-1. The algorithm performs terribly on stars.

(c) Yes — tree graphs (and more generally, bipartite graphs). For trees, greedy gives a 2-approximation, and with careful implementation is optimal. The structure avoids the star worst-case because in a tree, no node can be adjacent to an independent set of exponential size.

Problem 2 — PTAS vs FPTAS Analysis (Draw the Curve)
Determine which class each algorithm belongs to

For each algorithm below, determine: is it a PTAS? An FPTAS? Neither? Justify each.

(a) Algorithm A has runtime O(n^(2/ε)). For input size n=100 and ε=0.5, is this practical? What about ε=0.01?
(b) Algorithm B has runtime O(n² · 2^(1/ε)). Does this qualify as PTAS? As FPTAS?
(c) Algorithm C has runtime O((n/ε)³). Does this qualify as PTAS? As FPTAS?
(d) A knapsack algorithm solves the problem exactly in O(nW) where W is the total weight. Is this a PTAS? An FPTAS?
Solution

(a) PTAS but not FPTAS. O(n^(2/ε)) — for fixed ε=0.5 this is O(n⁴), polynomial in n. ✅ PTAS. But (2/ε) is not polynomial in 1/ε — it APPEARS in the exponent of n. For ε=0.5: n^4 = 100^4 = 10^8, manageable. For ε=0.01: n^200 = 10^400, completely infeasible. NOT FPTAS because the exponent grows as 1/ε, making runtime non-polynomial in 1/ε.

(b) Neither PTAS nor FPTAS. O(n² · 2^(1/ε)) — the 2^(1/ε) term is EXPONENTIAL in 1/ε. For fixed ε, 2^(1/ε) is a constant, so for fixed ε this is O(n²) — polynomial in n. BUT for ε=0.01, 2^100 ≈ 10^30 operations regardless of n. This completely fails the PTAS requirement that runtime must be polynomial in n for any fixed ε (since 2^(1/ε) is fixed for fixed ε, it's technically a constant... wait). Actually: for any FIXED ε > 0, 2^(1/ε) is a constant (just a big one). So this IS a PTAS! But it is NOT an FPTAS because 2^(1/ε) is not polynomial in 1/ε. Correction: this IS a PTAS but NOT an FPTAS.

(c) FPTAS (and hence also PTAS). O((n/ε)³) = O(n³/ε³) — polynomial in both n and 1/ε. For n=100, ε=0.01: (100/0.01)³ = 10000³ = 10^12 — large but polynomial. This qualifies as FPTAS. ✅

(d) Neither PTAS nor FPTAS as stated. O(nW) is "pseudo-polynomial" — polynomial in n and W, but W (the weight) is exponential in its INPUT SIZE (the number of bits to write W). If W = 2^100, this is not polynomial in the actual input size. However, we can turn this into an FPTAS by SCALING W. This is exactly the FPTAS for knapsack covered in Session 5 — by rounding and scaling weights down by a factor of (1+ε), we get O(n²/ε) runtime that IS polynomial in n and 1/ε.

Graph G with edges: s₁→A→B→t₁, s₁→C→t₁, s₂→A→D→t₂, s₂→C→D→t₂. (All directed, all capacity 1.)

Now for pair 2: paths available are s₂→A→D→t₂ (A still has s₂→A, D available) — wait, s₂→A is separate. Path s₂→A→D→t₂ uses s₂→A, A→D, D→t₂, which are all unused. So greedy routes BOTH pairs! But if greedy had picked s₁→C→t₁ for pair 1 instead, pair 2 would try s₂→C→D→t₂ — blocked on C. It would fall back to s₂→A→D→t₂ — fine! Still 2 pairs. In this graph greedy achieves optimal regardless of path choice.

Key lesson: the √|E| bound is a WORST-CASE guarantee. On many graphs greedy does much better. The tight examples require carefully constructed bottleneck structures.
Problem 4 — Approximation Ratio Lower Bound
Construct a graph where greedy achieves exactly ratio √|E|

To show an approximation ratio is tight, you need to construct an input where the algorithm performs EXACTLY as badly as the ratio predicts.

On your board: Construct a graph G and set of pairs such that:

• Greedy routes exactly 1 pair

• Optimal routes √|E| pairs

• The ratio achieved is exactly √|E|

Hint: Think about a bottleneck structure where one long path "burns through" many edges, preventing many short paths from being routed.

Solution

Construction: Let m = √|E| be an integer. Build a graph with:

• A long path s₀ → v₁ → v₂ → ... → vₘ → t₀ of length m (uses m edges). This is pair 0: (s₀, t₀).

• For each i=1,...,m: a short edge sᵢ → vᵢ (one edge each). Pair i: (sᵢ, tᵢ) where tᵢ is reachable from vᵢ via one more edge vᵢ→tᵢ... but vᵢ is already on the long path.

More cleanly — the standard tight example:

Graph: m "horizontal" short paths s₁→t₁, s₂→t₂, ..., sₘ→tₘ (each length 1). Plus ONE "vertical" long path that goes through a node on each horizontal path: s₀ → x₁ → x₂ → ... → xₘ → t₀, where xᵢ is the middle node of horizontal path i (splitting each horizontal path into two).

Total edges: m (long path) + 2m (split horizontals) = 3m = 3√|E|... so |E| = 3m, √|E| ≈ √(3m).

Greedy (unlucky): routes pair 0 (the long path), using ALL xᵢ nodes. Now each horizontal path sᵢ→xᵢ→tᵢ is broken — xᵢ→tᵢ edge is used. Greedy routes 1 pair.

Optimal: routes ALL m horizontal pairs: s₁→x₁→t₁, ..., sₘ→xₘ→tₘ. These are all edge-disjoint. Optimal = m pairs.

Ratio = m/1 = m = √(|E|/3) ≈ √|E|. ✅
This construction shows the √|E| bound is essentially tight — there exist graphs where greedy is √|E| times worse than optimal. This means we can't do better than √|E| with greedy, even with optimal tie-breaking.
Problem 5 — Mixed Concepts (Session 1+2)
Why can't we do better than √|E| using any greedy-style algorithm?

This problem tests deep understanding of WHY the approximation ratio is what it is.

(a) The greedy EDP algorithm uses BFS/DFS to find augmenting paths. Could we improve the ratio by using a smarter path-finding strategy — for example, always picking the SHORTEST available path?

(b) EDP is NP-hard. Greedy gives √|E|-approximation. Does there exist a constant-factor (say 5-approximation) polynomial algorithm for EDP? What would we need to prove/disprove this?

(c) Max flow (single commodity) solved EDP exactly in polynomial time. Multi-commodity flow (k ≥ 2 commodities) is NP-hard for integers. Is there a contradiction? Why not?

Solution

(a) Shortest path greedy helps — and IS better. If we always pick the shortest available path, the approximation ratio improves. Specifically, Edmonds-Karp style (BFS shortest paths) gives better bounds in practice. For the k-EDP problem, the algorithm in your notes (L3) already implies using shortest paths — the |E|^(-1/2) bound was proven for ANY path-finding strategy but shortest paths can tighten it. However, no polynomial-time algorithm is known that gives a constant-factor approximation for general k-EDP, so we're still stuck with superconstant ratios.

(b) Unknown — major open problem. Whether there's a constant approximation for k-EDP is an open question. To prove a constant approximation exists, you'd need to design such an algorithm and prove its ratio. To prove no constant approximation exists, you'd need to show that approximating EDP within any constant factor is NP-hard (an "inapproximability" result). Neither has been fully resolved for general k-EDP. We only know: greedy gives √|E|, and optimal is NP-hard.

(c) No contradiction — different problem structure. Single-commodity max flow asks: push as much flow as possible from ONE source to ONE sink. All flow is fungible — a unit entering at s is identical to any other unit. The LP relaxation is integral (total unimodularity of the constraint matrix), so the LP solution is automatically an integer solution. Multi-commodity flow has k separate "colors" of flow that can't mix. The LP relaxation is fractional — you can route 0.5 units of each commodity through a shared edge, satisfying capacity. But integer multi-commodity flow requires full units of each color, which may conflict. The LP-to-integer gap is what makes it NP-hard — you can solve the fractional version in poly time but rounding it to integers is hard.
