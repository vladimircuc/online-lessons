What Is an API?

Before we touch any code, let's understand the most important concept in full-stack development.

The Simple Definition

API stands for Application Programming Interface. It's a way for two programs to talk to each other. In web development, it almost always means: the frontend sends a request to a URL, and the backend sends back data.

🍕 The Pizza Analogy

Think of ordering pizza by phone. You (the frontend) call the pizza shop (the backend). You don't walk into the kitchen yourself. You tell them what you want through a known channel (the phone number, which is like the API URL). The kitchen prepares the pizza (queries the database, processes data) and delivers it back to you (JSON response). The phone number + the menu is the API. You don't need to know how they make the pizza, you just need to know what to order and where to call.

In Your Dashboard Project

Your dashboard page needs to show stats: how many messages were sent today, how many active keywords there are, etc. That data lives in a database. But the browser (frontend) can't talk to the database directly, that would be a massive security risk. So instead:

Browser(your Vue page)
→
API RequestGET /api/dashboard
→
Server(your .ts file)
→
Database(SQLite file)
← JSON response travels back the same path ←

The API is the middleman. The frontend asks for data by hitting a URL. The backend handles the request, grabs data from the database, and sends it back as JSON (which is just a structured text format that JavaScript can easily read).

What Does JSON Look Like?

JSON is the "language" the API speaks. Here's what your dashboard API sends back:

API ResponseJSON
{
  "messages": [
    {
      "id": "abc-123",
      "phone": "+15551234567",
      "keywordDetected": "HELP",
      "messageText": "I need help with my baby",
      "status": "delivered"
    }
  ],
  "stats": {
    "messagesToday": 12,
    "activeKeywords": 4,
    "activeCaregivers": 15
  }
}

It's just key-value pairs. The frontend receives this object and uses it to fill in the stat cards and the table rows you see on the dashboard.

🔑
Key takeaway: An API is just a URL that accepts requests and returns data. Your dashboard.get.ts file IS the API endpoint. When the frontend calls /api/dashboard, that file runs, queries the database, and returns JSON.

Vue, Nuxt, and Nitro: What's What?

These three names come up a lot. Here's how they relate, starting from the bottom.

Vue.js = The UI Library

Vue is a JavaScript library for building user interfaces. It gives you components, reactivity (when data changes the screen updates automatically), templates with directives like v-for and v-if, and tools like ref() and computed(). Vue handles only the frontend. It runs in the browser. It has no idea what a database is.

Nuxt = The Full-Stack Framework (Built on Vue)

Nuxt wraps around Vue and adds everything Vue is missing. You're still writing Vue components (same syntax, same reactivity), but now you also get:

📁
File-Based Routing
Create a file at pages/dashboard/index.vue and it automatically becomes the /dashboard URL. No router config needed.
⚙️
Built-in Server
Write API routes in server/api/. No need to set up a separate Express or Fastify server yourself.
🔄
Composables
Helper functions like useFetch() that make calling your API dead simple. One line to fetch, parse, and store data reactively.
🏗️ The Building Analogy

If Vue is the bricks and mortar (you can build anything, but you have to do all the plumbing and wiring yourself), then Nuxt is the pre-built house with plumbing and electrical already installed. You're still using the same bricks (Vue components), but Nuxt gives you the structure and infrastructure around them.

Nitro = The Backend Engine Inside Nuxt

When you write files inside the server/ folder, those run on Node.js, NOT in the browser. The engine that powers those server routes is called Nitro. You rarely interact with Nitro directly. The key function you'll see is defineEventHandler(), which is Nitro's way of creating a route handler (like Express's app.get()).

How File Names Become Routes

This is the convention that makes Nuxt "magical." The file path and name determine the URL and HTTP method:

File-Based Routing RulesConvention
// PAGES (frontend, what the user sees in the browser)
app/pages/index.vue           →  /
app/pages/dashboard/index.vue →  /dashboard
app/pages/flows/[id].vue      →  /flows/123  (dynamic param)

// API ROUTES (backend, runs on the server)
server/api/dashboard.get.ts   →  GET  /api/dashboard
server/api/auth/login.post.ts →  POST /api/auth/login
server/api/flows/[id].put.ts  →  PUT  /api/flows/123

The .get / .post / .put suffix tells Nuxt which HTTP method the route responds to. The folder path becomes the URL path. [id] is a dynamic segment that matches any value.

✅
Bottom line: You're writing Vue components like normal, but Nuxt organizes them and gives you a backend. Your dashboard.vue is a standard Vue component. Your dashboard.get.ts is a Nuxt/Nitro server route. Same project, two different worlds (browser vs server), connected by HTTP requests.

Prisma & SQLite: The Data Layer

SQLite: The Simplest Database

Most databases (MySQL, PostgreSQL) are separate servers you install and connect to over a network. SQLite is different: it's just a single file sitting in your project folder, usually called sqlite.db or dev.db. No server to install, no port, no password. Your app reads and writes directly to that file.

In this project, that file lives at /db/sqlite.db inside the repo on Trisha's machine. When the app runs with npm run dev, Prisma talks to that file. That file IS the database, containing all the tables: messages, keywords, caregivers, workflow steps, logs, etc.

Prisma: The Translator Between TypeScript and SQL

Prisma is an ORM (Object-Relational Mapper). It lets you interact with your database using TypeScript instead of writing raw SQL queries.

Without Prisma (raw SQL)
query.sqlSQL
SELECT COUNT(*) 
FROM caregivers 
WHERE status = 'ACTIVE';
With Prisma (TypeScript)
query.tsTS
prisma.caregiver.count({
  where: { status: "ACTIVE" }
})

Both produce the exact same result. Prisma translates the TypeScript method call into SQL, sends it to the SQLite file, and returns a JavaScript value.

How Prisma Knows Your Tables: The Schema

There's a file called prisma/schema.prisma in the project. It defines every table (Prisma calls them "models"), their columns, and relationships:

prisma/schema.prismaPrisma Schema
model Message {
  id          String   @id @default(uuid())
  phone       String
  messageText String
  direction   String    // "inbound" or "outbound"
  createdAt   DateTime @default(now())
  logs        Log[]    // one message has many logs (relation)
}

model Log {
  id        String   @id
  status    String   // "delivered", "failed", etc.
  messageId String
  message   Message  @relation(fields: [messageId], references: [id])
  createdAt DateTime
}

When you run npx prisma generate, Prisma reads this schema and generates a typed client library customized to your tables. That's why you can write prisma.message.findMany() and get autocomplete for fields like phone and messageText.

Prisma Methods Used in This Project

findMany()
Returns an array of records matching a filter. Equivalent to SELECT * FROM ...
count()
Returns a number (the count). Equivalent to SELECT COUNT(*) FROM ...
include: {}
Fetches related records too (like a SQL JOIN). include: { logs: { take: 1 } } grabs the most recent log for each message.
🔑
The full chain: Schema defines the structure → prisma generate creates the client → your code uses prisma.model.method() → Prisma converts it to SQL → SQL runs against the SQLite file → results come back as JavaScript objects.

The 4 Layers of This App

Click each layer to expand it and see what it does, what files belong to it, and how it connects to the next layer.

LAYER 1
Vue Template (What You See)
The HTML with stat cards, table rows, filters, and pagination buttons.
<template>

This is the <template> section of the Vue file. It's HTML with Vue directives sprinkled in: {{ stats?.messagesToday }} displays a number. v-for="msg in pageMessages" loops through messages to create table rows. v-if conditionally shows icons. @click handles button presses.

This layer doesn't fetch data or know where it comes from. It just reads from reactive variables that the script layer provides.

Connects to Layer 2 by: reading stats, pageMessages, filteredMessages, and other reactive variables.

LAYER 2
Vue Script (Reactive Logic)
Fetches data, filters it, paginates it, handles user interactions.
<script setup>

This is the <script setup> section. It runs when the page loads. Its job: call useFetch('/api/dashboard') to get data from the backend, then process that data using computed() properties (filtering by time, keyword, search text, then slicing for pagination).

It also manages user interaction state with ref(): which page they're on (currentPage), what they typed in search (search), which keywords they toggled (selectedKeywords).

Connects to Layer 3 by: calling useFetch('/api/dashboard'), which sends an HTTP GET request to the server.

LAYER 3
Nuxt Server Route (API Backend)
Receives the HTTP request, runs database queries, returns JSON.
dashboard.get.ts

This file runs ONLY on the server (Node.js), never in the browser. When the frontend hits /api/dashboard, Nuxt routes the request here because of the naming convention.

Inside defineEventHandler, it calculates UTC date boundaries, fires 5 Prisma queries in parallel using Promise.all, processes the results (percent change, message status mapping), and returns a JSON object.

Connects to Layer 4 by: calling Prisma methods like prisma.message.findMany() and prisma.caregiver.count().

LAYER 4
Prisma + SQLite (Database)
Translates TypeScript to SQL, reads/writes the database file on disk.
schema.prisma + sqlite.db

Prisma takes each method call and translates it into SQL. It sends that SQL to the SQLite file and returns the result as JavaScript values.

The schema defines the structure (what tables exist, what columns they have, how tables relate). The SQLite file is the actual storage, a single binary file on disk.

This is the bottom of the stack. Data flows back up: SQLite → Prisma → Server Route → useFetch → computed() → template → your screen.

✅
The rule: Each layer only talks to its immediate neighbor. The template reads from the script. The script calls the API. The API calls Prisma. Prisma queries SQLite. No layer skips over another.

Backend Breakdown: dashboard.get.ts

Let's go through the actual code piece by piece.

1. The Import and Handler Wrapper

dashboard.get.tsLines 1-4
import { prisma } from '../utils/prisma'

export default defineEventHandler(async (event) => {
  // everything happens in here
})

Line 1 imports the Prisma client (a pre-configured instance ready to query the database). The rest wraps everything in defineEventHandler, Nuxt/Nitro's way of defining an API route handler. The async keyword is necessary because database queries are asynchronous (they take time, you have to wait).

2. Date Boundaries (Why UTC Matters)

dashboard.get.tsLines 6-12
const now = new Date()

// "Today" = midnight UTC today to midnight UTC tomorrow
const todayStart = new Date(Date.UTC(
  now.getUTCFullYear(),  // 2026
  now.getUTCMonth(),     // 3 (months are 0-indexed, 3 = April)
  now.getUTCDate(),      // 6
  0, 0, 0, 0              // hours, min, sec, ms all zeroed
))
⚠️
Why UTC? The database stores timestamps in UTC. If you used local Central Time (UTC-5), a message created at 11 PM Central on April 5th is actually 4 AM April 6th in UTC. Using UTC boundaries prevents this mismatch. The database speaks UTC, so we ask questions in UTC.

3. Promise.all: Running 5 Queries at Once

dashboard.get.tsLines 14-33
const [
  messagesMonthList,      // Query 1: all messages this month
  messagesYesterdayCount, // Query 2: count of yesterday's messages
  keyword,                // Query 3: count of active keywords
  workflowStep,           // Query 4: count of workflow steps
  caregiverCount          // Query 5: count of active caregivers
] = await Promise.all([
  prisma.message.findMany({/* month filter + include logs */}),
  prisma.message.count({/* yesterday filter */}),
  prisma.keyword.count({ where: { isActive: true } }),
  prisma.workflowStep.count(),
  prisma.caregiver.count({ where: { status: "ACTIVE" } })
])
🍳 The Breakfast Analogy for Promise.all

Making breakfast one thing at a time: toast bread (2 min), then fry eggs (3 min), then pour juice (30 sec) = 5.5 minutes. Making breakfast in parallel with Promise.all: start everything at the same time, finish when the slowest is done = 3 minutes. Same results, much faster. That's what Promise.all does with database queries.

The left side uses array destructuring: the results come back as [result1, result2, result3, result4, result5], and we give each one a name. Order matters, the first name matches the first query.

4. Post-Processing and Return

dashboard.get.tsLines 35-60
// Filter the month list to count just today's messages
const messagesToday = messagesMonthList.filter(msg => {
  const created = new Date(msg.createdAt)
  return created >= todayStart && created < tomorrowStart
}).length  // .length gives the count after filtering

// Compute % change vs yesterday
let percentChange = 0
if (yesterdayCount === 0) {
  percentChange = messagesToday === 0 ? 0 : 100
} else {
  percentChange = Math.round(
    ((messagesToday - yesterdayCount) / yesterdayCount) * 100
  )
}

// Return the final JSON response
return {
  messages: /* mapped array with clean shape */,
  stats: { messagesToday, percentChange, activeKeywords, ... }
}

Whatever you return, Nuxt automatically converts it to JSON and sends it as the HTTP response. The frontend receives this exact shape.

Frontend Breakdown: dashboard.vue

1. Fetching Data

dashboard.vueLine 7
const { data } = await useFetch('/api/dashboard')

This single line sends an HTTP GET request to /api/dashboard, waits for the JSON response, and stores it in a reactive variable called data. "Reactive" means if this value changes, anything on screen that depends on it re-renders automatically.

2. ref() vs computed()

ref() = A box you put things in

A value you can directly read and write. User types in search? That updates a ref. User clicks page 2? That updates a ref.

Examples: search, currentPage, selectedKeywords

computed() = An Excel formula

A value that derives from other values. You never set it directly. It recalculates automatically when its inputs change.

Examples: messages, filteredMessages, totalPages, pageMessages

3. The Operators: ?. and ??

Operator CheatsheetJS/TS
// ?. = Optional Chaining
// "Access .messages, but if data.value is null, just return
//  undefined instead of crashing."
data.value?.messages

// ?? = Nullish Coalescing
// "If the left side is null or undefined, use the right side."
data.value?.messages ?? []
// = "Give me messages, or an empty array if there aren't any."

4. The Filtering Pipeline

All Messagesfrom API
→
Time Filtertoday/week/month
→
Keyword FilterHELP/COPE/etc
→
Search Filtertext matching
Each step narrows the list. 50 msgs → 20 (this week) → 8 (HELP only) → 3 (matching "555")

Each filter uses JavaScript's .filter() method, which returns a new array with only the items that pass the test. They chain: the output of one becomes the input of the next.

5. Pagination Math

dashboard.vuePagination
const pageSize = 4  // 4 messages per page

// 12 messages / 4 per page = 3 pages
const totalPages = computed(() =>
  Math.ceil(filteredMessages.value.length / pageSize)
)

// Which 4 to show? Page 2 = items 4,5,6,7
const pageMessages = computed(() => {
  const start = (currentPage.value - 1) * pageSize
  return filteredMessages.value.slice(start, start + pageSize)
})
